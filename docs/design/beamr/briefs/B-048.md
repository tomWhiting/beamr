# B-048: Loader validating-boundary contract with BoundedCursor and DecodeBudget

> **Cluster:** beamr
> **Checklist:**
> - C178 — Loader uses a BoundedCursor that validates all element counts against remaining payload bytes before pre-allocating
> - C179 — A DecodeBudget (recursion depth, total decoded nodes, total allocated bytes) is threaded through every recursive ETF and chunk decoder
> - C180 — Zlib literal decompression is capped upfront via take(limit) — never inflate-then-check
> - C181 — Per-load atom budget prevents a single module from interning more than MAX_ATOMS_PER_MODULE atoms into the global table
> - C182 — All pre-allocation in the loader routes through an ensure_count helper that rejects counts exceeding remaining bytes or MAX_TABLE_ENTRIES
> **Stories:**
> - S33 (Reviewer, Verifying beamr Implementation) — As a reviewer, I want to verify that the loader rejects adversarial .beam files (deep nesting, huge counts, zlib bombs) within bounded resource usage, so that untrusted bytecode cannot crash or OOM the host during decode.

## Purpose

PR #5 patched three specific loader DoS holes (ETF recursion, unbounded Vec::with_capacity, zlib bomb), but the fixes are ad-hoc — each new chunk decoder can reopen the same vulnerability class. This brief installs the structural contract: a BoundedCursor that makes infeasible pre-allocations unspellable, a DecodeBudget that uniformly caps recursion/nodes/bytes across all decoders, and a per-load atom budget. The goal is that adding a new decoder inherits the safety properties automatically.

## Task

Refactor the loader's decode layer to use a unified BoundedCursor (validates counts against remaining bytes), a DecodeBudget (threaded through all recursive decoders), and an ensure_count helper. Consolidate the existing ad-hoc MAX_ETF_DEPTH and MAX_TABLE_ENTRIES checks into the budget. Add a per-load atom budget. The existing etf.rs depth-bounding and decompress_bounded in chunks.rs are kept but wired through the budget. Scope: loader/decode/ only. Does not touch import resolution or the interpreter.

## Requirements

### R1: BoundedCursor wrapper

THE SYSTEM SHALL provide a BoundedCursor that wraps the existing Cursor and validates all element counts against remaining payload bytes before any pre-allocation. WHEN a decoder calls bounded_cursor.ensure_count(count, min_element_size), THE SYSTEM SHALL return an error if count * min_element_size > remaining bytes or count > MAX_TABLE_ENTRIES.

**Acceptance:**
- BoundedCursor struct wraps Cursor and delegates read_u8/u16/u32/i32/read_bytes to it
- ensure_count(count: usize, min_elem_bytes: usize) rejects if count > MAX_TABLE_ENTRIES or count * min_elem_bytes > self.remaining()
- All Vec::with_capacity calls in chunks.rs are preceded by ensure_count
- A crafted chunk with count=0xFFFFFFFF and 12 bytes of payload is rejected before any allocation

**Files:**
- create: crates/beamr/src/loader/decode/bounded.rs
- modify: crates/beamr/src/loader/decode/mod.rs
- modify: crates/beamr/src/loader/decode/chunks.rs

**Checklist:**
- C178 — Loader uses a BoundedCursor that validates all element counts against remaining payload bytes before pre-allocating
- C182 — All pre-allocation in the loader routes through an ensure_count helper that rejects counts exceeding remaining bytes or MAX_TABLE_ENTRIES

### R2: DecodeBudget struct

THE SYSTEM SHALL define a DecodeBudget with three counters: remaining recursion depth, remaining decoded nodes, and remaining allocated bytes. WHEN any counter reaches zero, THE SYSTEM SHALL return a budget-exhausted error. THE SYSTEM SHALL thread the budget &mut through every recursive decoder.

**Acceptance:**
- DecodeBudget struct: { depth_remaining: usize, nodes_remaining: usize, bytes_remaining: usize }
- charge_node(&mut self) decrements nodes_remaining, returns Err if zero
- charge_bytes(&mut self, n: usize) decrements bytes_remaining, returns Err if insufficient
- descend(&mut self) decrements depth_remaining, returns Err if zero
- ascend(&mut self) increments depth_remaining (restores after returning from recursion)
- Default budget: depth=256, nodes=1_000_000, bytes=64MB (configurable via constructor)
- Budget is created per load_module call, not shared across loads

**Files:**
- create: crates/beamr/src/loader/decode/budget.rs
- modify: crates/beamr/src/loader/decode/mod.rs

**Checklist:**
- C179 — A DecodeBudget (recursion depth, total decoded nodes, total allocated bytes) is threaded through every recursive ETF and chunk decoder

### R3: Thread budget through ETF decoder

WHEN decode_external_term is called, THE SYSTEM SHALL pass the DecodeBudget &mut and call budget.descend() at each recursive entry and budget.ascend() on return. THE SYSTEM SHALL call budget.charge_node() for each decoded term. THE SYSTEM SHALL replace the existing MAX_ETF_DEPTH depth parameter with the budget's depth counter.

**Acceptance:**
- decode_external_term signature includes &mut DecodeBudget parameter
- The existing depth: usize parameter is removed — depth tracking lives in the budget
- Every recursive call (tuple elements, list elements, map pairs) descends/ascends the budget
- Every decoded term calls budget.charge_node()
- Existing etf.rs tests pass with the budget threaded through
- A term with 1_000_001 nodes (e.g. a flat list of 1M elements) is rejected by node budget

**Files:**
- modify: crates/beamr/src/loader/decode/etf.rs

**Checklist:**
- C179 — A DecodeBudget (recursion depth, total decoded nodes, total allocated bytes) is threaded through every recursive ETF and chunk decoder

### R4: Thread budget through chunk decoders

WHEN decode_atom_chunk, decode_import_chunk, decode_export_chunk, decode_lambda_chunk, decode_line_chunk, and decode_literal_chunk are called, THE SYSTEM SHALL pass the DecodeBudget and call budget.charge_bytes() for pre-allocations and budget.charge_node() for each decoded entry.

**Acceptance:**
- All decode_*_chunk functions accept &mut DecodeBudget
- Pre-allocations charge the budget before allocating
- A module with atom_count=100_000 and export_count=100_000 and literal_count=800_001 exceeds node budget and is rejected
- load.rs's top-level load_module creates a fresh DecodeBudget and passes it to all decoders

**Files:**

### R5: Bounded zlib decompression via budget

WHEN the LitT chunk is decompressed, THE SYSTEM SHALL cap the decompressor output at budget.bytes_remaining via take(limit) and charge the budget for the decompressed bytes. THE SYSTEM SHALL NOT inflate the entire stream before checking size.

**Acceptance:**
- decompress_bounded uses decoder.take(budget.bytes_remaining + 1) to cap output
- After decompression, budget.charge_bytes(output.len()) is called
- A zlib bomb (small compressed payload that expands to > budget bytes_remaining) is rejected before exhausting memory
- Existing decompress_bounded logic is preserved but wired through the budget's byte counter

**Files:**
- modify: crates/beamr/src/loader/decode/chunks.rs

**Checklist:**
- C180 — Zlib literal decompression is capped upfront via take(limit) — never inflate-then-check

### R6: Per-load atom budget

WHEN atoms are interned during module loading, THE SYSTEM SHALL track the count of newly interned atoms against a per-load budget (MAX_ATOMS_PER_MODULE). IF the budget is exceeded, THE SYSTEM SHALL reject the module load. THE SYSTEM SHALL NOT count atoms that are already interned (they are free lookups).

**Acceptance:**
- MAX_ATOMS_PER_MODULE constant defined (default: 65_536 — matches BEAM's atom chunk u16 count limit)
- DecodeBudget gains atoms_remaining: usize field
- decode_atom_chunk increments atom count only for newly interned atoms (intern returns new index, not existing)
- A module with 65_537 unique atom names is rejected during decode
- A module that re-uses existing atoms (e.g. ok, error, true) does not count them against the budget

## Boundaries

- SHALL NOT change the loader's public API beyond adding budget/cursor parameters to internal decode functions
- SHALL NOT add runtime/interpreter checks — this brief is decode-time only
- SHALL NOT implement ETF safe-mode (rejecting fun/external-ref/port tags in literals) — that is a follow-up
- SHALL NOT touch import resolution or the Module struct
- SHALL NOT remove the existing ad-hoc hardening until the budget-based replacements are verified
