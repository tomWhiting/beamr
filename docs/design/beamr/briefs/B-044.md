# B-044: Closure version binding for hot code loading

> **Cluster:** beamr
> **Depends on:** B-041, B-042
> **Checklist:**
> - C130 — Process struct holds x-registers, y-registers, stack, mailbox, code position
> - C150 — BIF registry maps (Module, Function, Arity) to a Rust function pointer

## Purpose

Closures (funs) are heap-allocated objects that capture a module name, a lambda index into the module's `lambdas` table, free variables, and arity. When a closure is called, the interpreter looks up the module from the registry and indexes into its `lambdas` Vec using the stored `function_index`.

Under hot code loading, if the module is reloaded with a different lambda table (functions reordered, added, or removed), the closure's `function_index` becomes stale — it references the wrong lambda or goes out of bounds. Unlike call_ext targets (which can be resolved dynamically by MFA), closures encode a positional index that has no meaning outside its original module version.

BEAM/OTP solves this with unique fun identifiers that are matched against the module's fun table. This brief implements a similar mechanism for beamr.

## Boundaries

- This brief SHALL NOT change the heap layout of closures unless necessary (adding a generation field is acceptable if it fits the existing header scheme)
- This brief SHALL handle closures created in old module versions that are called after the module is reloaded
- This brief SHALL ensure closures work correctly in the trampoline pattern (higher-order BIFs like lists:map that call closures via re-entry)

## Requirements

### R1: Unique fun identifier

Add a unique identifier to each lambda entry so closures can be matched against the correct lambda regardless of table position.

**Acceptance:**
- `LambdaEntry` gains a `unique_id: u64` field, computed from a hash of `(module_name, function_name, arity, num_free)` at load time
- `Module.lambdas` entries include the unique_id
- The unique_id is deterministic for the same source function across recompilations (same function name + arity = same unique_id), so closures created under v1 can find their lambda in v2 if the function still exists
- New method: `Module::find_lambda_by_id(unique_id: u64) -> Option<&LambdaEntry>` scans lambdas for matching unique_id
- Test: create two versions of a module with the same functions in different order. Verify find_lambda_by_id returns the correct lambda in both versions.

**Files:**
- modify: crates/beamr/src/loader/decode/chunks.rs (LambdaEntry)
- modify: crates/beamr/src/loader/load.rs (unique_id computation)
- modify: crates/beamr/src/module.rs (find_lambda_by_id)

**Checklist:** C150

### R2: Closure heap layout gains generation and unique_id

Extend the closure heap representation to include the module generation and the lambda's unique_id, so that at call time the interpreter can detect version mismatches and resolve the closure against the correct module version.

**Acceptance:**
- Closure heap layout becomes: `[header] [module_atom] [function_index] [arity] [num_free] [generation] [unique_id] [free_var_0] ... [free_var_n]`
- `write_closure` takes additional `generation: u64` and `unique_id: u64` parameters
- `Closure` accessor gains: `generation() -> u64`, `unique_id() -> u64`
- Header tag or word count updated to accommodate the two extra words
- All existing closure tests updated for new layout
- `make_fun2` / `make_fun3` opcode handlers pass the correct generation (from the currently executing module) and unique_id (from the lambda entry)

**Files:**
- modify: crates/beamr/src/term/boxed/mod.rs (write_closure)
- modify: crates/beamr/src/term/boxed/accessors.rs (Closure)
- modify: crates/beamr/src/interpreter/opcodes/closures.rs (make_fun2, make_fun3)

**Checklist:** C130

### R3: Version-aware closure dispatch

When calling a closure, the interpreter must handle version mismatches: if the closure's generation doesn't match the current module's generation, resolve the lambda by unique_id instead of by index.

**Acceptance:**
- `call_fun` checks if `closure.generation()` matches the target module's generation
- If generations match: use `closure.function_index()` directly (fast path, O(1))
- If generations differ: use `module.find_lambda_by_id(closure.unique_id())` to find the lambda in the current version
- If the lambda is not found in the current version (function was removed): check the registry's old version. If found there, execute against the old version. If not found anywhere, raise `badfun` error.
- Test: create closure in module A v1. Reload module A to v2 (same functions, different order). Call closure — resolves via unique_id. Reload module A to v3 (function removed). Call closure — raises badfun.

**Files:**
- modify: crates/beamr/src/interpreter/opcodes/closures.rs (call_fun)

**Checklist:** C150

### R4: Trampoline compatibility

The trampoline pattern (used by lists:map, lists:filter, etc.) calls closures via `TrampolineRequest::CallClosure`. Verify this path correctly handles version-aware closure dispatch.

**Acceptance:**
- Audit `TrampolineRequest::CallClosure` handling in `crates/beamr/src/interpreter/opcodes/trampoline.rs`
- Ensure the trampoline passes the closure through the same version-aware dispatch as R3
- Test: create a closure in module A v1. Pass it to lists:map (trampoline path). Mid-map, reload module A to v2. The closure continues executing against the version it was created with (v1 generation + unique_id resolution).

**Files:**
- modify: crates/beamr/src/interpreter/opcodes/trampoline.rs (if changes needed)

**Checklist:** C150

### R5: GC traversal updated for new closure layout

The GC traces closure free variables starting at a fixed word offset. The new layout shifts free variables by 2 words (generation + unique_id). Update the GC's closure traversal.

**Acceptance:**
- `rewrite_copied_object` in gc/mod.rs correctly identifies the new free variable start offset for closures
- Generation and unique_id words are copied verbatim (they are not heap pointers, no tracing needed)
- Free variables at the new offset are still correctly traced and rewritten during collection
- Test: create a closure with free variables. Trigger GC. Verify closure is intact with correct generation, unique_id, and free variables after collection.

**Files:**
- modify: crates/beamr/src/gc/mod.rs (rewrite_copied_object, closure case)

**Checklist:** C130
