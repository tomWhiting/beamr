# B-051: Process-heap allocation for BIF and NIF results — eliminate Box::leak in native boundary

> **Cluster:** beamr
> **Checklist:**
> - C188 — BIF and NIF boxed results are allocated on the calling process's heap via ProcessContext heap allocation — not via Box::leak
> **Stories:**
> - S32 (Workflow Author, Writing Gleam Workflows) — As a workflow author, I want the VM not to leak memory monotonically when my workflow calls BIFs or materialises literals in a loop, so that long-running workflows don't OOM the host.

## Purpose

BIF and NIF results that return boxed terms (tuples, lists, binaries, floats, maps) are allocated via Box::leak through ProcessContext's alloc_tuple/alloc_cons and direct Box::leak calls across 20+ native files. These allocations are permanent — never freed, never garbage collected. Every BIF call that returns a boxed term leaks memory. This brief routes all BIF/NIF boxed result allocation through the calling process's heap via ProcessContext, making them GC-reclaimable.

## Task

Give ProcessContext access to the process heap. Replace alloc_tuple and alloc_cons in context.rs to allocate on the process heap (with GC-safe ensure-space). Provide additional process-heap allocation helpers (alloc_float, alloc_binary, alloc_bigint, alloc_map). Migrate all BIF/NIF files that call Box::leak directly to use the new ProcessContext helpers. The meridian_ffi helpers (ok_binary, err_binary, ok_nil) are migrated too. Scope: native boundary allocation only. Does not touch interpreter literal path (that is B-050).

## Requirements

### R1: ProcessContext gains process heap access

THE SYSTEM SHALL give ProcessContext a mutable reference to the calling process's heap so that BIF/NIF results can be allocated on GC-managed memory. THE SYSTEM SHALL provide ensure_heap_space(words: usize) to trigger GC if insufficient space.

**Acceptance:**
- ProcessContext gains a field or method to access the process heap (&mut [u64] or a HeapAllocator trait)
- ensure_heap_space(n) checks available heap words, triggers minor GC if insufficient, triggers major GC if still insufficient
- The heap reference is valid for the duration of the BIF call and points to the process's current heap
- ProcessContext's existing fields (pid, registers, atom_table, etc.) are unchanged

**Files:**
- modify: crates/beamr/src/native/context.rs
- modify: crates/beamr/src/interpreter/mod.rs

**Checklist:**

### R2: Process-heap allocation helpers in ProcessContext

THE SYSTEM SHALL replace alloc_tuple and alloc_cons with process-heap-allocating versions. THE SYSTEM SHALL add alloc_float, alloc_binary, alloc_bigint, alloc_list, and alloc_map helpers. THE SYSTEM SHALL NOT use Box::leak in any allocation helper.

**Acceptance:**
- alloc_tuple(&mut self, elements: &[Term]) -> Result<Term, Term> allocates on process heap, calls ensure_heap_space first
- alloc_cons(&mut self, head: Term, tail: Term) -> Result<Term, Term> allocates 2 words on process heap
- alloc_float(&mut self, value: f64) -> Result<Term, Term> allocates header + f64 on process heap
- alloc_binary(&mut self, bytes: &[u8]) -> Result<Term, Term> allocates header + length + data on process heap
- alloc_bigint(&mut self, sign: bool, limbs: &[u64]) -> Result<Term, Term> allocates header + sign + limbs on process heap
- alloc_list(&mut self, elements: &[Term]) -> Result<Term, Term> builds a proper list (chain of cons cells) on process heap
- alloc_map(&mut self, keys: &[Term], values: &[Term]) -> Result<Term, Term> allocates flatmap on process heap
- All helpers use write_float/write_tuple/write_cons/write_binary from term::boxed — same term layout as before
- No Box::leak in context.rs

**Files:**
- modify: crates/beamr/src/native/context.rs

## Boundaries

- SHALL NOT touch interpreter literal allocation — that is B-050
- SHALL NOT touch the atom-name intern Box::leak in atom/table.rs — that is legitimate and permanent by design
- SHALL NOT change NativeFn signature — BIFs still receive &[Term] and &mut ProcessContext, return Result<Term, Term>
- SHALL NOT change term layout or encoding — only where the memory backing the term lives
- SHALL NOT implement a global term allocator or arena — allocation targets the calling process's heap specifically
