## Purpose

Gleam's string type is a binary. Every string literal, every string operation, and every binary protocol handler compiles to binary construction and matching instructions. These are among the trickiest BEAM opcodes — the encoding uses bit-level addressing and match contexts that track position through a binary. Without them, the interpreter cannot handle any Gleam code that touches strings or binary data.

## Task

Implement binary construction and matching opcodes in crates/beamr/src/interpreter/opcodes.rs. Binary construction: bs_init2 initializes a binary builder on the heap, bs_put_integer and bs_put_binary append segments. Binary matching: bs_start_match2 creates a match context from a binary term, bs_get_integer2 and bs_get_binary2 extract segments advancing the position, bs_match_string matches a literal byte sequence, bs_test_tail2 verifies no remaining bytes. All operations work through a BinaryBuilder and MatchContext that track bit-level position.

## Requirements

### R1: Binary builder initialization — bs_init2

WHEN the interpreter executes bs_init2 THE SYSTEM SHALL allocate a binary construction context on the process heap with the specified size hint. The context SHALL track: write position (in bits), allocated capacity, and a pointer to the data region. IF heap space is insufficient THE SYSTEM SHALL trigger GC via the fail mechanism. It SHALL NOT pre-fill the buffer — only allocate.

**Acceptance:**
- bs_init2 with size 10 allocates a binary builder on the heap
- The builder starts with write position 0
- The builder's capacity is at least the requested size in bytes
- Insufficient heap space branches to fail label (GC opportunity)

### R2: Binary segment appending — bs_put_integer, bs_put_binary

WHEN the interpreter executes bs_put_integer THE SYSTEM SHALL append an integer value as a binary segment with the specified bit size and endianness. bs_put_binary SHALL append a binary term's bytes to the builder. Both SHALL advance the write position. IF the segment doesn't fit THE SYSTEM SHALL raise a badarg. It SHALL NOT allow writing past the allocated capacity.

**Acceptance:**
- bs_put_integer with value 65, size 8, big-endian appends byte 0x41
- bs_put_binary appends the bytes of a binary term to the builder
- Write position advances by the segment's bit size
- Writing past capacity raises badarg

**Checklist:**

### R3: Match context initialization — bs_start_match2

WHEN the interpreter executes bs_start_match2 THE SYSTEM SHALL create a match context from a binary term, storing: a pointer to the binary data, the total bit size, and the current read position (initially 0). The match context SHALL be stored in a register. IF the operand is not a binary THE SYSTEM SHALL branch to the fail label. It SHALL NOT copy the binary data — the match context references the original.

**Acceptance:**
- bs_start_match2 on a 5-byte binary creates a context with position 0 and size 40 bits
- bs_start_match2 on a non-binary term branches to fail label
- The match context references the original binary data (no copy)

**Files:**

**Checklist:**

### R4: Segment extraction — bs_get_integer2, bs_get_binary2

WHEN the interpreter executes bs_get_integer2 THE SYSTEM SHALL extract an integer from the match context with the specified bit size and endianness, advancing the position. bs_get_binary2 SHALL extract a sub-binary. IF there are not enough bits remaining THE SYSTEM SHALL branch to the fail label. It SHALL NOT modify the source binary.

**Acceptance:**
- bs_get_integer2 with size 8 from context at position 0 of [0x41, 0x42] returns 65 (0x41)
- Position advances by 8 bits after extraction
- bs_get_binary2 with size 16 extracts a 2-byte sub-binary
- Extracting past the end of the binary branches to fail label

### R5: Literal matching and tail check — bs_match_string, bs_test_tail2

WHEN the interpreter executes bs_match_string THE SYSTEM SHALL compare the next N bytes in the match context against a literal byte sequence and branch to the fail label if they don't match, advancing the position if they do. bs_test_tail2 SHALL verify that exactly N bits remain in the context (typically 0 for exhaustive matching). It SHALL NOT advance the position on a failed match.

**Acceptance:**
- bs_match_string matching 'he' against 'hello' at position 0 succeeds and advances by 16 bits
- bs_match_string matching 'xx' against 'hello' at position 0 branches to fail label
- bs_test_tail2 with 0 bits after consuming all data succeeds
- bs_test_tail2 with 0 bits but data remaining branches to fail label

## Boundaries

- SHALL NOT implement bit-level operations beyond byte boundaries — Gate 1 needs byte-aligned binaries only
- SHALL NOT implement binary comprehensions — not used by Gleam's core patterns
- SHALL NOT implement sub-binary sharing or reference-counted binaries — all binaries are heap-copied for now (optimization deferred)
- SHALL NOT implement utf8/utf16/utf32 segment types — byte segments only for Gate 1
