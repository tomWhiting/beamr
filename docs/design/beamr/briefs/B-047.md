# B-047: Ordered link and monitor sets for deterministic exit propagation

> **Cluster:** beamr
> **Checklist:**
> - C183 — Process link and monitor sets use insertion-ordered storage (not HashSet) so exit signal delivery order is deterministic and reproducible across runs
> **Stories:**
> - S29 (Reviewer, Verifying beamr Implementation) — As a reviewer, I want the exit signal propagation to be tested with multi-level link chains so that I can verify cascading failures propagate correctly through supervision trees.

## Purpose

Process links are stored as HashSet<u64>, which makes exit signal delivery order nondeterministic — two runs of the same program can deliver exit signals to linked processes in different orders, producing different supervision behaviour. This is the cheapest, most self-contained fix within the determinism gap (#10) and lands independently of the full determinism seam. Monitors have the same issue with HashMap iteration order.

## Task

Replace HashSet<u64> for process links with an insertion-ordered Vec<u64> (deduplicated on insert). Replace any HashMap-based monitor storage with an insertion-ordered equivalent. Ensure exit signal delivery iterates in insertion order so the cascade is reproducible. Scope: link and monitor storage only. Does not touch timers, PID counters, entropy, or the virtual clock seam.

## Requirements

### R1: Replace HashSet<u64> links with Vec<u64>

THE SYSTEM SHALL store process links as a Vec<u64> that maintains insertion order. WHEN add_link(pid) is called, THE SYSTEM SHALL append the pid only if it is not already present (deduplicate). WHEN remove_link(pid) is called, THE SYSTEM SHALL remove the pid while preserving the relative order of remaining entries.

**Acceptance:**
- Process::links field type is Vec<u64>, not HashSet<u64>
- add_link checks contains() before pushing — O(n) is acceptable for typical link counts (< 100)
- remove_link uses retain or swap_remove-equivalent that preserves insertion order of remaining elements (not swap_remove, which reorders)
- take_links returns Vec<u64> via std::mem::take
- links() returns &[u64] (slice reference)

### R2: Replace HashMap-based monitor storage with ordered equivalent

IF monitors are stored in a HashMap or HashSet, THE SYSTEM SHALL replace them with a Vec or BTreeMap to ensure deterministic iteration order. THE SYSTEM SHALL preserve insertion order for monitors so that DOWN message delivery order is reproducible.

**Acceptance:**
- Monitor storage iterates in insertion order (or a stable, deterministic order)
- Monitor addition, removal, and iteration semantics are unchanged except for ordering guarantees
- demonitor correctly removes entries without disturbing order of remaining monitors

**Files:**
- modify: crates/beamr/src/process/mod.rs

**Checklist:**

### R3: Exit propagation iterates in insertion order

WHEN a process exits and its links are drained for exit signal propagation, THE SYSTEM SHALL iterate the link set in insertion order so that exit signals are delivered to linked processes in the order the links were established.

**Acceptance:**
- supervision/link.rs take_links().into_iter() iterates in insertion order (guaranteed by Vec)
- If process A links to B, then C, then D — exit signals are delivered in order B, C, D
- The cascade Vec in propagate_exit_signals processes entries in the same deterministic order

### R4: Tests for deterministic exit signal order

THE SYSTEM SHALL include tests that verify exit signal delivery order matches link establishment order.

**Acceptance:**
- Test: process P links to A, B, C in that order. P exits. A receives exit signal before B, B before C (verified via message ordering or a recording sink)
- Test: unlink B from P, P exits. A receives signal before C — order preserved minus removed entry
- Test: monitor ordering — if P is monitored by X then Y, DOWN to X is delivered before DOWN to Y
- Existing supervision tests continue to pass

## Boundaries

- SHALL NOT implement the full determinism seam (virtual clock, decision log, seeded entropy) — only link and monitor ordering
- SHALL NOT change link/monitor semantics — only the iteration order guarantee changes
- SHALL NOT use BTreeSet keyed by PID value for links — insertion order is the required property, not numeric order
- SHALL NOT optimise for large link counts (> 1000) — Vec with linear scan is acceptable for the typical case
