# B-080: process_info/1,2 BIF — process introspection

> **Cluster:** beamr
> **Depends on:** B-016
> **Checklist:**
> - C150 — BIF registry maps (Module, Function, Arity) to a Rust function pointer

## Purpose

`erlang:process_info/1` and `erlang:process_info/2` are the primary process introspection BIFs in BEAM. They return information about a running process: its current function, heap size, message queue length, registered name, status, and more. OTP libraries (gen_server, supervisor) use these for diagnostics, and Gleam's process module wraps them. Without them, any OTP code that calls process_info fails with an unresolved import error.

## Task

Implement process_info/1 (all info) and process_info/2 (specific item or list of items) as BIFs. Start with the subset of info items needed by OTP: current_function, heap_size, message_queue_len, registered_name, status, trap_exit, links, monitors.

## Requirements

### R1: process_info/2 — single item query

Register `erlang:process_info/2` as a BIF. Accepts (Pid, Item) where Item is an atom. Returns `{Item, Value}` or `undefined` if the process doesn't exist.

**Acceptance:**
- Supported items: `current_function` → `{M, F, A}` tuple, `heap_size` → word count, `message_queue_len` → integer, `registered_name` → atom or `[]`, `status` → atom (running/waiting/suspended), `trap_exit` → boolean, `links` → list of PIDs, `monitors` → list of monitor info
- Non-existent PID returns `undefined`
- Unknown item returns badarg
- Test: spawn a process, query each supported item, verify return format

**Files:**
- create: crates/beamr/src/native/process_info_bifs.rs
- modify: crates/beamr/src/native/mod.rs

**Checklist:**
- C150 — BIF registry maps (Module, Function, Arity) to a Rust function pointer

### R2: process_info/1 — all items query

Register `erlang:process_info/1` as a BIF. Accepts (Pid) and returns a list of `{Item, Value}` tuples for all default items.

**Acceptance:**
- Returns a list containing tuples for all supported items
- Order matches BEAM convention (or is at least deterministic)
- Non-existent PID returns `undefined`
- Test: spawn a process, call process_info/1, verify list contains expected items

**Files:**
- modify: crates/beamr/src/native/process_info_bifs.rs

**Checklist:**
- C150 — BIF registry maps (Module, Function, Arity) to a Rust function pointer

### R3: Process state access via SharedState

The BIF needs to read process state from the process table. Add accessor methods that safely read process properties without requiring exclusive ownership.

**Acceptance:**
- SharedState provides `process_info(pid, item) -> Option<Term>` that reads from the process table
- Handles the ProcessSlot states correctly (Present, Executing, Absent)
- For Executing processes: returns metadata from ProcessMetadata (current_function, status=running)
- For Present processes: reads from the Process struct directly
- No deadlocks — does not hold process mutex while allocating terms
- Test: query process_info on a running process (Executing state) returns current_function

**Files:**
- modify: crates/beamr/src/scheduler/mod.rs (SharedState accessor)

**Checklist:**
- C150 — BIF registry maps (Module, Function, Arity) to a Rust function pointer

## Boundaries

- SHALL NOT implement all ~30 process_info items — only the subset needed by OTP stdlib and Gleam
- SHALL NOT return sensitive internal data (heap pointers, raw stack frames) — only safe observable properties
- SHALL NOT block the calling process to inspect the target — read from SharedState/process table
