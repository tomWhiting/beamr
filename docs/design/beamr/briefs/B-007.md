# B-007: Implement term comparison and ordering

> **Cluster:** beamr
> **Depends on:** B-005, B-006
> **Checklist:**
> - C46 — Term equality comparison (== semantics): same-type immediates compare by value, boxed terms compare structurally
> - C47 — Term exact equality comparison (=:= semantics): integer 1 and float 1.0 are not equal
> - C48 — Term ordering follows the BEAM order: number < atom < reference < fun < port < pid < tuple < map < nil < list < binary

## Purpose

The interpreter needs term comparison for every pattern match, every guard, every case expression. Gleam compiles exhaustive case expressions into select_val/is_eq_exact chains that depend on correct equality and ordering. Without comparison, the interpreter cannot branch. This brief delivers both == semantics (number coercion) and =:= semantics (exact), plus the full BEAM term ordering that select_val relies on.

## Task

Implement term comparison in crates/beamr/src/term/compare.rs. Three operations: equality (== semantics, where 1 == 1.0), exact equality (=:= semantics, where 1 =/= 1.0), and ordering (the BEAM term order for < and > comparisons). Equality and ordering must handle both immediate and boxed terms. Structural comparison for boxed terms recurses into elements (tuple element-by-element, list head then tail, map by sorted keys then values). Implement PartialEq, Eq (for exact equality), and PartialOrd/Ord (for BEAM ordering) on Term.

## Requirements

### R1: Exact equality (=:= semantics)

beamr::term::compare SHALL implement exact equality comparison for terms. Two terms are exactly equal if and only if they have the same tag AND the same value. Integer 1 and float 1.0 SHALL NOT be exactly equal. Two atoms with the same index SHALL be exactly equal. Two tuples SHALL be exactly equal if they have the same arity and all elements are exactly equal. Term SHALL implement PartialEq and Eq using exact equality semantics.

**Acceptance:**
- Term::small_int(1) == Term::small_int(1) is true
- Term::small_int(1) == Term::small_int(2) is false
- Term::atom(Atom::OK) == Term::atom(Atom::OK) is true
- Term::atom(Atom::OK) == Term::atom(Atom::ERROR) is false
- A small int term and a float term with the same numeric value are NOT equal under ==
- Two tuples {1, 2} and {1, 2} are equal; {1, 2} and {1, 3} are not

### R2: Numeric equality (== semantics)

beamr::term::compare SHALL provide a numeric_eq(a, b) function implementing == semantics where integer 1 and float 1.0 ARE equal. This is a separate function from PartialEq — PartialEq uses exact semantics. Numeric equality SHALL convert the integer to f64 for comparison when one operand is an integer and the other is a float. For non-numeric types, numeric_eq SHALL behave identically to exact equality.

**Acceptance:**
- numeric_eq(Term::small_int(1), float_term(1.0)) returns true
- numeric_eq(Term::small_int(1), float_term(1.5)) returns false
- numeric_eq(Term::small_int(1), Term::small_int(1)) returns true
- numeric_eq(Term::atom(Atom::OK), Term::atom(Atom::OK)) returns true
- numeric_eq(Term::small_int(1), Term::atom(Atom::OK)) returns false

### R3: BEAM term ordering

beamr::term::compare SHALL implement the BEAM term ordering: number < atom < reference < fun < port < pid < tuple < map < nil < list < binary. Within the same type: integers and floats are compared numerically (with cross-type coercion), atoms by index, tuples by arity then element-by-element, lists by head then tail, maps by sorted keys then values, binaries lexicographically. Term SHALL implement PartialOrd and Ord. It SHALL NOT deviate from the BEAM ordering — select_val depends on it.

**Acceptance:**
- Term::small_int(1) < Term::atom(Atom::OK) is true (number < atom)
- Term::atom(Atom::OK) < Term::pid(1) is true (atom < pid per BEAM order)
- Term::small_int(1) < Term::small_int(2) is true
- Term::small_int(2) < Term::small_int(1) is false
- Tuple {1, 2} < Tuple {1, 3} (element-by-element comparison)
- Tuple {1} < Tuple {1, 2} (shorter arity is less)
- A proper list [1, 2] < [1, 3] (head then tail comparison)

### R4: Structural comparison for boxed terms

WHEN comparing boxed terms for equality or ordering THE SYSTEM SHALL recurse into the structure: tuples compare element-by-element, lists compare head then tail, maps compare by sorted key sequence then values. The recursion SHALL terminate at immediate values or empty structures. It SHALL NOT stack-overflow on deeply nested structures — use an iterative approach or bounded recursion for lists.

**Acceptance:**
- Nested tuple {{1, 2}, 3} == {{1, 2}, 3} is true
- Nested tuple {{1, 2}, 3} == {{1, 3}, 3} is false
- List comparison [1, [2, 3]] < [1, [2, 4]] is true
- Map comparison respects sorted key order
- Comparing two lists of 10,000 cons cells does not stack overflow

## Boundaries

- SHALL NOT implement pattern matching dispatch — that is the interpreter's job using these comparison primitives
- SHALL NOT implement binary comparison optimizations (bitwise comparison for heap binaries) — basic lexicographic comparison is sufficient for Gate 1
- SHALL NOT modify the Term struct or tag layout — use the types from B-005 and B-006 as-is
