# B-071: Scheduler module splitting — architectural decomposition

> **Cluster:** beamr

## Purpose

scheduler/mod.rs is 1429 lines — nearly three times the 500-line limit. It contains the Scheduler struct definition, constructor variants, module management (hot code loading, purge), process spawning (8 variants), process lifecycle management, messaging helpers, execution coordination, namespace management, test helpers, and infrastructure accessors. These are distinct concerns that happen to share a struct. The module has grown organically as briefs landed — each new capability added methods to the monolith. The result is a file where finding any specific concern requires scanning through unrelated code. The existing submodules (run_queue.rs, steal.rs, timer_integration.rs, process_slot.rs) show the pattern works; mod.rs just hasn't been decomposed yet. This brief splits mod.rs into focused submodules while preserving the public API. No behavioral changes. The Scheduler struct stays in mod.rs as the central type; methods move to impl blocks in their respective submodules using Rust's split-impl pattern. Note: supervision_integration.rs is also 642 lines (over limit) but is a separate concern — this brief focuses on mod.rs only.

## Task

Split scheduler/mod.rs (1429 lines) into focused submodules using Rust's split-impl pattern: extract module management methods into module_management.rs, spawn variants into spawning.rs, execution and lifecycle methods into execution.rs, and test helpers into test_helpers.rs. The Scheduler struct definition stays in mod.rs. No behavioral changes.

## Requirements

### R1: Module management methods → module_management.rs

THE SYSTEM SHALL extract all hot code loading, purge, and module registry methods from scheduler/mod.rs into a new scheduler/module_management.rs file as an impl Scheduler block. THE SYSTEM SHALL move associated types HotLoadResult and PurgeResult with the methods.

**Acceptance:**
- New file: crates/beamr/src/scheduler/module_management.rs
- Methods moved (as impl Scheduler block in new file): hot_load_module, hot_load_module_in, load_module_in, purge_module, force_purge_module, purge_module_in, force_purge_module_in, lookup_module_in, delete_module, check_old_code, check_process_code
- Associated types moved: HotLoadResult, PurgeResult
- mod.rs retains use and mod declarations
- All module management tests continue to pass

**Files:**
- create: crates/beamr/src/scheduler/module_management.rs
- modify: crates/beamr/src/scheduler/mod.rs

### R2: Process spawning methods → spawning.rs

THE SYSTEM SHALL extract all spawn variants and link-at-spawn logic from scheduler/mod.rs into a new scheduler/spawning.rs file as an impl Scheduler block.

**Acceptance:**
- New file: crates/beamr/src/scheduler/spawning.rs
- Methods moved: spawn, spawn_process, spawn_trap_exit, spawn_in, spawn_in_trap_exit, spawn_link, spawn_link_dirty
- Internal helpers used only by spawn methods move with them
- All spawn tests continue to pass

**Files:**
- create: crates/beamr/src/scheduler/spawning.rs
- modify: crates/beamr/src/scheduler/mod.rs

### R3: Execution and lifecycle methods → execution.rs

THE SYSTEM SHALL extract process execution, wake/resume, and shutdown logic from scheduler/mod.rs into a new scheduler/execution.rs file as an impl Scheduler block. Thread pool management (thread_count, worker_names) SHALL remain in mod.rs as part of struct identity.

**Acceptance:**
- New file: crates/beamr/src/scheduler/execution.rs
- Methods moved: run_until_exit, wake_process, resume_process, wake_notifier, wake_with_result, terminate_process, shutdown, take_exit_error, take_exit_exception
- Thread pool management (thread_count, worker_names) stays in mod.rs (part of struct identity)
- All execution tests continue to pass

**Files:**
- create: crates/beamr/src/scheduler/execution.rs
- modify: crates/beamr/src/scheduler/mod.rs

### R4: Test helpers → test_helpers.rs

THE SYSTEM SHALL extract #[cfg(test)]-gated helper methods from scheduler/mod.rs into a new scheduler/test_helpers.rs file gated with #[cfg(test)].

**Acceptance:**
- New file: crates/beamr/src/scheduler/test_helpers.rs (gated with #[cfg(test)])
- Methods moved: spawn_test_process, spawn_test_process_in, spawn_linked_test_process, has_message, has_trapped_exit_message, enqueue_atom_message (if only used in tests — verify first)
- mod.rs reduced to ~250 lines: struct definition, constructors (new, with_code_server, with_code_server_and_policy), namespace management, infrastructure accessors (hook, timers, process_table, set_output_sink), process state queries (set_trap_exit, trap_exit, is_linked, process_namespace)

**Files:**
- create: crates/beamr/src/scheduler/test_helpers.rs
- modify: crates/beamr/src/scheduler/mod.rs

### R5: Verify line counts

WHEN all submodule extractions are complete, THE SYSTEM SHALL verify that scheduler/mod.rs is at most 300 lines, module_management.rs is at most 250 lines, spawning.rs is at most 300 lines, execution.rs is at most 250 lines, and test_helpers.rs is at most 200 lines.

**Acceptance:**

- scheduler/module_management.rs ≤ 250 lines
- scheduler/spawning.rs ≤ 300 lines
- scheduler/execution.rs ≤ 250 lines

- Note: supervision_integration.rs (642) and tests.rs (573) remain over limit — separate concern
