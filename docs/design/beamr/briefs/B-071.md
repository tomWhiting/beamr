# B-071: Scheduler module splitting — architectural decomposition

> **Cluster:** beamr
> **Depends on:** (none — pure restructuring)
> **Checklist:**
> - (no specific checklist item — structural health)

## Purpose

`scheduler/mod.rs` is 1429 lines — nearly three times the 500-line limit. It contains the Scheduler struct definition, constructor variants, module management (hot code loading, purge), process spawning (8 variants), process lifecycle management, messaging helpers, execution coordination, namespace management, test helpers, and infrastructure accessors. These are distinct concerns that happen to share a struct.

The module has grown organically as briefs landed — each new capability added methods to the monolith. The result is a file where finding any specific concern requires scanning through unrelated code. The existing submodules (`run_queue.rs`, `steal.rs`, `timer_integration.rs`, `process_slot.rs`) show the pattern works; `mod.rs` just hasn't been decomposed yet.

This brief splits `mod.rs` into focused submodules while preserving the public API. No behavioral changes. The Scheduler struct stays in `mod.rs` as the central type; methods move to `impl` blocks in their respective submodules using Rust's split-impl pattern.

Note: `supervision_integration.rs` is also 642 lines (over limit) but is a separate concern — this brief focuses on `mod.rs` only.

## Boundaries

- This brief SHALL decompose `scheduler/mod.rs` into submodules
- This brief SHALL NOT change any public API signatures
- This brief SHALL NOT change any behavior
- This brief SHALL NOT split `supervision_integration.rs` (642 lines — separate brief if needed)
- This brief SHALL NOT move the Scheduler struct definition out of `mod.rs`
