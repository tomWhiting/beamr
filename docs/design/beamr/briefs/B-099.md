# B-099: ETS ownership transfer — heir and give_away

> **Cluster:** beamr
> **Depends on:** B-094
> **Checklist:**
> - C150 — BIF registry maps (Module, Function, Arity) to a Rust function pointer

## Purpose

By default, an ETS table dies with its owner. The `{heir, Pid, Data}` option designates a successor — when the owner dies, the table is transferred to the heir instead of being deleted. `ets:give_away/3` transfers ownership explicitly. The heir receives an `{'ETS-TRANSFER', Tab, FromPid, Data}` message. OTP supervisors use this to preserve tables across process restarts.

## Task

Implement the heir option on ets:new, the ets:give_away/3 BIF, and the ETS-TRANSFER message delivery.

## Requirements

### R1: Heir option on ets:new

Heir option on ets:new

**Acceptance:**
- `{heir, Pid, HeirData}` parsed from ets:new options, stored in metadata
- `{heir, none}` explicitly disables heir (default)
- HeirData is an arbitrary term stored alongside the heir PID
- Invalid heir PID (non-existent process) is accepted at creation (checked at transfer time)
- Test: create table with heir option, verify metadata contains heir info

**Files:**
- modify: crates/beamr/src/ets/table.rs (metadata)
- modify: crates/beamr/src/ets/mod.rs (option parsing)

**Checklist:**
- C150 — BIF registry maps (Module, Function, Arity) to a Rust function pointer

### R2: Owner death transfers to heir

Modify the owner-death cleanup (B-090 R4) to check for an heir before deleting.

**Acceptance:**
- When owner dies and heir is set: transfer ownership to heir, deliver ETS-TRANSFER message
- When owner dies and heir is dead/nonexistent: delete the table (fallback)
- When owner dies and heir is None: delete the table (existing behavior)
- ETS-TRANSFER message format: `{'ETS-TRANSFER', TableId, FromPid, HeirData}` delivered to heir's mailbox
- After transfer, the table's owner field is updated to the heir's PID
- Test: owner dies, heir receives ETS-TRANSFER, table is accessible by heir

**Files:**
- modify: crates/beamr/src/scheduler/mod.rs (cleanup_exited_process)

**Checklist:**
- C150 — BIF registry maps (Module, Function, Arity) to a Rust function pointer

### R3: ets:give_away/3

ets:give_away/3

**Acceptance:**
- `ets:give_away(Tab, Pid, GiftData)` transfers ownership from caller to Pid
- Only the current owner can give away
- Recipient receives `{'ETS-TRANSFER', Tab, FromPid, GiftData}` message
- After give_away, caller can no longer write to a protected/private table
- Test: give_away to another process, verify ownership transfer and message delivery

**Files:**
- modify: crates/beamr/src/native/ets_bifs.rs

**Checklist:**
- C150 — BIF registry maps (Module, Function, Arity) to a Rust function pointer

## Boundaries

- SHALL NOT implement recursive heir chains (heir of heir) — if the heir is also dead, table is deleted
- SHALL NOT implement ets:setopts for changing heir after creation — heir is set at creation or via give_away
