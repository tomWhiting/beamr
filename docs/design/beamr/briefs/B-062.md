# B-062: raw_raise ‚Äî re-raise exceptions with class/reason/trace

> **Cluster:** beamr
> **Depends on:** B-059 (exceptions.rs, HandlerKind)
> **Checklist:**
> - C87 ‚Äî raise instruction: re-raise a caught exception (extends)

## Purpose

The existing `raise` instruction (opcode 108, landed in B-015) re-raises an exception using the class from `current_exception()` and explicit stacktrace/reason operands. It cannot change the exception class.

`raw_raise` (opcode 161) is the unrestricted form: it reads class, reason, and stacktrace from `x(0)`, `x(1)`, `x(2)` respectively and raises with all three explicitly specified. This is the instruction the compiler emits for `erlang:raise/3` and for re-raising in catch clauses where the original class must be preserved.

The instruction is already decoded (`Instruction::RawRaise`, zero operands) but falls through to `UnsupportedOpcode` in the dispatch table. Without it, any Erlang or Gleam code that uses `erlang:raise/3` or re-raises with an explicit class fails.

## Boundaries

- This brief SHALL implement `raw_raise` (opcode 161) as a dispatch to `raise_exception` with class/reason/stacktrace from x registers
- This brief SHALL validate the class argument (must be `error`, `throw`, or `exit`)
- This brief SHALL NOT modify `raise_exception` ‚Äî it uses the function as-is from B-059
- This brief SHALL NOT implement `build_stacktrace` (B-061) ‚Äî raw_raise accepts whatever stacktrace term it's given

## Requirements

### R1: raw_raise opcode handler (opcode 161)

Implement the `raw_raise` instruction. Reads exception class, reason, and stacktrace from x registers and raises.

**Acceptance:**
- `raw_raise(process)` reads:
  - `x(0)` = class (atom: `error`, `throw`, or `exit`)
  - `x(1)` = reason (any term)
  - `x(2)` = stacktrace (any term ‚Äî typically a list from `build_stacktrace`, but raw_raise doesn't validate its structure)
- Validates class is one of `Atom::ERROR`, `Atom::THROW`, or `Atom::EXIT_CLASS` (atoms added in B-059 R2). If not, returns `ExecError::Badarg`
- Calls `raise_exception(process, Exception { class, reason, stacktrace })` ‚Äî leveraging the existing dispatch logic including handler-kind-aware wrapping (B-059 R3) and raw stacktrace capture (B-061 R1, if landed)
- `Instruction::RawRaise` dispatched in `opcodes/mod.rs` to the new handler
- Test: raw_raise with class=error ‚Äî exception dispatched to handler, class preserved
- Test: raw_raise with class=throw ‚Äî exception dispatched, throw wrapping in catch handler
- Test: raw_raise with class=exit ‚Äî exception dispatched, exit wrapping in catch handler
- Test: raw_raise with invalid class (e.g., atom `badclass`) ‚Äî returns Badarg
- Test: raw_raise with no handler installed ‚Äî process exits

**Files:**

**Checklist:** C87

### R2: erlang:raise/3 BIF registration

Register `erlang:raise/3` as a BIF so Erlang code can call it via the standard BIF mechanism. The BIF implementation delegates to `raw_raise`'s logic.

**Acceptance:**
- `erlang:raise/3` registered in the BIF registry with args `(Class, Reason, Stacktrace)`
- NativeFn returns `Result<Term, Term>` (not `Result<Term, ExecError>` ‚Äî Larry's correction). The BIF returns `Err(reason)` to signal an exception.
- The BIF must pass both class AND stacktrace through the BIF‚Üíinterpreter boundary. Currently only `ProcessContext::set_exception_class()` exists. Add `ProcessContext::set_exception_stacktrace(trace: Term)` so the interpreter's BIF error path can construct `Exception { class: take_exception_class(), reason, stacktrace: take_exception_stacktrace() }` instead of defaulting stacktrace to `Term::NIL`.
- The BIF implementation: validate class, call `process.set_exception_class(class)` and `process.set_exception_stacktrace(stacktrace)`, return `Err(reason)`.
- The interpreter's BIF error handler reads both fields when constructing the Exception for `raise_exception`.
- Test: call `erlang:raise/3` via the BIF mechanism ‚Äî exception is raised with correct class/reason/stacktrace (not NIL stacktrace)
