# B-078: Dirty NIF dispatch and process migration

> **Cluster:** beamr
> **Depends on:** B-077 (dirty thread pools)
> **Checklist:**
> - C150 — BIF registry maps (Module, Function, Arity) to a Rust function pointer

## Purpose

With dirty thread pools in place (B-077), the interpreter needs to detect dirty NIF calls and route them correctly. When `call_external_target` encounters a `NativeEntry` with `dirty_kind = Some(Cpu|Io)`, the process must be suspended on the normal scheduler, the NIF executed on a dirty thread, and the process resumed on a normal thread with the result.

The ProcessSlot migration infrastructure (Present → Executing → Present) already enables hand-off between threads. This brief wires the interpreter to the dirty pool through that mechanism.

## Task

Modify `call_external_target` to detect dirty NIFs, suspend the process, submit a DirtyJob, and handle the async result. The normal scheduler thread is freed immediately — it does not block waiting for the dirty call.

## Boundaries

- SHALL NOT run dirty calls on normal scheduler threads — the whole point is freeing the normal thread
- SHALL NOT change the NativeFn signature — dirty NIFs use the same function signature as normal ones
- SHALL NOT implement dirty NIF preemption — dirty calls run to completion (no reduction counting on dirty threads)
- SHALL handle the case where the dirty NIF raises an exception (Err result)

## Requirements

### R1: Interpreter detects dirty NIF and yields

When `call_external_target` resolves a `ResolvedImportTarget::Native(entry)` with `dirty_kind.is_some()`, return a new `InstructionOutcome::DirtyCall` instead of executing the NIF inline.

**Acceptance:**
- New `InstructionOutcome::DirtyCall { entry: NativeEntry, args: Vec<Term>, kind: DirtySchedulerKind }` variant
- `call_external_target` checks `entry.dirty_kind` before executing the NIF
- If dirty: stores args, returns DirtyCall outcome (process yields to scheduler)
- If not dirty: existing inline execution (unchanged)
- Test: calling a dirty NIF returns DirtyCall outcome, not Continue

**Files:**
- modify: crates/beamr/src/interpreter/opcodes/core.rs (call_external_target)
- modify: crates/beamr/src/interpreter/mod.rs (InstructionOutcome)

**Checklist:** C150

### R2: Scheduler handles DirtyCall outcome

When the scheduler's execute_slice receives a DirtyCall outcome, it submits the job to the appropriate dirty pool and marks the process as waiting for the dirty result.

**Acceptance:**
- execute_slice matches DirtyCall, constructs a DirtyJob, submits to dirty_cpu or dirty_io pool based on kind
- The process is NOT placed back on a run queue — it waits for the dirty result
- The normal scheduler thread returns to its work loop (freed immediately)
- ProcessSlot transitions to a waiting-for-dirty state (or uses existing Executing metadata)
- Test: submit a dirty call, verify the normal thread is free to execute other processes

**Files:**
- modify: crates/beamr/src/scheduler/mod.rs (execute_slice)

**Checklist:** C150

### R3: Dirty result handling and process resumption

When the dirty thread completes the NIF call, the result must be delivered back and the process resumed on a normal scheduler thread.

**Acceptance:**
- The DirtyJob's result_sender delivers the NIF result (Ok or Err) back to the scheduler
- A background task or waker moves the completed process from dirty-waiting back to a normal run queue
- The BIF result is stored in x(0) (same as normal NIF execution)
- If the NIF returned Err, the exception is raised on the process (same as normal NIF error path)
- The process resumes execution on the next normal scheduler tick
- Test: dirty NIF returns 42 → process resumes with x(0)=42
- Test: dirty NIF returns Err(badarg) → process raises exception

**Files:**
- modify: crates/beamr/src/scheduler/mod.rs
- modify: crates/beamr/src/scheduler/dirty.rs

**Checklist:** C150

### R4: Integration test — dirty NIF round-trip

End-to-end test that registers a dirty NIF, calls it from BEAM code, and verifies the result.

**Acceptance:**
- Test registers a NIF with dirty_kind=Some(Cpu) that sleeps for 10ms then returns a value
- A BEAM process calls the dirty NIF
- The normal scheduler thread is NOT blocked for the 10ms (other processes can run)
- The calling process receives the correct result after the dirty call completes
- Test: concurrent dirty and normal execution — normal process makes progress while dirty NIF runs

**Files:**
- create: crates/beamr/tests/dirty_scheduler.rs

**Checklist:** C150
