## Purpose

The loader is the front door — nothing runs until we can read a .beam file off disk and turn it into structured data. This brief delivers the container parser (FOR1/BEAM header + chunk extraction) and all chunk decoders (atoms, instructions, strings, imports, exports, lambdas, literals, line info, operands). It does NOT resolve imports or register modules — that is B-004. The split keeps the parser pure-input and independently testable against real .beam files from gleam build.

## Task

Implement the .beam chunked binary parser in parser.rs and the chunk decoders in decode/chunks.rs, the compact term encoding decoder in decode/compact.rs, and the Instruction enum in decode/mod.rs. The parser reads the FOR1/BEAM header, iterates chunk headers, and extracts each chunk as a typed struct with raw byte payloads. The decoder processes each chunk type: registers atoms in the AtomTable, decodes the Code chunk into Instruction values using compact term encoding, decompresses and deserializes the LitT chunk, and parses ImpT/ExpT/FunT/StrT/Line chunks into structured data. Define the Instruction enum and all operand types. The loader module's public API is a load_beam_chunks() function that takes bytes and an AtomTable reference and returns a ParsedModule (all chunks decoded, imports unresolved).

## Requirements

### R1: FOR1/BEAM container parser

WHEN given a byte slice THE SYSTEM SHALL parse the .beam chunked binary format: verify the FOR1 header, verify the BEAM form type, and extract each chunk as a (tag, data) pair. IF the header is missing or the form type is not BEAM THEN THE SYSTEM SHALL return LoadError::InvalidFormat. It SHALL NOT attempt to interpret chunk contents — only extract them by tag.

**Acceptance:**
- parse_beam_chunks(valid_beam_bytes) returns a Vec of (FourCC, &[u8]) chunk pairs
- parse_beam_chunks(bytes_with_bad_magic) returns Err(LoadError::InvalidFormat)
- parse_beam_chunks(bytes_with_wrong_form_type) returns Err(LoadError::InvalidFormat)
- Chunk tags are preserved as 4-byte ASCII identifiers (e.g. b"AtU8", b"Code", b"StrT")

**Files:**
- modify: crates/beamr/src/loader/parser.rs

**Checklist:**
- C18 — Parses the .beam chunked binary container format (FOR1/BEAM header, chunk headers, chunk data)

### R2: Atom chunk decoder

WHEN the parser extracts an Atom or AtU8 chunk THE SYSTEM SHALL decode the atom table entries and register each atom in the provided AtomTable. The decoder SHALL return a Vec<Atom> preserving the file's atom ordering (index 0 is the module name). It SHALL NOT skip or silently drop malformed atom entries — each produces a LoadError::DecodeError.

**Acceptance:**
- decode_atom_chunk(chunk_bytes, &atom_table) returns Vec<Atom> with length matching the atom count header
- Each decoded atom is interned in the AtomTable and resolvable back to its string
- The first atom in the returned vec is the module name atom
- Truncated or malformed atom entries produce LoadError::DecodeError

**Files:**
- modify: crates/beamr/src/loader/decode/chunks.rs

**Checklist:**
- C19 — Decodes the Atom/AtU8 chunk and registers all atoms in the global atom table

### R3: ImpT, ExpT, FunT chunk decoders

THE SYSTEM SHALL decode the ImpT chunk into ImportEntry structs (module atom, function atom, arity), the ExpT chunk into ExportEntry structs (function atom, arity, label), and the FunT chunk into LambdaEntry structs (function atom, arity, label, num_free). All atom references SHALL be resolved against the file's local atom table (decoded in R2). IF an atom index is out of range THEN THE SYSTEM SHALL return LoadError::DecodeError.

**Acceptance:**
- decode_import_chunk returns Vec<ImportEntry> with module: Atom, function: Atom, arity: u8
- decode_export_chunk returns Vec<ExportEntry> with function: Atom, arity: u8, label: u32
- decode_lambda_chunk returns Vec<LambdaEntry> with function: Atom, arity: u8, label: u32, num_free: u32
- Out-of-range atom indices produce LoadError::DecodeError

**Files:**
- modify: crates/beamr/src/loader/decode/chunks.rs

**Checklist:**
- C22 — Decodes the ImpT chunk into import entries (module, function, arity)
- C23 — Decodes the ExpT chunk into export entries (function, arity, label)
- C24 — Decodes the FunT chunk into lambda entries (function, arity, label, num_free)

### R4: StrT and Line chunk decoders

THE SYSTEM SHALL decode the StrT chunk as a raw byte buffer (string table) and the Line chunk into source location entries (file name indices and line numbers). Both SHALL be stored in the ParsedModule for downstream use. IF the StrT or Line chunks are absent THE SYSTEM SHALL use empty defaults — their absence is not an error.

**Acceptance:**
- decode_string_chunk returns a Vec<u8> byte buffer
- decode_line_chunk returns a Vec<LineInfo> with file reference and line number
- Missing StrT chunk results in an empty byte buffer, not an error
- Missing Line chunk results in an empty Vec<LineInfo>, not an error

**Files:**

**Checklist:**

### R5: LitT chunk decoder (compressed literals)

WHEN the LitT chunk is present THE SYSTEM SHALL decompress its payload using zlib and deserialize the external term format entries into a Vec<Literal>. Each literal is a self-contained term value (integer, atom, tuple, binary, list, map) used as instruction operands. IF decompression fails or the external term format is malformed THEN THE SYSTEM SHALL return LoadError::DecodeError.

**Acceptance:**
- decode_literal_chunk decompresses a zlib-compressed payload and returns Vec<Literal>
- Integer, atom, binary, tuple, and list literals round-trip correctly
- Corrupted zlib data produces LoadError::DecodeError
- Malformed external term format produces LoadError::DecodeError

**Files:**
- modify: crates/beamr/Cargo.toml
- modify: crates/beamr/src/loader/decode/chunks.rs

**Checklist:**
- C25 — Decodes the LitT chunk by decompressing (zlib) and deserializing the external term format literals

### R6: Instruction enum and compact term encoding decoder

THE SYSTEM SHALL define an Instruction enum covering the BEAM opcodes that Gleam emits (label, func_info, move, call, call_only, call_ext, call_ext_only, call_last, call_ext_last, return, allocate, allocate_zero, deallocate, test_heap, put_list, put_tuple2, get_tuple_element, get_hd, get_tl, type tests, comparisons, select_val, select_tuple_arity, jump, gc_bif, bif, send, loop_rec, wait, try/catch, binary ops, map ops, make_fun2, call_fun, apply, badmatch, case_end, if_end, raise). Each instruction's operands SHALL be decoded using the BEAM compact term encoding (tag byte, variable-length integers, extended tags for literals/atoms/labels/registers). Unknown opcodes SHALL produce LoadError::DecodeError rather than being silently skipped. The Instruction enum SHALL be re-exported from beamr::loader so that downstream consumers (interpreter, validator, module struct) import it as beamr::loader::Instruction rather than coupling to beamr::loader::decode internals.

**Acceptance:**
- Instruction is a public enum in beamr::loader::decode
- decode_instructions(code_bytes, &atoms, &literals) returns Vec<Instruction>
- Compact term encoding correctly decodes: small integers (tag 0), atoms (tag 1), X registers (tag 2), Y registers (tag 3), labels (tag 4), extended literals (tag 7+0), extended list (tag 7+1)
- A known .beam file from gleam build decodes without errors
- Unknown or unsupported opcodes produce LoadError::DecodeError with the opcode number
- beamr::loader::Instruction is accessible without importing from beamr::loader::decode directly

**Files:**
- modify: crates/beamr/src/loader/decode/compact.rs
- modify: crates/beamr/src/loader/decode/mod.rs
- modify: crates/beamr/src/loader/mod.rs

**Checklist:**

### R7: Code chunk to instruction stream

WHEN the Code chunk is decoded THE SYSTEM SHALL produce a Vec<Instruction> indexed by label number for jump/call resolution. The instruction stream SHALL preserve label positions so that label N maps to a code offset. IF the Code chunk header (sub_size, instruction_set, opcode_max, label_count, function_count) contains unsupported values THEN THE SYSTEM SHALL return LoadError::DecodeError.

**Acceptance:**
- Code chunk header fields are parsed: instruction_set, opcode_max, label_count, function_count
- Labels in the instruction stream are addressable by their label number
- instruction_set value of 0 is accepted (standard BEAM)
- An unsupported instruction_set value produces LoadError::DecodeError

**Files:**
- modify: crates/beamr/src/loader/decode/mod.rs

**Checklist:**
- C20 — Decodes the Code chunk into a vector of internal Instruction values

### R8: ParsedModule struct and public load API

THE SYSTEM SHALL define a ParsedModule struct that aggregates all decoded chunks: module name (Atom), atoms (Vec<Atom>), instructions (Vec<Instruction>), imports (Vec<ImportEntry>), exports (Vec<ExportEntry>), lambdas (Vec<LambdaEntry>), literals (Vec<Literal>), string table (Vec<u8>), and line info (Vec<LineInfo>). The public API SHALL be a load_beam_chunks(bytes: &[u8], atom_table: &AtomTable) -> Result<ParsedModule, LoadError> function defined in loader/load.rs and re-exported from loader/mod.rs. It SHALL NOT resolve imports or store the module in a registry — that is B-004.

**Acceptance:**
- ParsedModule is a public struct with all fields listed above
- load_beam_chunks(valid_beam_bytes, &table) returns Ok(ParsedModule)
- load_beam_chunks(garbage_bytes, &table) returns Err(LoadError::InvalidFormat)
- ParsedModule.name is the first atom from the atom chunk
- ParsedModule.imports contains unresolved ImportEntry values (no resolution attempted)
- load_beam_chunks is defined in loader/load.rs, not in loader/mod.rs

## Boundaries

- SHALL NOT resolve imports against loaded modules or the BIF registry (B-004 scope)
- SHALL NOT store the parsed module in the module registry (B-004 scope)
- SHALL NOT validate instruction operands beyond basic decoding (B-004 scope)
- SHALL NOT implement Term values — literals are stored in a loader-local Literal type, not as runtime Terms (B-005/B-006 scope)
- SHALL NOT implement any BIFs or native functions (B-008 and later scope)
