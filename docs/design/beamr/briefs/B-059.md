# B-059: catch/catch_end — traditional exception handling

> **Cluster:** beamr
> **Depends on:** B-015
> **Checklist:**
> - C86 — try, try_end, try_case, try_case_end instructions: exception handling for Gleam's try expressions

## Purpose

BEAM has two exception handling mechanisms. The modern try/try_end/try_case family (landed in B-015) gives the catch clause access to the exception class, reason, and stacktrace as separate values. The traditional catch/catch_end mechanism (opcodes 62/63) is the older form used by Erlang's catch Expr expression — it wraps the exception into a single value in x(0) with different wrapping per class: throw:Value passes through unchanged, error:Reason becomes {'EXIT', {Reason, Stacktrace}}, exit:Reason becomes {'EXIT', Reason}. Both mechanisms push handlers onto the same ExceptionHandler stack on Process; the difference is entirely in how raise_exception delivers the caught value. The Instruction::Catch and Instruction::CatchEnd variants already decode correctly but fall through to UnsupportedOpcode. A second latent issue affects both try and catch: raise_exception does not unwind intermediate stack frames on exception, causing Y-register corruption at the handler label. This brief also extracts exception-handling code from messaging.rs (685 lines) into a focused exceptions.rs submodule.

## Task

Add a HandlerKind discriminant (Try/Catch) and stack_depth field to ExceptionHandler. Modify raise_exception to wrap values for Catch handlers according to exception class, truncate the stack to handler depth for both handler kinds, and dispatch to the handler label. Implement catch (opcode 62) and catch_end (opcode 63) handlers. Add common atoms for throw and exit exception classes. Extract all exception-handling functions from messaging.rs into a new exceptions.rs submodule.

## Requirements

### R1: HandlerKind discriminant and stack depth on ExceptionHandler

THE SYSTEM SHALL define a HandlerKind enum with variants Try and Catch. THE SYSTEM SHALL add kind: HandlerKind and stack_depth: usize fields to ExceptionHandler. WHEN the existing try_ handler pushes a handler, THE SYSTEM SHALL pass kind: HandlerKind::Try and stack_depth: process.stack().len().

**Acceptance:**

- ExceptionHandler gains kind: HandlerKind and stack_depth: usize
- ExceptionHandler derives remain (Copy, Clone, Debug, Eq, PartialEq)
- Existing try_ handler passes kind: HandlerKind::Try and stack_depth: process.stack().len()
- All existing tests compile and pass with the new fields

### R3: Catch-aware exception dispatch and stack unwinding

WHEN raise_exception dispatches an exception, THE SYSTEM SHALL truncate the process stack to handler.stack_depth before delivering the exception. WHEN the handler kind is Try, THE SYSTEM SHALL set current_exception, write exception.reason to handler.destination register, and jump to handler.catch_position. WHEN the handler kind is Catch, THE SYSTEM SHALL set current_exception, wrap the value based on exception class (throw: bare reason, error: {'EXIT', {Reason, Stacktrace}}, exit: {'EXIT', Reason}), write the wrapped value to x(0), and jump to handler.catch_position.

**Acceptance:**
- Stack::truncate(len: usize) added — delegates to Vec::truncate on the internal frames vector
- raise_exception truncates process stack to handler.stack_depth for BOTH handler kinds
- HandlerKind::Try: existing behaviour preserved (set exception, write reason to dest, jump)
- HandlerKind::Catch with throw class: x(0) = exception.reason (no allocation)
- HandlerKind::Catch with error class: x(0) = {'EXIT', {reason, stacktrace}} (two tuple allocations)
- HandlerKind::Catch with exit class: x(0) = {'EXIT', reason} (one tuple allocation)
- Unknown class treated as error (defensive)
- Tuple construction uses Atom::EXIT (uppercase) for the 'EXIT' key
- Test: Try handler preserves existing behaviour with stack truncation
- Test: Catch handler with error class wraps correctly
- Test: Catch handler with throw class passes reason through
- Test: Catch handler with exit class wraps correctly
- Test: exception in nested call unwinds intermediate stack frames

### R5: catch_end opcode handler (opcode 63)

WHEN the interpreter encounters Instruction::CatchEnd, THE SYSTEM SHALL pop the topmost exception handler, clear current_exception, and write Term::NIL to the Y-register referenced by the source operand. THE SYSTEM SHALL NOT modify x(0).

**Acceptance:**

- Instruction::CatchEnd dispatched in opcodes/mod.rs to the new handler
- Y-register referenced by source cleared to Term::NIL
