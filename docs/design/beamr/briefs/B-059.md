# B-059: catch/catch_end — traditional exception handling

> **Cluster:** beamr
> **Depends on:** B-015 (exception handling try/catch — landed)
> **Checklist:**
> - C86 — Exception handling instructions (extends to traditional catch form)

## Purpose

BEAM has two exception handling mechanisms. The modern `try`/`try_end`/`try_case` family (landed in B-015) gives the catch clause access to the exception class, reason, and stacktrace as separate values. The traditional `catch`/`catch_end` mechanism (opcodes 62/63) is the older form used by Erlang's `catch Expr` expression — it wraps the exception into a single value in `x(0)` with different wrapping per class:

- `throw:Value` → `Value` (passed through unchanged)
- `error:Reason` → `{'EXIT', {Reason, Stacktrace}}`
- `exit:Reason` → `{'EXIT', Reason}`

On the normal path (no exception), `x(0)` holds the expression result unchanged.

Both `catch` and `try` push handlers onto the same `ExceptionHandler` stack on Process. The difference is entirely in how `raise_exception` delivers the caught value. This means the shared `ExceptionHandler` struct needs a kind discriminant so the dispatch logic can choose the right delivery.

The `Instruction::Catch` and `Instruction::CatchEnd` variants already exist in the loader decoder (opcodes 62/63 decode correctly) but fall through to `UnsupportedOpcode` in the dispatch table. Any compiled code using `catch Expr` — including OTP library modules — hits this wall.

A second latent issue affects both try and catch: when an exception occurs inside a nested function call within the protected region, `raise_exception` jumps to the handler label but does not unwind the call stack. Stack frames from intermediate calls remain, causing Y-register accesses at the handler label to reference the wrong frame. This brief fixes that by recording stack depth when a handler is pushed and truncating on exception dispatch.

Finally, exception-handling code currently lives in `messaging.rs` (685 lines) alongside mailbox/receive operations. This brief extracts it into a focused `exceptions.rs` submodule.

## Boundaries

- This brief SHALL implement `catch` (opcode 62) and `catch_end` (opcode 63) dispatch and handlers
- This brief SHALL add a `HandlerKind` discriminant to `ExceptionHandler` and modify `raise_exception` to wrap values for Catch handlers
- This brief SHALL add stack depth tracking and truncation on exception dispatch (fixes both catch and try)
- This brief SHALL extract exception-handling code from `messaging.rs` into `exceptions.rs`
- This brief SHALL NOT implement `erlang:throw/1` or `erlang:exit/1` BIFs — those create throw/exit-class exceptions (B-060)
- This brief SHALL NOT implement `build_stacktrace` (B-061) or `raw_raise` (B-062)
- This brief SHALL NOT change the `Exception` struct — it already supports all three classes via the `class: Term` field

## Requirements

### R1: HandlerKind discriminant and stack depth on ExceptionHandler

Add a kind field to distinguish try and catch handlers, and a stack depth field for unwinding.

**Acceptance:**
- New enum `HandlerKind { Try, Catch }` in `process/mod.rs`
- `ExceptionHandler` gains `kind: HandlerKind` and `stack_depth: usize`
- `ExceptionHandler` derives remain (`Copy, Clone, Debug, Eq, PartialEq`)
- Existing `try_` handler updated to pass `kind: HandlerKind::Try` and `stack_depth: process.stack().len()`
- All existing tests compile and pass with the new fields

**Files:**
- modify: crates/beamr/src/process/mod.rs (ExceptionHandler, HandlerKind)

**Checklist:** C86

### R2: New common atoms for exception classes

Add `throw` and `exit` (lowercase) atoms for exception class comparison in the catch wrapping logic.

**Acceptance:**
- `Atom::THROW` added as common atom with string `"throw"`
- `Atom::EXIT_CLASS` added as common atom with string `"exit"` — distinguished from `Atom::EXIT` which is uppercase `"EXIT"` used in message tuples
- Both added to `COMMON_ATOMS` array with sequential indices after the current last entry (`Atom::MODULE` = 29)
- `Atom::THROW` = index 30, `Atom::EXIT_CLASS` = index 31

**Files:**
- modify: crates/beamr/src/atom/table.rs

**Checklist:** C86

### R3: Catch-aware exception dispatch and stack unwinding

Modify `raise_exception` to handle Catch handlers differently from Try handlers, and add stack unwinding for both.

**Acceptance:**
- `Stack::truncate(len: usize)` added — delegates to `Vec::truncate` on the internal frames vector
- On exception dispatch, `raise_exception` truncates the process stack to `handler.stack_depth` before delivering the exception (applies to BOTH Try and Catch handlers)
- For `HandlerKind::Try` (existing behaviour): set `current_exception`, write `exception.reason` to `handler.destination` register, jump to `handler.catch_position`
- For `HandlerKind::Catch`: set `current_exception`, wrap value based on exception class, write wrapped value to `x(0)`, jump to `handler.catch_position`:
  - Class is `throw` (`Atom::THROW`): `x(0) = exception.reason` (bare value, no allocation)
  - Class is `error` (`Atom::ERROR`): `x(0) = {'EXIT', {exception.reason, exception.stacktrace}}` (two tuple allocations on process heap)
  - Class is `exit` (`Atom::EXIT_CLASS`): `x(0) = {'EXIT', exception.reason}` (one tuple allocation)
  - Unknown class: treat as error (defensive, shouldn't happen)
- Tuple construction uses `Atom::EXIT` (uppercase) for the `'EXIT'` key in the wrapping tuples
- Test: raise exception with Try handler — existing behaviour preserved, stack truncated
- Test: raise exception with Catch handler, error class — x(0) gets `{'EXIT', {Reason, Stacktrace}}`
- Test: raise exception with Catch handler, throw class — x(0) gets reason directly
- Test: raise exception with Catch handler, exit class — x(0) gets `{'EXIT', Reason}`
- Test: exception inside nested function call — stack frames from intermediate calls are removed before jumping to handler

**Files:**
- modify: crates/beamr/src/process/stack.rs (Stack::truncate)
- modify: crates/beamr/src/interpreter/opcodes/messaging.rs (raise_exception — will move to exceptions.rs in R6)

**Checklist:** C86

- `Instruction::Catch` dispatched in `opcodes/mod.rs` to the new handler
- No `UnsupportedOpcode` error when executing modules containing opcode 62

### R5: catch_end opcode handler (opcode 63)

Implement the `catch_end` instruction handler. Functionally identical to `try_end` — pops the handler and clears the current exception. On the normal path (no exception), `x(0)` is left unchanged with the expression result. On the exception path, `x(0)` was already set by `raise_exception`'s catch wrapping logic (R3).

**Acceptance:**
- `catch_end(process, source)` pops the topmost exception handler and clears `current_exception`
- `Instruction::CatchEnd` dispatched in `opcodes/mod.rs` to the new handler
- The Y-register referenced by `source` is cleared (written to `Term::NIL`)
- Test: normal path — push catch handler, run catch_end, verify handler popped and x(0) unchanged
- Test: exception path — raise with catch handler (wraps x(0)), then catch_end clears exception state

**Files:**
- modify: crates/beamr/src/interpreter/opcodes/mod.rs (dispatch arm)
- modify: crates/beamr/src/interpreter/opcodes/messaging.rs (handler — will move to exceptions.rs in R6)

**Checklist:** C86
