## Purpose

Every Gleam case expression and function clause compiles into a sequence of type tests, comparisons, and multi-way branches. Without these opcodes, the interpreter can call functions and move data but cannot make any decisions. This brief delivers the branching infrastructure that turns the interpreter from a straight-line executor into one that can run real Gleam programs with pattern matching, guards, and conditional flow.

## Task

Implement the remaining interpreter opcodes for guards, type tests, and branching in crates/beamr/src/interpreter/opcodes.rs and crates/beamr/src/interpreter/pattern.rs. This includes: get_hd/get_tl for list decomposition, the full set of is_* type test instructions, is_eq_exact/is_ne_exact and is_lt/is_ge for comparison-and-branch, select_val and select_tuple_arity for multi-way dispatch, unconditional jump, and gc_bif/bif calls for guard expressions. All branch instructions follow the same pattern: test a condition, jump to a fail label if false, fall through if true.

## Requirements

### R1: List decomposition opcodes

WHEN the interpreter executes get_hd or get_tl THE SYSTEM SHALL extract the head or tail from a cons cell term into a destination register. IF the operand is not a cons cell THE SYSTEM SHALL raise a badarg error. It SHALL NOT copy the list — only extract the pointer to head or tail.

**Acceptance:**
- get_hd on a cons cell [1 | 2] stores 1 in the destination register
- get_tl on a cons cell [1 | 2] stores 2 in the destination register
- get_hd on a non-list term raises badarg

**Files:**
- modify: crates/beamr/src/interpreter/opcodes.rs

**Checklist:**
- C73 — get_hd and get_tl instructions: extract head or tail from a cons cell

### R2: Type test instructions

WHEN the interpreter executes a type test instruction (is_integer, is_float, is_atom, is_list, is_tuple, is_nil, is_binary, is_boolean, is_map, is_function2) THE SYSTEM SHALL check the term's tag and branch to the fail label if the test fails. IF the test passes THE SYSTEM SHALL fall through to the next instruction. It SHALL NOT modify the tested term or any register other than the instruction pointer.

**Acceptance:**
- is_integer on Term::small_int(1) falls through; on Term::atom(ok) jumps to fail label
- is_atom on Term::atom(ok) falls through; on Term::small_int(1) jumps to fail label
- is_tuple on a tuple term falls through; on a list term jumps to fail label
- is_nil on Term::NIL falls through; on Term::small_int(0) jumps to fail label
- is_boolean on Term::atom(Atom::TRUE) falls through; on Term::atom(Atom::OK) jumps to fail label
- is_function2 checks both function type and arity

**Files:**

### R3: Exact equality and inequality branch instructions

WHEN the interpreter executes is_eq_exact THE SYSTEM SHALL compare two terms using exact equality (=:= semantics from B-007) and branch to the fail label if they are not exactly equal. is_ne_exact SHALL branch if they ARE exactly equal. It SHALL NOT use numeric equality (==) — these instructions use =:= semantics.

**Acceptance:**
- is_eq_exact with small_int(1) and small_int(1) falls through
- is_eq_exact with small_int(1) and small_int(2) jumps to fail label
- is_eq_exact with small_int(1) and float(1.0) jumps to fail label (exact, not numeric)
- is_ne_exact with small_int(1) and small_int(2) falls through
- is_ne_exact with small_int(1) and small_int(1) jumps to fail label

**Files:**
- modify: crates/beamr/src/interpreter/opcodes.rs

**Checklist:**
- C75 — is_eq_exact and is_ne_exact instructions: exact equality comparison, branch on result

### R4: Ordering comparison branch instructions

WHEN the interpreter executes is_lt THE SYSTEM SHALL compare two terms using BEAM term ordering (from B-007) and branch to the fail label if the first is not less than the second. is_ge SHALL branch if the first is not greater than or equal to the second. It SHALL NOT deviate from the BEAM ordering.

**Acceptance:**
- is_lt with small_int(1) and small_int(2) falls through (1 < 2)
- is_lt with small_int(2) and small_int(1) jumps to fail label
- is_ge with small_int(2) and small_int(1) falls through (2 >= 1)
- is_ge with small_int(1) and small_int(2) jumps to fail label
- is_lt with small_int(1) and atom(ok) falls through (number < atom in BEAM order)

**Files:**
- modify: crates/beamr/src/interpreter/opcodes.rs

**Checklist:**
- C76 — is_lt and is_ge instructions: term ordering comparison, branch on result

### R5: Multi-way branch instructions

WHEN the interpreter executes select_val THE SYSTEM SHALL look up the source term in a list of value/label pairs and jump to the matching label, or jump to the fail label if no match. select_tuple_arity SHALL match on the source tuple's arity. Both SHALL use exact equality for value matching. It SHALL NOT fall through if no match — the fail label is mandatory.

**Acceptance:**
- select_val with atom(ok) matching {ok: L1, error: L2} jumps to L1
- select_val with atom(error) matching {ok: L1, error: L2} jumps to L2
- select_val with atom(unknown) matching {ok: L1, error: L2} jumps to fail label
- select_tuple_arity with a 2-tuple matching {2: L1, 3: L2} jumps to L1
- select_tuple_arity with a 4-tuple matching {2: L1, 3: L2} jumps to fail label

**Files:**

### R7: Guard BIF call instructions

WHEN the interpreter executes gc_bif1/gc_bif2/gc_bif3 THE SYSTEM SHALL call the named BIF with the provided arguments, store the result in the destination register, and branch to the fail label if the BIF returns Err. gc_bif variants MAY trigger GC before the call (via test_heap logic). bif0/bif1/bif2 SHALL call a guard BIF that cannot trigger GC. It SHALL NOT propagate BIF errors as process exits in guard context — guard failures branch to the fail label.

**Acceptance:**
- gc_bif2 calling erlang:+/2 with args [3, 4] stores 7 in the destination register
- gc_bif2 calling erlang:+/2 with args [atom, 1] jumps to the fail label (guard failure, not process exit)
- bif1 calling a guard BIF stores the result in the destination register
- Guard BIF failure branches to fail label, does NOT exit the process
