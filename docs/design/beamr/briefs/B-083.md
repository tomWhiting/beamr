# B-083: Process priorities — low/normal/high/max scheduling

> **Cluster:** beamr
> **Depends on:** B-032 (scheduler)
> **Checklist:**
> - C150 — BIF registry maps (Module, Function, Arity) to a Rust function pointer

## Purpose

BEAM processes have four priority levels: low, normal, high, and max. Higher-priority processes are scheduled before lower-priority ones. The scheduler dequeues from max first, then high, then normal, then low. This is used by OTP for time-critical processes (e.g., net_kernel, code_server). Without priorities, all processes compete equally — which is fine for most workflows but blocks OTP modules that set process priorities.

## Task

Add a priority field to Process, implement priority-aware run queue dequeuing, and register the `erlang:process_flag(priority, Level)` BIF extension.

## Boundaries

- SHALL NOT implement priority inheritance or priority inversion prevention
- SHALL NOT starve low-priority processes indefinitely — guarantee progress within a bounded window
- SHALL NOT change the work-stealing algorithm — priorities are per-queue, not per-steal

## Requirements

### R1: Priority field and enum

Add a `Priority` enum and a `priority` field to Process.

**Acceptance:**
- `Priority` enum: `Low`, `Normal`, `High`, `Max` (matching BEAM)
- Process gains `priority: Priority` field, default `Normal`
- Priority is settable at spawn and via process_flag
- Test: new process has Normal priority

**Files:**
- modify: crates/beamr/src/process/mod.rs

**Checklist:** C150

### R2: Priority-aware run queue dequeuing

Modify the run queue to dequeue higher-priority processes first.

**Acceptance:**
- Run queue maintains separate internal queues per priority level (or uses priority-based ordering)
- Dequeue order: Max → High → Normal → Low
- Low-priority processes run at least once every N scheduling rounds (anti-starvation, N configurable, default 8)
- Test: enqueue max and normal processes, verify max dequeues first
- Test: low-priority process makes progress alongside high-priority processes

**Files:**
- modify: crates/beamr/src/scheduler/run_queue.rs

**Checklist:** C150

### R3: process_flag(priority, Level) BIF extension

Extend the existing `erlang:process_flag/2` BIF to handle the `priority` flag.

**Acceptance:**
- `process_flag(priority, max)` sets the process priority to Max, returns previous priority
- `process_flag(priority, high)` sets High
- `process_flag(priority, normal)` sets Normal
- `process_flag(priority, low)` sets Low
- Invalid priority atom returns badarg
- Test: set priority to high, verify process_info(priority) returns high

**Files:**
- modify: crates/beamr/src/native/process_bifs/mod.rs (process_flag handler)

**Checklist:** C150
