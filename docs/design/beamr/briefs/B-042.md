# B-042: Process version pinning and safe version transitions

> **Cluster:** beamr
> **Depends on:** B-041
> **Checklist:**
> - C110 — Module registry maps module names to loaded Module structs
> - C130 — Process struct holds x-registers, y-registers, stack, mailbox, code position

## Purpose

When a module is hot-loaded, running processes must continue executing the version they started with until they make a fully-qualified external call (BEAM semantics). Currently, the interpreter re-resolves the module from the registry on every instruction cycle, meaning a process that yields mid-function will silently switch to the new version on resume — with an instruction pointer that indexes into the old code layout. This causes undefined behaviour: wrong instruction execution, out-of-bounds panics, or silent corruption.

Process version pinning ensures a process is bound to a specific module version and only transitions to the new version at well-defined safe points.

## Task

Pin processes to a specific module version during execution. Version transitions happen only at fully-qualified external call boundaries (call_ext with explicit module reference), not at arbitrary yield/resume points or intra-module calls.

## Boundaries

- This brief SHALL NOT implement closure version pinning (that's B-044)
- This brief SHALL NOT implement import re-resolution (that's B-043)
- This brief SHALL handle stack frame return addresses alongside code position pinning
- This brief SHALL ensure that intra-module calls (call, call_only, call_last) stay on the pinned version

## Requirements

### R1: Process module pin

Add a field to Process that pins it to a specific module version during execution. The pin holds an `Arc<Module>` so the old version stays alive even after it's replaced in the registry.

**Acceptance:**
- Process gains a `pinned_module: Option<Arc<Module>>` field
- When a process starts executing a function in module X, it pins to the current version of X from the registry
- `pinned_module` is set at process spawn and at fully-qualified external call entry
- `pinned_module` is cleared on process exit
- Accessor: `Process::pinned_module() -> Option<&Arc<Module>>`
- Mutator: `Process::pin_module(module: Arc<Module>)`

**Files:**
- modify: crates/beamr/src/process/mod.rs

**Checklist:** C130

### R2: Interpreter uses pinned module

Modify the interpreter run loop to use the pinned module instead of re-resolving from the registry on every iteration. The registry lookup is only used when the process needs to transition to a different module.

**Acceptance:**
- `run_loop` checks `process.pinned_module()` before falling back to `registry.lookup()`
- If `process.pinned_module()` matches the current `code_position.module`, use the pinned version
- If `code_position.module` differs from the pinned module's name (cross-module jump happened), resolve from registry and update the pin
- Intra-module calls (`call`, `call_only`, `call_last`) never trigger a registry re-lookup — they use the pinned version
- Test: load module A v1 with function `foo` that calls `bar` internally. Start process running `foo`. Mid-execution (between `foo` and `bar`), load module A v2. Process continues executing v1 for both `foo` and `bar`.

**Files:**
- modify: crates/beamr/src/interpreter/mod.rs (run_loop)

**Checklist:** C130

### R3: External calls trigger version transition

Fully-qualified external calls (`call_ext`, `call_ext_only`, `call_ext_last`) are the safe points where a process transitions to the current version of the target module.

**Acceptance:**
- When `call_external_target` resolves a `ResolvedImportTarget::Code` target, it looks up the **current** version from the registry (not the old version)
- The process's `pinned_module` is updated to the target module's current version
- If the process returns from the external call back to the calling module, it re-pins to the calling module — but picks up the current version of the calling module at that point (matching BEAM semantics: returning to a module after an external call uses the current version)
- Test: process in module A v1 calls module B. While in B, module A is reloaded to v2. When B returns, the process runs A v2 (not v1).

**Files:**
- modify: crates/beamr/src/interpreter/opcodes/core.rs (call_external_target)

**Checklist:** C130

### R4: Stack frame version-aware returns

Stack frames currently store `return_module: Atom` and `return_ip: usize`. On return, the module is re-resolved from the registry. If the module was reloaded, the return IP may be invalid for the new version. Stack frames must either pin a version or translate the return address.

**Acceptance:**
- `StackFrame` gains a `return_generation: u64` field storing the module generation at push time
- On `pop_frame`, if the registry's current version generation matches `return_generation`, use the return IP directly
- If the generation differs (module was reloaded), the return must be to a label-based address rather than a raw IP. Add `return_label: Option<u32>` to StackFrame, populated from the call instruction's continuation label. On generation mismatch, resolve `return_label` via the new module's label index.
- If no return_label is available and generation mismatches, the process exits with `ExitReason::Error` (code was purged under it — this matches BEAM behaviour where a process running old code that gets purged is killed)
- Test: push a stack frame for module A v1 at IP 42. Reload module A to v2 with same label layout. Pop frame — return succeeds via label resolution. Reload module A to v3 with different label layout — return resolves to the new label position.

**Files:**
- modify: crates/beamr/src/process/stack.rs (StackFrame, push_frame, pop_frame)
- modify: crates/beamr/src/interpreter/opcodes/core.rs (return_, call sites that push frames)

**Checklist:** C130

### R5: Scheduler slice entry respects pinning

The scheduler's `execute_slice` currently resolves the module at slice entry and passes it as `initial_module`. It should check the process's pinned module first.

**Acceptance:**
- `execute_slice` checks `process.pinned_module()` before doing a registry lookup
- If the process has a pinned module matching `code_position.module`, pass the pinned module to the interpreter
- If no pin exists (fresh process or after a version transition), resolve from registry and set the pin
- Test: start a process with a large reduction budget. Mid-execution, reload the module. The process continues on the pinned version until it yields and re-enters execute_slice. Verify the IP is still valid for the pinned version.

**Files:**
- modify: crates/beamr/src/scheduler/mod.rs (execute_slice)

**Checklist:** C130
