# B-023a: Process creation BIFs — self, spawn, spawn_link

> **Cluster:** beamr
> **Depends on:** B-010, B-018, B-021
> **Checklist:**
> - C51 — spawn(Module, Function, Args) creates a new process, pushes the initial call frame, assigns a unique Pid
> - C57 — Each process has a unique Pid that is never reused during a VM instance
> - C133 — link(PidA, PidB) creates a bidirectional link: both processes' link sets contain the other's Pid
> - C150 — BIF registry maps (Module, Function, Arity) to a Rust function pointer
> - C154 — Native function registration is available before any module is loaded
> **Stories:**
> - S2 (Workflow Author) — As a workflow author, I want each workflow step to run in its own isolated process so that a crash in one step does not corrupt the state of another.
> - S8 (Meridian Runtime) — As the Meridian runtime, I want to create a beamr VM instance, load .beam modules, and spawn processes from Rust so that I can embed workflow execution in the server process.

## Purpose

Running BEAM code cannot currently create new processes — there is no spawn BIF. The scheduler's `Scheduler::spawn()` exists at the Rust API level, but the interpreter has no way to reach it. This brief creates the `process_bifs.rs` file, extends `ProcessContext` with a spawn channel, and registers `self/0`, `spawn/3`, and `spawn_link/3`. These are the foundation for all process-lifecycle BIFs — subsequent briefs (B-023b, B-023c) build on the file and `ProcessContext` extensions established here.

## Task

Create `crates/beamr/src/native/process_bifs.rs` with three BIFs following the `NativeFn` signature. Extend `ProcessContext` with an optional spawn facility (channel or callback) so that `spawn/3` can request process creation from the scheduler. Register all three via `register_gate2_bifs()` called alongside `register_gate1_bifs()`.

## Boundaries

- This brief SHALL NOT implement spawn_monitor/3 (covered by combining spawn + monitor in later briefs)
- This brief SHALL NOT implement named process registration (erlang:register/2)
- This brief SHALL NOT implement spawn with options (erlang:spawn_opt/4)

## Requirements

### R1: erlang:self/0

`erlang:self/0` SHALL return the calling process's PID as a Pid term. The PID is available via `ProcessContext::pid()` which already returns `Option<u64>`. If pid is `None`, the BIF SHALL return `Err(badarg)`.

**Acceptance:**
- self() returns the correct PID for the calling process
- Result is a valid Pid term usable with send, link, and monitor
- Calling from a context without a PID returns badarg

**Files:**
- create: crates/beamr/src/native/process_bifs.rs
- modify: crates/beamr/src/native/mod.rs

**Checklist:**
- C57

### R2: erlang:spawn/3

`erlang:spawn/3` SHALL accept (Module, Function, Args) where Module and Function are atoms and Args is a list. It SHALL request process creation from the scheduler via the `ProcessContext` spawn channel, set the new process's code position to the export matching Module:Function/length(Args), load Args into the new process's X-registers, and return the new process's PID. Unresolved Module:Function SHALL return `Err(badarg)`.

**Acceptance:**
- spawn(Module, Function, []) returns a valid PID
- spawn(Module, Function, [A, B]) creates a process with x(0)=A, x(1)=B
- The spawned process executes independently on the scheduler
- Unresolved Module:Function returns badarg

**Files:**
- create: crates/beamr/src/native/process_bifs.rs
- modify: crates/beamr/src/native/mod.rs

**Checklist:**
- C51

### R3: erlang:spawn_link/3 and BIF registration

`erlang:spawn_link/3` SHALL behave identically to `spawn/3` but SHALL additionally establish a bidirectional link between the calling process and the spawned process before the spawned process begins execution. The link SHALL be atomic with the spawn.

All three BIFs SHALL be registered via `register_gate2_bifs()` in `process_bifs.rs`, following the `register_gate1_bifs()` pattern. The function SHALL be called from the same initialization path. `ProcessContext` SHALL be extended with a spawn facility without breaking existing Gate 1 BIFs.

**Acceptance:**
- spawn_link returns a valid PID with link already established
- Parent link set contains child PID and vice versa
- All 3 BIFs registered under the erlang module
- Import resolution resolves erlang:spawn/3, erlang:self/0, erlang:spawn_link/3
- ProcessContext extension does not break Gate 1 BIFs
- No panics in production code

**Files:**
- create: crates/beamr/src/native/process_bifs.rs
- modify: crates/beamr/src/native/mod.rs
- modify: crates/beamr/src/native/bifs.rs
