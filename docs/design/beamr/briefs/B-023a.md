# B-023a: Process creation BIFs — self, spawn, spawn_link

> **Cluster:** beamr
> **Depends on:** B-010, B-018, B-021
> **Checklist:**
> - C51 — spawn(Module, Function, Args) creates a new process, pushes the initial call frame, assigns a unique pid, and places it on a run queue
> - C57 — Each process has a unique Pid that is never reused during a VM instance
> - C133 — link(PidA, PidB) creates a bidirectional link: both processes' link sets contain the other
> - C150 — BIF registry maps (Module, Function, Arity) to a Rust function pointer
> - C154 — Native function registration is available before any module is loaded, so BIFs are present for import resolution
> **Stories:**
> - S2 (Workflow Author, Writing Gleam Workflows) — As a workflow author, I want each workflow step to run in its own isolated process so that a crash in one step does not corrupt the state of another.
> - S8 (Meridian Runtime, Embedding beamr for Workflow Execution) — As the Meridian runtime, I want to create a beamr VM instance, load .beam modules, and spawn processes from Rust so that I can embed workflow execution in the server process.

## Purpose

Running BEAM code cannot create new processes. This brief creates process_bifs.rs, extends ProcessContext with a spawn channel, and registers self/0, spawn/3, and spawn_link/3.

## Task

Create crates/beamr/src/native/process_bifs.rs with three BIFs. Extend ProcessContext with a spawn facility. Register via register_gate2_bifs().

## Requirements

### R1: erlang:self/0

erlang:self/0 SHALL return the calling process PID as a Pid term. If pid is None, return Err(badarg).

**Acceptance:**
- self() returns the correct PID
- Result is a valid Pid term
- Missing PID returns badarg

**Files:**
- create: crates/beamr/src/native/process_bifs.rs
- modify: crates/beamr/src/native/mod.rs

**Checklist:**
- C57 — Each process has a unique Pid that is never reused during a VM instance

**Stories:**
- S8 (Meridian Runtime, Embedding beamr for Workflow Execution) — As the Meridian runtime, I want to create a beamr VM instance, load .beam modules, and spawn processes from Rust so that I can embed workflow execution in the server process.

### R2: erlang:spawn/3

erlang:spawn/3 SHALL accept (Module, Function, Args), create a new process via ProcessContext spawn channel, set code position, load Args into X-registers, return new PID.

**Acceptance:**
- spawn(Module, Function, []) returns a valid PID
- spawn(Module, Function, [A, B]) creates process with x(0)=A, x(1)=B
- Spawned process executes independently on scheduler
- Unresolved Module:Function returns badarg

**Files:**
- create: crates/beamr/src/native/process_bifs.rs
- modify: crates/beamr/src/native/mod.rs

**Checklist:**
- C51 — spawn(Module, Function, Args) creates a new process, pushes the initial call frame, assigns a unique pid, and places it on a run queue

**Stories:**
- S2 (Workflow Author, Writing Gleam Workflows) — As a workflow author, I want each workflow step to run in its own isolated process so that a crash in one step does not corrupt the state of another.
- S8 (Meridian Runtime, Embedding beamr for Workflow Execution) — As the Meridian runtime, I want to create a beamr VM instance, load .beam modules, and spawn processes from Rust so that I can embed workflow execution in the server process.

### R3: erlang:spawn_link/3 and BIF registration

erlang:spawn_link/3 SHALL behave like spawn/3 but atomically establish a bidirectional link. All three BIFs registered via register_gate2_bifs(). ProcessContext extended without breaking Gate 1.

**Acceptance:**
- spawn_link returns a valid PID with link established
- Parent and child link sets contain each other
- All 3 BIFs registered under erlang module
- Import resolution resolves all three
- ProcessContext extension does not break Gate 1 BIFs
- No panics in production code

**Files:**
- create: crates/beamr/src/native/process_bifs.rs
- modify: crates/beamr/src/native/mod.rs
- modify: crates/beamr/src/native/bifs.rs
