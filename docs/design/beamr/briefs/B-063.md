# B-063: is_tagged_tuple opcode — record pattern matching guard

> **Cluster:** beamr
> **Depends on:** B-011 (pattern matching)
> **Checklist:**
> - C73 — Guard and branching opcodes dispatch correctly

## Purpose

The `is_tagged_tuple` opcode (opcode 159) is a compound guard test emitted by the compiler for record pattern matching. It checks three conditions in one instruction: (1) the term is a tuple, (2) the tuple has the expected arity, and (3) the first element is a specific atom (the record tag). Without it, any Erlang or Gleam code using record patterns hits `UnknownOpcode(159)` — a fatal execution error that crashes the process. This blocks all OTP modules that pattern-match on records.

Currently, opcode 159 is decoded via `type_test4` which produces `Instruction::Generic { opcode: 159, ... }`. The dispatch table routes all Generic variants to `UnknownOpcode` — the opcode never reaches the guard type test path. The `TypeTestOp::IsTaggedTuple` match arm in `matches_type()` is dead code.

## Task

Add a dedicated `Instruction::IsTaggedTuple` variant to the loader decoder, wire it into the dispatch table, and implement a standalone handler in `guards.rs`. This cannot use the existing `matches_type()` path because that function's signature `(op, value, arity)` has no slot for the tag atom parameter.

## Boundaries

- SHALL NOT route through the existing `matches_type()` function — the signature lacks a tag parameter; use a dedicated handler
- SHALL NOT duplicate `test_arity` or `is_tuple` logic inline — extract shared helpers if needed
- SHALL NOT modify the cons or map type test paths

## Requirements

### R1: Loader decode — dedicated Instruction variant

Replace the `type_test4` to `Generic` decode path for opcode 159 with a dedicated `Instruction::IsTaggedTuple { fail, value, arity, tag }` variant.

**Acceptance:**
- `Instruction::IsTaggedTuple` variant exists with `fail: Operand`, `value: Operand`, `arity: Operand`, `tag: Operand`
- Opcode 159 decodes to `Instruction::IsTaggedTuple` instead of `Instruction::Generic`
- Existing .beam files containing `is_tagged_tuple` load without errors
- The dead `TypeTestOp::IsTaggedTuple` variant and its `matches_type()` stub can be removed (or left as dead code for now)
- Test: verify the decoder produces the correct variant with all four operands

**Files:**
- modify: crates/beamr/src/loader/decode/code.rs
- modify: crates/beamr/src/loader/decode/instruction.rs

**Checklist:** C73

### R2: Dispatch wiring

Add `Instruction::IsTaggedTuple` to the dispatch table in `opcodes/mod.rs`, routing to a dedicated handler in `guards.rs`.

**Acceptance:**
- `Instruction::IsTaggedTuple { fail, value, arity, tag }` is matched in the dispatch table
- Dispatch routes to `guards::is_tagged_tuple()`
- No `UnknownOpcode(159)` error when executing modules containing `is_tagged_tuple`

**Files:**
- modify: crates/beamr/src/interpreter/opcodes/mod.rs

**Checklist:** C73

### R3: is_tagged_tuple handler implementation

Implement `is_tagged_tuple` as a dedicated function in `guards.rs`. The handler reads the value, checks if it is a tuple of the given arity with the tag atom at element 0. On failure, branches to the fail label. On success, continues.

**Acceptance:**
- `is_tagged_tuple(process, module, fail, value, arity, tag)` performs the compound check
- Value is read via `core::read_term()`, arity extracted as unsigned, tag extracted as atom
- Check: `Tuple::new(value)` succeeds, `tuple.arity() == arity`, `tuple.get(0) == tag_atom`
- On any check failure: jump to fail label (set code pointer to fail IP)
- On success: continue to next instruction
- Test: `is_tagged_tuple({ok, 42}, 2, ok)` succeeds
- Test: `is_tagged_tuple({error, reason}, 2, ok)` branches to fail (wrong tag)
- Test: `is_tagged_tuple({ok, 1, 2}, 2, ok)` branches to fail (wrong arity)
- Test: `is_tagged_tuple(not_a_tuple, 2, ok)` branches to fail
- Test: `is_tagged_tuple({}, 0, ok)` branches to fail (no element 0 to check)

**Files:**
- modify: crates/beamr/src/interpreter/opcodes/guards.rs (335 lines — room for ~30 more)

**Checklist:** C73

### R4: Integration test with compiled record pattern

Test is_tagged_tuple with a real compiled module that uses record patterns.

**Acceptance:**
- A test loads a module containing `case X of {ok, V} -> V; {error, R} -> R end`
- The compiled bytecode uses `is_tagged_tuple` for the pattern match
- Both branches execute correctly
- Nested record patterns work (tuple inside tuple)

**Files:**
- modify: crates/beamr/src/interpreter/opcodes/guards.rs (test section)

**Checklist:** C73
