# B-049: Capability-based native imports with deny-by-default policy

> **Cluster:** beamr
> **Checklist:**
> - C184 — Every registered native entry carries a capability class tag: Pure, Clock, Entropy, or ExternalIo
> - C185 — The loader evaluates each resolved native import against a host-supplied CapabilityPolicy at link time; denied imports are bound to a undef-trapping stub
> - C186 — The default capability policy grants only Pure natives; Clock, Entropy, and ExternalIo require explicit host grants
> **Stories:**
> - S31 (Meridian Runtime, Embedding beamr for Workflow Execution) — As the Meridian runtime, I want native function access gated by a capability policy so that untrusted Gleam code cannot reach shell commands or filesystem operations without explicit host authorization.

## Purpose

Any loaded .beam module can currently call meridian_ffi:run_cmd/1 (arbitrary shell) and read_file/write_file (unrestricted filesystem) with no capability check — the module inherits the full ambient authority of the host process. For a runtime aimed at executing untrusted Gleam, this voids the safety property entirely. This brief introduces a capability taxonomy on every registered native, a CapabilityPolicy trait evaluated at import resolution, and a deny-by-default least-authority policy so that dangerous operations require explicit host grants.

## Task

Add a Capability enum to native entries, a CapabilityPolicy trait with a LeastAuthorityPolicy default, and wire the policy into the loader's import resolution. Denied imports are bound to a undef-trapping stub (not panic, not silent success). Tag all existing natives with their capability class. The host can supply a custom policy to grant broader capabilities when needed. Scope: beamr core only — beamr-meridian wiring is Scott's responsibility.

## Requirements

### R1: Capability enum

THE SYSTEM SHALL define a Capability enum with variants Pure, Clock, Entropy, and ExternalIo, carried by every NativeEntry.

**Acceptance:**
- Capability enum in crates/beamr/src/native/capability.rs with #[derive(Debug, Clone, Copy, PartialEq, Eq, Hash)]
- Pure: no side effects, no external I/O, no nondeterminism (arithmetic, type checks, list ops)
- Clock: reads wall-clock time (erlang:system_time, os:timestamp, timer functions)
- Entropy: consumes randomness (rand:uniform, crypto operations)
- ExternalIo: shell commands, filesystem, network, environment variables
- NativeEntry gains a capability: Capability field

**Files:**
- create: crates/beamr/src/native/capability.rs
- modify: crates/beamr/src/native/mod.rs

**Checklist:**

### R2: Tag all existing natives with capability class

THE SYSTEM SHALL tag every registered BIF and NIF with its correct Capability class. THE SYSTEM SHALL default to Pure for arithmetic, comparison, type-test, list, tuple, map, binary, and guard BIFs. THE SYSTEM SHALL tag time-related BIFs as Clock, rand BIFs as Entropy, and meridian_ffi/os:* BIFs as ExternalIo.

**Acceptance:**
- All register_* functions pass a Capability parameter or use a tagged registration helper
- erlang:+/2, erlang:length/1, lists:sort/1, etc. are Pure
- erlang:system_time/0, timer:send_after/3, os:timestamp/0 are Clock
- rand:uniform/0, rand:uniform/1 are Entropy
- meridian_ffi:run_cmd/1, meridian_ffi:read_file/1, meridian_ffi:write_file/2, os:getenv/1, os:putenv/2 are ExternalIo
- No native is registered without an explicit capability tag — the API requires it

### R3: CapabilityPolicy trait and LeastAuthorityPolicy

THE SYSTEM SHALL define a CapabilityPolicy trait with a single method: fn is_granted(&self, capability: Capability) -> bool. THE SYSTEM SHALL provide a LeastAuthorityPolicy that grants only Pure. THE SYSTEM SHALL provide an AllCapabilitiesPolicy that grants everything (for trusted embedders).

**Acceptance:**
- CapabilityPolicy trait in capability.rs with is_granted(&self, Capability) -> bool
- LeastAuthorityPolicy: is_granted returns true only for Capability::Pure
- AllCapabilitiesPolicy: is_granted returns true for all variants
- CapabilitySet struct for building custom policies: from_slice(&[Capability]), grants(Capability) -> bool
- All policies are Send + Sync for use across scheduler threads

**Files:**
- modify: crates/beamr/src/native/capability.rs

### R4: Denial stub for denied imports

WHEN a native import's capability is not granted by the active policy, THE SYSTEM SHALL bind the import to a denial stub that returns ExecError::Undef when called. THE SYSTEM SHALL NOT panic, silently succeed, or leave the import unresolved.

**Acceptance:**
- denial_stub function in capability.rs: fn(&[Term], &mut ProcessContext) -> Result<Term, Term> that returns Err(undef atom)
- DenialMode enum with Undef variant (extensible for future Trap variant)
- The stub preserves the correct arity so it occupies the import table slot correctly
- A test calling a denied native receives a {undef, {Module, Function, Args}} error, not a panic

### R5: Policy evaluation at import resolution

WHEN the loader resolves native imports, THE SYSTEM SHALL evaluate each resolved NativeEntry against the active CapabilityPolicy. IF the policy denies the capability, THE SYSTEM SHALL replace the resolved target with the denial stub. THE SYSTEM SHALL log denied imports in the unresolved-import report with reason 'capability denied'.

**Acceptance:**
- resolve_imports in load.rs accepts a &dyn CapabilityPolicy parameter
- For each resolved Native import, the entry's capability is checked against the policy
- Denied entries are replaced with ResolvedImportTarget::Native pointing to the denial stub
- The unresolved-import report includes denied imports with their capability class and denial reason
- load_module's public API accepts an optional policy (defaults to LeastAuthorityPolicy)

## Boundaries

- SHALL NOT implement per-process or per-module capability policies — the policy is per-scheduler (VM-wide)
- SHALL NOT implement runtime capability revocation — capabilities are checked once at import resolution
- SHALL NOT remove or disable meridian_ffi NIFs — they are tagged ExternalIo and denied by policy, not deleted
- SHALL NOT implement capability inheritance or delegation between processes
- SHALL NOT change the default policy for existing embedders — AllCapabilitiesPolicy is the backwards-compatible default

- AllCapabilitiesPolicy allows all natives (backwards compatibility)
- Denied import produces undef error at call time, not panic
