# B-085: spawn_opt/2,4 — spawn with options

> **Cluster:** beamr
> **Depends on:** B-084, B-083
> **Checklist:**
> - C150 — BIF registry maps (Module, Function, Arity) to a Rust function pointer

## Purpose

`erlang:spawn_opt/2` and `erlang:spawn_opt/4` spawn a process with a list of options: `link`, `monitor`, `{priority, Level}`, `{min_heap_size, Words}`, `{fullsweep_after, N}`. OTP supervisors use spawn_opt to configure child processes at spawn time. Without it, supervisor modules that pass spawn options fail.

## Task

Implement spawn_opt/2 (fun + options) and spawn_opt/4 (MFA + options) as BIFs that parse the option list and configure the spawned process accordingly.

## Requirements

### R1: Option list parsing

Parse the spawn options list into a `SpawnOptions` struct.

**Acceptance:**
- `SpawnOptions` struct with fields: `link: bool`, `monitor: bool`, `priority: Option<Priority>`, `min_heap_size: Option<usize>`
- Parse function extracts known options from a proper list, ignoring unknown atoms (BEAM silently ignores unknown options)
- `link` atom in list → link=true
- `monitor` atom in list → monitor=true
- `{priority, Level}` tuple → priority=Some(Level)
- `{min_heap_size, N}` tuple → min_heap_size=Some(N)
- Test: parse `[link, monitor, {priority, high}]` → correct SpawnOptions

**Files:**
- modify: crates/beamr/src/native/process_bifs/mod.rs

**Checklist:**
- C150 — BIF registry maps (Module, Function, Arity) to a Rust function pointer

### R2: erlang:spawn_opt/4

Register `erlang:spawn_opt/4` accepting (Module, Function, Args, Options). Returns Pid or {Pid, Ref} depending on monitor option.

**Acceptance:**
- Without monitor option: returns Pid (same as spawn/3)
- With monitor option: returns {Pid, Ref} (same as spawn_monitor/3)
- With link option: link established atomically (same as spawn_link/3)
- With priority option: child process starts with specified priority
- With min_heap_size option: child process heap starts at specified size
- Test: spawn_opt with [link, {priority, high}] → child is linked and high-priority

**Files:**
- modify: crates/beamr/src/native/process_bifs/mod.rs

**Checklist:**
- C150 — BIF registry maps (Module, Function, Arity) to a Rust function pointer

### R3: erlang:spawn_opt/2

Register `erlang:spawn_opt/2` accepting (Fun, Options). Same semantics as spawn_opt/4 but with a fun.

**Acceptance:**
- Accepts a fun term + options list
- Applies all options from R1
- Return type depends on monitor option (Pid or {Pid, Ref})
- Test: spawn_opt with a closure and [monitor] → returns {Pid, Ref}

**Files:**
- modify: crates/beamr/src/native/process_bifs/mod.rs

**Checklist:**
- C150 — BIF registry maps (Module, Function, Arity) to a Rust function pointer

## Boundaries

- SHALL NOT implement all spawn_opt options — only: link, monitor, {priority, Level}, {min_heap_size, Words}
- SHALL NOT implement {fullsweep_after, N} — GC tuning is out of scope for now
- SHALL NOT implement {message_queue_data, off_heap} — message queue optimization is out of scope
