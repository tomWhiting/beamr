# B-093: ETS ordered_set table type — B-tree based

> **Cluster:** beamr
> **Depends on:** B-090, B-008
> **Checklist:**
> - C150 — BIF registry maps (Module, Function, Arity) to a Rust function pointer

## Purpose

`ordered_set` tables maintain entries sorted by key using Erlang's term ordering. Unlike hash-based sets, ordered_set supports range queries and ordered iteration (first/next/last/prev). It's backed by a B-tree for O(log n) operations. OTP uses ordered_set for time-ordered indexes and priority queues.

## Task

Implement `EtsOrderedSet` using a `BTreeMap<Term, Term>` wrapped with appropriate concurrency control.

## Requirements

### R1: EtsOrderedSet implementation

EtsOrderedSet implementation

**Acceptance:**
- `EtsOrderedSet` wraps `Mutex<BTreeMap<TermKey, Term>>` where `TermKey` implements `Ord` using term comparison
- `insert`, `lookup`, `delete_key`, `tab2list` all work with ordered semantics
- `tab2list()` returns tuples in key order
- Test: insert {3,c}, {1,a}, {2,b} → tab2list returns [{1,a},{2,b},{3,c}]

**Files:**
- create: crates/beamr/src/ets/ordered_set.rs

**Checklist:**
- C150 — BIF registry maps (Module, Function, Arity) to a Rust function pointer

### R2: TermKey wrapper for Ord

Create a wrapper that implements `Ord` for Term using the existing term comparison infrastructure.

**Acceptance:**
- `TermKey(Term)` implements `Eq`, `PartialEq`, `Ord`, `PartialOrd`
- Ordering matches Erlang's term ordering: number < atom < reference < fun < port < pid < tuple < map < nil < list < binary
- Test: TermKey(atom(a)) < TermKey(atom(b)), TermKey(1) < TermKey(atom(a))

**Files:**
- create: crates/beamr/src/ets/term_key.rs (or add to ets/mod.rs)

**Checklist:**
- C150 — BIF registry maps (Module, Function, Arity) to a Rust function pointer

### R3: Ordered iteration — first/next/last/prev

Add ordered traversal methods to EtsOrderedSet that B-095 will expose as BIFs.

**Acceptance:**
- `first() -> Option<Term>` returns the smallest key
- `next(key) -> Option<Term>` returns the next key after the given key
- `last() -> Option<Term>` returns the largest key
- `prev(key) -> Option<Term>` returns the previous key before the given key
- Returns None at the boundaries (no next after last, no prev before first)
- Test: insert 3 entries, verify first/next/last/prev traversal

**Files:**
- modify: crates/beamr/src/ets/ordered_set.rs

**Checklist:**
- C150 — BIF registry maps (Module, Function, Arity) to a Rust function pointer

## Boundaries

- SHALL NOT implement concurrent read/write access in this brief — that's B-098
- SHALL use standard library BTreeMap with a Mutex for thread safety (B-098 adds fine-grained locking later)
- SHALL reuse the existing term ordering from B-007
