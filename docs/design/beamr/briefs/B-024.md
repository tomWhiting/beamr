# B-024: Scheduler supervision integration — exit signal and DOWN propagation

> **Cluster:** beamr
> **Depends on:** B-018, B-019, B-023a, B-023b, B-023c
> **Checklist:**
> - C55 — Process exit deallocates its heap, removes it from the scheduler, and propagates exit signals to linked processes
> - C106 — Message arrival to a waiting process moves it back to a run queue
> - C134 — When a linked process exits, an exit signal with the exit reason is sent to all linked processes
> - C135 — A process receiving an exit signal with a non-normal reason exits with that reason (unless trap_exit is set)
> - C136 — A process receiving an exit signal with reason 'normal' is not terminated
> - C137 — A process with trap_exit=true receives exit signals as {'EXIT', Pid, Reason} messages in its mailbox
> - C138 — An exit signal with reason 'kill' terminates the target process unconditionally, even if it traps exits
> - C139 — A process killed by 'kill' propagates exit reason 'killed' (not 'kill') to its links
> - C140 — monitor(Pid) returns a reference and registers a monitor; the monitored process's death sends {'DOWN', Ref, process, Pid, Reason} message
> - C133 — link(PidA, PidB) creates a bidirectional link
> - C141 — demonitor(Ref) removes the monitor; no DOWN message sent after demonitor
> - C142 — unlink(PidA, PidB) removes the bidirectional link
> **Stories:**
> - S2 (Workflow Author) — As a workflow author, I want each workflow step to run in its own isolated process so that a crash in one step does not corrupt the state of another.
> - S6 (Workflow Author) — As a workflow author, I want to define supervision trees over my workflow steps so that failed steps are automatically restarted according to a declared strategy.
> - S13 (Meridian Runtime) — As the Meridian runtime, I want the VM to guarantee preemptive fairness via reduction counting so that no single workflow can monopolize CPU and starve other workflows.

## Purpose

The supervision data structures work correctly in isolation — `LinkSet::process_exited()` cascades exit signals, `MonitorSet::process_exited()` delivers DOWN messages, the signal semantics (trap_exit, kill, normal, cascade) are all tested. But the scheduler's `cleanup_exited_process()` does not call either of them. When a process exits on the scheduler, its links and monitors are silently discarded. This brief wires the scheduler to the supervision layer so that process death actually propagates signals to linked and monitored processes, completing the OTP supervision loop.

## Task

Modify `cleanup_exited_process()` in `crates/beamr/src/scheduler/mod.rs` to call `LinkSet::process_exited()` and `MonitorSet::process_exited()` before removing the process from the process table. Add `LinkSet` and `MonitorSet` to `SharedState`. Ensure that processes woken by exit signals or DOWN messages are moved from the wait set to a run queue. Handle the cascade case: if exit signal propagation causes additional processes to exit, those processes must also be cleaned up (iterative, not recursive).

## Boundaries

- This brief SHALL NOT implement supervisor behaviours (one_for_one, one_for_all, rest_for_one — those are Gleam library code per ADR-007)
- This brief SHALL NOT implement application controllers or application start/stop
- This brief SHALL NOT implement process registry cleanup on exit (erlang:unregister is out of scope)

## Requirements

### R1: SharedState holds LinkSet and MonitorSet

`SharedState` SHALL contain a `LinkSet` (from `supervision::link`) and a `MonitorSet` (from `supervision::monitor`). Both SHALL be wrapped in `Mutex` for thread-safe access from scheduler workers. The `Scheduler::new()` constructor SHALL initialize both as empty.

**Acceptance:**
- SharedState contains link_set: Mutex<LinkSet> and monitor_set: Mutex<MonitorSet>
- Scheduler::new() initializes both as empty
- Existing scheduler tests continue to pass

**Files:**
- modify: crates/beamr/src/scheduler/mod.rs

**Checklist:**
- C55 — Process exit propagates exit signals to linked processes

### R2: cleanup_exited_process propagates exit signals

When a process exits, `cleanup_exited_process()` SHALL:
1. Take the process's link set via `process.take_links()`
2. For each linked PID, call `LinkSet::deliver_exit_signal()` with the exit reason
3. If the linked process does not trap exits and the reason is non-normal, mark it for termination
4. If the linked process traps exits, enqueue `{EXIT, DeadPid, Reason}` in its mailbox
5. If a waiting process receives a message (EXIT or DOWN), wake it from the wait set

Exit signal propagation SHALL be iterative: if signal delivery causes another process to die, that process SHALL also be cleaned up in the same pass (worklist pattern, not recursion).

**Acceptance:**
- Linked process receives exit signal when partner dies
- Process with trap_exit=true receives {EXIT, Pid, Reason} message
- Process without trap_exit dies from non-normal linked exit
- Normal exit does not kill linked processes
- Kill signal terminates regardless of trap_exit
- Killed process propagates 'killed' (not 'kill')
- Cascade: A links B links C, A dies with error -> B dies -> C dies
- A waiting process woken by EXIT message moves to run queue

**Files:**
- modify: crates/beamr/src/scheduler/mod.rs

**Checklist:**
- C134 — Exit signal sent to all linked processes
- C135 — Non-normal signal terminates unless trapping
- C136 — Normal signal does not terminate
- C137 — trap_exit delivers as message
- C138 — kill terminates unconditionally
- C139 — kill propagates as 'killed'

### R3: cleanup_exited_process delivers DOWN messages

When a process exits, `cleanup_exited_process()` SHALL:
1. Query the `MonitorSet` for all monitors watching the exiting PID
2. For each monitor, construct a `{DOWN, Ref, process, Pid, Reason}` 5-tuple on the watcher's heap
3. Enqueue the DOWN message in the watcher's mailbox
4. Remove the monitor from the MonitorSet
5. If the watcher is in the wait set, wake it

**Acceptance:**
- Monitored process exit delivers DOWN to watcher
- DOWN message contains correct Ref, Pid, and Reason
- Watcher in wait set is woken by DOWN message
- Multiple monitors on same target each deliver independent DOWN messages
- Demonitored target death does not send DOWN

**Files:**

**Checklist:**

### R4: BIF integration with SharedState

The process BIFs from B-023 (link, unlink, monitor, demonitor, spawn_link) SHALL route through the `SharedState`'s `LinkSet` and `MonitorSet`. `ProcessContext` SHALL carry a reference to the shared supervision state so that BIF calls from the interpreter update the scheduler-owned sets, not per-process copies.

**Acceptance:**
- link/1 BIF adds to SharedState's LinkSet
- unlink/1 BIF removes from SharedState's LinkSet
- monitor/2 BIF adds to SharedState's MonitorSet
- demonitor/1 BIF removes from SharedState's MonitorSet
- spawn_link/3 atomically spawns and links via SharedState

- modify: crates/beamr/src/native/mod.rs

**Checklist:**

### R5: Integration tests

End-to-end tests SHALL verify the complete supervision loop: spawn a process, link or monitor it, cause it to exit, and verify the signal or DOWN message arrives at the watching process. Tests SHALL cover: normal exit (no cascade), error exit (cascade through links), trap_exit (message delivery), kill (bypass trap_exit), and monitor DOWN delivery.

**Acceptance:**
- Test: spawn + link + error exit -> linked process dies
- Test: spawn + link + normal exit -> linked process survives
- Test: spawn + link + trap_exit -> {EXIT, Pid, Reason} message received
- Test: spawn + link + kill -> bypasses trap_exit, propagates 'killed'
- Test: spawn + monitor + exit -> {DOWN, Ref, process, Pid, Reason} received
- All tests run on the scheduler (not unit-level data structure tests)
