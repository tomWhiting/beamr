# B-024: Scheduler supervision integration — exit signal and DOWN propagation

> **Cluster:** beamr
> **Depends on:** B-018, B-019, B-023a, B-023b, B-023c
> **Checklist:**
> - C55 — Process exit deallocates its heap, removes it from the scheduler, and propagates exit signals to linked processes
> - C106 — Message arrival to a waiting process moves it back to a run queue
> - C133 — link(PidA, PidB) creates a bidirectional link: both processes' link sets contain the other
> - C134 — When a linked process exits, an exit signal with the exit reason is sent to all linked processes
> - C135 — A process receiving an exit signal with a non-normal reason exits with that reason (unless it traps exits)
> - C136 — A process receiving an exit signal with reason 'normal' is not terminated
> - C137 — A process with trap_exit=true receives exit signals as {'EXIT', Pid, Reason} messages in its mailbox instead of dying
> - C138 — An exit signal with reason 'kill' terminates the target process unconditionally, even if it traps exits
> - C139 — A process killed by 'kill' propagates exit reason 'killed' (not 'kill') to its links
> - C140 — monitor(Pid) returns a reference and registers a monitor; the monitored process's death sends {'DOWN', Ref, process, Pid, Reason} to the monitor owner
> - C141 — demonitor(Ref) removes the monitor; no DOWN message is sent after demonitor
> - C142 — unlink(PidA, PidB) removes the bidirectional link; no exit signal is sent after unlink
> **Stories:**
> - S2 (Workflow Author, Writing Gleam Workflows) — As a workflow author, I want each workflow step to run in its own isolated process so that a crash in one step does not corrupt the state of another.
> - S6 (Workflow Author, Writing Gleam Workflows) — As a workflow author, I want to define supervision trees over my workflow steps so that failed steps are automatically restarted according to a declared strategy.
> - S13 (Meridian Runtime, Embedding beamr for Workflow Execution) — As the Meridian runtime, I want the VM to guarantee preemptive fairness via reduction counting so that no single workflow can monopolize CPU and starve other workflows.

## Purpose

The supervision data structures work correctly in isolation but the scheduler's cleanup_exited_process() does not call them. Process death silently discards links and monitors. This brief wires the scheduler to the supervision layer so process death propagates signals, completing the OTP supervision loop.

## Task

Modify cleanup_exited_process() to call LinkSet::process_exited() and MonitorSet::process_exited(). Add LinkSet and MonitorSet to SharedState. Ensure woken processes move from wait set to run queue. Handle cascade iteratively.

## Requirements

### R1: SharedState holds LinkSet and MonitorSet

SharedState SHALL contain LinkSet and MonitorSet wrapped in Mutex. Scheduler::new() SHALL initialize both as empty.

**Acceptance:**
- SharedState contains link_set: Mutex<LinkSet> and monitor_set: Mutex<MonitorSet>
- Scheduler::new() initializes both as empty
- Existing scheduler tests continue to pass

### R2: cleanup_exited_process propagates exit signals

When a process exits, cleanup_exited_process() SHALL take links, deliver exit signals, terminate non-trapping linked processes, enqueue EXIT messages for trapping processes, and wake waiting processes. Cascade SHALL be iterative via worklist pattern.

**Acceptance:**
- Linked process receives exit signal when partner dies
- Process with trap_exit=true receives {EXIT, Pid, Reason} message
- Process without trap_exit dies from non-normal linked exit
- Normal exit does not kill linked processes
- Kill signal terminates regardless of trap_exit
- Killed process propagates killed not kill
- Cascade: A links B links C, A dies -> B dies -> C dies
- Waiting process woken by EXIT message moves to run queue

**Files:**
- modify: crates/beamr/src/scheduler/mod.rs

**Checklist:**
- C134 — When a linked process exits, an exit signal with the exit reason is sent to all linked processes
- C135 — A process receiving an exit signal with a non-normal reason exits with that reason (unless it traps exits)
- C136 — A process receiving an exit signal with reason 'normal' is not terminated
- C137 — A process with trap_exit=true receives exit signals as {'EXIT', Pid, Reason} messages in its mailbox instead of dying
- C138 — An exit signal with reason 'kill' terminates the target process unconditionally, even if it traps exits
- C139 — A process killed by 'kill' propagates exit reason 'killed' (not 'kill') to its links

### R3: cleanup_exited_process delivers DOWN messages

When a process exits, cleanup_exited_process() SHALL query MonitorSet for watchers, construct {DOWN, Ref, process, Pid, Reason} on watcher heap, enqueue in mailbox, remove monitor, and wake waiting watchers.

**Acceptance:**
- Monitored process exit delivers DOWN to watcher
- DOWN message contains correct Ref, Pid, and Reason
- Watcher in wait set is woken by DOWN message
- Multiple monitors deliver independent DOWN messages
- Demonitored target death does not send DOWN
