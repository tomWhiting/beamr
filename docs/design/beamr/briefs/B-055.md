# B-055: Process slot sentinel — metadata accessible during scheduler execution

> **Cluster:** beamr
> **Stories:**
> - S8 (Meridian Runtime, Embedding beamr for Workflow Execution) — As the Meridian runtime, I want to create a beamr VM instance, load .beam modules, and spawn processes from Rust so that I can embed workflow execution in the server process.

## Purpose

When the scheduler takes a process body for execution via take_runnable_process, the process_bodies slot becomes None. Any concurrent access to process metadata (links, namespace_id, trap_exit) via with_process or direct slot lookup returns None, causing: (1) exit signals silently dropped, (2) spawn_link namespace lookup fails with Badarg, (3) parent-side links not established, (4) tombstones missed after store-back. Current fixes (process_namespaces DashMap, pending_links DashMap, tombstone-on-absent-body) are piecemeal — every with_process caller is vulnerable. This brief replaces the Option<ScheduledProcess> slot with an enum that leaves a metadata sentinel during execution, making links, namespace_id, and trap_exit accessible without the full process body.

## Task

Replace the process_bodies slot type from Mutex<Option<ScheduledProcess>> to Mutex<ProcessSlot> where ProcessSlot is an enum: Present(ScheduledProcess), Executing(ProcessMetadata), or Absent. When take_runnable_process extracts the body, it leaves Executing with a snapshot of the process's links, namespace_id, and trap_exit flag. All callers that currently match on Option<ScheduledProcess> are updated to handle the Executing variant. The process_namespaces and pending_links DashMaps added in 0.3.14 are removed — their data lives in the sentinel instead. Exit signal delivery, spawn_link namespace lookup, is_linked queries, and tombstone rechecks all work correctly during execution.

## Requirements

### R1: ProcessSlot enum and ProcessMetadata struct

THE SYSTEM SHALL define ProcessSlot as an enum with variants Present(ScheduledProcess), Executing(ProcessMetadata), and Absent. ProcessMetadata SHALL contain namespace_id: NamespaceId, links: Vec<u64>, and trap_exit: bool. THE SYSTEM SHALL replace Mutex<Option<ScheduledProcess>> with Mutex<ProcessSlot> in SharedState::process_bodies.

**Acceptance:**
- ProcessSlot enum defined with three variants
- ProcessMetadata struct contains namespace_id, links (clone of process links), trap_exit
- SharedState::process_bodies type changes to DashMap<u64, Mutex<ProcessSlot>>
- ProcessSlot::Absent is the default (empty slot)

**Files:**
- create: crates/beamr/src/scheduler/process_slot.rs
- modify: crates/beamr/src/scheduler/mod.rs

### R2: take_runnable_process leaves Executing sentinel

WHEN take_runnable_process extracts a process body, THE SYSTEM SHALL replace the slot with Executing(ProcessMetadata) containing a snapshot of the process's namespace_id, links, and trap_exit. THE SYSTEM SHALL NOT leave the slot as Absent/None.

**Acceptance:**
- take_runnable_process reads namespace_id, clones links, reads trap_exit from the process before taking it
- Slot transitions from Present(process) to Executing(metadata)
- The extracted Process is returned as before
- store_runnable_process transitions from Executing back to Present

**Files:**
- modify: crates/beamr/src/scheduler/mod.rs

### R3: process_exit_signal reads sentinel for executing processes

WHEN process_exit_signal encounters an Executing slot, THE SYSTEM SHALL read trap_exit from the sentinel to determine whether the process should be killed or receive an EXIT message. For Kill signals, THE SYSTEM SHALL set the tombstone. For non-Kill signals to trap_exit processes, THE SYSTEM SHALL queue a pending EXIT message.

**Acceptance:**
- process_exit_signal matches ProcessSlot::Executing and reads metadata.trap_exit
- Kill signal with Executing slot: tombstone set, links read from sentinel for cascade
- Non-Kill signal with Executing + trap_exit=true: EXIT message queued for delivery when body returns
- Non-Kill signal with Executing + trap_exit=false: tombstone set
- Normal exit signal with Executing: no action (matches BEAM semantics)

**Files:**
- modify: crates/beamr/src/scheduler/supervision_integration.rs

### R4: take_links_from reads sentinel for executing processes

WHEN take_links_from encounters an Executing slot, THE SYSTEM SHALL read and return the links from the sentinel metadata.

**Acceptance:**
- take_links_from matches ProcessSlot::Executing and returns metadata.links.clone()
- Exit cascade finds linked children even when source process is being executed

**Files:**
- modify: crates/beamr/src/scheduler/supervision_integration.rs

### R5: spawn_link and SpawnFacility use sentinel for namespace and links

WHEN SpawnFacility::spawn needs the parent namespace or link state, THE SYSTEM SHALL read from the Executing sentinel if the parent body is taken. Scheduler::spawn_link SHALL read namespace_id from the sentinel. THE SYSTEM SHALL add the child link to the sentinel's link set when the body is absent.

**Acceptance:**
- SchedulerSpawnFacility::caller_namespace reads from Executing sentinel
- Scheduler::spawn_link reads namespace_id from Executing sentinel (no Badarg)
- Parent-side link added to sentinel's links, merged back when body is stored
- process_namespaces and pending_links DashMaps removed from SharedState

**Files:**
- modify: crates/beamr/src/scheduler/mod.rs
- modify: crates/beamr/src/scheduler/supervision_integration.rs

### R6: store_runnable_process merges sentinel state

WHEN store_runnable_process stores a body back after execution, THE SYSTEM SHALL merge any links added to the Executing sentinel's link set into the process body before transitioning the slot to Present.

**Acceptance:**
- store_runnable_process reads the current slot (Executing)
- Links in sentinel but not in process body are added to the process
- Slot transitions from Executing to Present with the merged process
- If a tombstone was set during execution, cleanup_exited_process is called after store

**Files:**
- modify: crates/beamr/src/scheduler/mod.rs

### R7: Tombstone recheck after store-back closes execution race window

WHEN a process body is stored back after execution (Requeue, Wait, Suspended outcomes), THE SYSTEM SHALL check for tombstones set during the execution window and call cleanup_exited_process if found, BEFORE parking or requeuing the process.

**Acceptance:**
- Requeue path: after store_runnable_process, checks tombstone before queue.push
- Wait path: after store_runnable_process, checks tombstone before ws.waiting.insert
- Suspended path: after store_runnable_process, checks tombstone before ws.waiting.insert
- If tombstone found: cleanup_exited_process called, process not parked
- Race window between take and store is closed: tombstone set during execution is always caught

**Files:**
- modify: crates/beamr/src/scheduler/mod.rs

### R8: is_linked, process_namespace, with_process handle Executing variant

THE SYSTEM SHALL update all ProcessSlot consumers to handle the Executing variant. is_linked SHALL read links from the sentinel. process_namespace SHALL read namespace_id from the sentinel.

**Acceptance:**
- is_linked returns correct result when one or both processes are being executed
- process_namespace returns Some(namespace_id) when process is being executed
- with_process returns None for Executing (body not available for mutation)
- No caller silently drops information when encountering Executing

## Boundaries

- SHALL NOT change the external Scheduler API — all public methods remain unchanged
- SHALL NOT introduce new locking granularity — the existing Mutex<ProcessSlot> is sufficient
- SHALL NOT persist sentinel state across scheduler restarts — sentinel is transient execution metadata
- SHALL NOT change the interpreter or opcode dispatch — only scheduler-level process management

## Verification

- cargo test --workspace passes with all existing tests plus new execution-window tests
- process_namespaces and pending_links DashMaps are removed from SharedState
- Exit signals delivered correctly when target body is taken for execution
- spawn_link succeeds when parent body is taken for execution
- Tombstones set during execution window are caught before process is parked
- Links added during execution are visible after store-back
