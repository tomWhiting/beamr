# B-025: Loader hardening — EXPORT_EXT literals and multi-module loading

> **Cluster:** beamr
> **Depends on:** B-003
> **Checklist:**
> - C25 — Decodes the LitT chunk by decompressing (zlib) and deserializing the external term format literals
> - C31 — Stores a successfully loaded module in the module registry by its atom name
> - C32 — Loading a .beam file produced by gleam build + erlc succeeds for a pure Gleam module with no external dependencies beyond erlang: built-ins
> **Stories:**
> - S1 (Workflow Author, Writing Gleam Workflows) — As a workflow author, I want to write workflow steps as typed Gleam functions that compile to .beam bytecode so that I get compile-time type checking on my entire pipeline.
> - S8 (Meridian Runtime, Embedding beamr for Workflow Execution) — As the Meridian runtime, I want to create a beamr VM instance, load .beam modules, and spawn processes from Rust so that I can embed workflow execution in the server process.

## Purpose

The loader cannot load gleam@otp@actor.beam due to unsupported EXPORT_EXT (tag 113) in the LitT chunk. The CLI only loads single files. This brief adds EXPORT_EXT support and multi-module loading.

## Task

Add EXPORT_EXT decoding to the ETF literal decoder. Add --dir flag to CLI for multi-module loading.

## Requirements

### R1: EXPORT_EXT literal decoding

The ETF decoder SHALL handle tag 113 (EXPORT_EXT) and any other ETF tags encountered in gleam_erlang and gleam_otp .beam files.

**Acceptance:**
- gleam@otp@actor.beam loads without errors
- All gleam_otp .beam files load successfully
- EXPORT_EXT decodes to correct 3-tuples
- Existing fixtures still load correctly

**Files:**
- modify: crates/beamr/src/loader/decode/
