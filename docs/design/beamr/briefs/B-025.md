# B-025: Loader hardening — EXPORT_EXT literals and multi-module loading

> **Cluster:** beamr
> **Depends on:** B-003
> **Checklist:**
> - C25 — Decodes the LitT chunk by decompressing (zlib) and deserializing the external term format literals
> - C31 — Stores a successfully loaded module in the module registry by its atom name
> - C32 — Loading a .beam file produced by gleam build + erlc succeeds for a pure Gleam module
> **Stories:**
> - S1 (Workflow Author) — As a workflow author, I want to write workflow steps as typed Gleam functions that compile to .beam bytecode.
> - S8 (Meridian Runtime) — As the Meridian runtime, I want to create a beamr VM instance, load .beam modules, and spawn processes from Rust.

## Purpose

The loader cannot load `gleam@otp@actor.beam` because the LitT chunk contains an EXPORT_EXT (tag 113) literal — a `fun module:function/arity` reference. The ETF decoder does not handle this tag. Additionally, the CLI only loads a single .beam file. Real Gleam projects produce multiple .beam files (one per module) that cross-reference each other. This brief adds EXPORT_EXT support and multi-module loading so that gleam_otp modules can be loaded.

## Task

Add EXPORT_EXT (tag 113) decoding to the ETF literal decoder in `crates/beamr/src/loader/decode/`. EXPORT_EXT encodes `{Module, Function, Arity}` — decode it as a tuple term `{atom(Module), atom(Function), int(Arity)}`. Add a `--dir` flag to the CLI that loads all .beam files from a directory into the module registry before execution.

## Boundaries

- This brief SHALL NOT implement NEW_FUN_EXT (tag 112) — closures in literals are a separate concern
- This brief SHALL NOT implement FUN_EXT (tag 117) — deprecated format
- This brief SHALL NOT implement hot code loading or module versioning

## Requirements

### R1: EXPORT_EXT literal decoding

The ETF decoder SHALL handle tag 113 (EXPORT_EXT) by reading Module (atom), Function (atom), Arity (small integer) and constructing a 3-tuple `{Module, Function, Arity}` on the process heap. The decoder SHALL also handle any other ETF tags encountered in gleam_erlang and gleam_otp .beam files that are not yet supported.

**Acceptance:**
- `gleam@otp@actor.beam` loads without errors
- All .beam files from a compiled gleam_otp project load successfully
- ETF literals containing EXPORT_EXT decode to correct 3-tuples
- Existing proof.beam and hello.beam fixtures still load correctly

**Files:**
- modify: crates/beamr/src/loader/decode/

**Checklist:**
- C25

### R2: Multi-module loading in CLI

The CLI SHALL accept a `--dir <path>` flag that loads all .beam files from the specified directory into the module registry before resolving the entry function. Import resolution SHALL check the module registry for cross-module calls. The existing single-file mode SHALL continue to work.

**Acceptance:**
- `beamr --dir build/dev/erlang/*/ebin --entry module:function/arity` loads all modules then executes
- Cross-module imports resolve against loaded modules
- Single-file mode without --dir still works
- Missing --dir directory prints a clear error

**Files:**
- modify: crates/beamr-cli/src/main.rs

**Checklist:**
