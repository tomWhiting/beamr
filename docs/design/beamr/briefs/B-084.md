# B-084: spawn_monitor/1,3 — atomic spawn with monitor

> **Cluster:** beamr
> **Depends on:** B-021
> **Checklist:**
> - C150 — BIF registry maps (Module, Function, Arity) to a Rust function pointer

## Purpose

`erlang:spawn_monitor/1` and `erlang:spawn_monitor/3` atomically spawn a process and establish a monitor — there is no window where the child runs without the monitor. This is the recommended way to spawn monitored processes in OTP. Without it, code must call `spawn` then `monitor` separately, which has a race condition: the child could crash between spawn and monitor, and the DOWN message would be lost.

## Task

Implement spawn_monitor/1 (fun) and spawn_monitor/3 (MFA) as BIFs that atomically spawn + monitor.

## Requirements

### R1: erlang:spawn_monitor/3

Register `erlang:spawn_monitor/3` accepting (Module, Function, Args). Returns `{Pid, Ref}` tuple.

**Acceptance:**
- Atomically spawns a new process and establishes a monitor from caller to child
- Returns a 2-tuple `{ChildPid, MonitorRef}`
- The monitor is established BEFORE the child executes any instructions
- If the child crashes, the caller receives `{DOWN, Ref, process, Pid, Reason}`
- Test: spawn_monitor a process that crashes — DOWN received with correct Ref

**Files:**
- modify: crates/beamr/src/native/process_bifs/mod.rs
- modify: crates/beamr/src/native/mod.rs

**Checklist:**
- C150 — BIF registry maps (Module, Function, Arity) to a Rust function pointer

### R2: erlang:spawn_monitor/1

Register `erlang:spawn_monitor/1` accepting a fun (zero-arity closure or MFA export). Returns `{Pid, Ref}`.

**Acceptance:**
- Accepts a fun term (closure or MFA export)
- Spawns and monitors atomically, same semantics as spawn_monitor/3
- Test: spawn_monitor with a closure — returns {Pid, Ref}, DOWN received on crash

**Files:**
- modify: crates/beamr/src/native/process_bifs/mod.rs

**Checklist:**

## Boundaries

- SHALL NOT implement spawn_monitor/4 (with options) — that's spawn_opt territory (B-085)
- SHALL NOT allow any execution of the child before the monitor is established
- SHALL reuse existing spawn and monitor infrastructure
