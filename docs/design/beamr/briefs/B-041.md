# B-041: Dual-version module registry with purge detection

> **Cluster:** beamr
> **Depends on:** B-032
> **Checklist:**
> - C145 — Module registry stores loaded modules indexed by atom name
> - C146 — Module lookup by name returns the current version of the module or an error if not loaded
> - C147 — Loading a module with the same name as an already-loaded module replaces the current version
> - C148 — Function lookup by MFA (module, function, arity) resolves to a code pointer in the target module
> - C149 — Function lookup for an unloaded module or missing export returns an undef error

## Purpose

The module registry is currently single-version: loading a new version of an existing module silently drops the old one. For hot code loading, the BEAM keeps two versions of each module — the current version (used for new fully-qualified calls) and the old version (still executing in processes that started before the reload). The old version is purged only when no process references it. Without dual-version support, any module reload during workflow execution will corrupt running processes.

## Task

Replace the single-version ModuleRegistry with a dual-version registry that retains both current and old module versions, provides version-aware lookup, and supports safe purge of old code when no processes reference it. Also build a label-to-IP index on Module to replace the linear scan in label_ip.

## Requirements

### R1: ModuleVersions struct

WHEN a module is stored in the registry, THE SYSTEM SHALL wrap it in a ModuleVersions struct that holds both the current and optionally the old module version.

**Acceptance:**
- ModuleVersions struct with fields: current: Arc<Module>, old: Option<Arc<Module>>
- ModuleRegistry internal storage changes from DashMap<Atom, Arc<Module>> to DashMap<Atom, ModuleVersions>
- All existing registry methods (lookup, lookup_mfa, insert, insert_arc) continue to work, returning the current version by default
- All 620 existing tests pass without modification

**Files:**
- modify: crates/beamr/src/module.rs

**Checklist:**
- C145 — Module registry stores loaded modules indexed by atom name

### R2: Version-aware insert with old-version retention

WHEN inserting a new module version, THE SYSTEM SHALL promote the existing current to old and install the new version as current. IF an old version already exists, THE SYSTEM SHALL drop the previous old version.

**Acceptance:**
- insert(module) promotes current to old, inserts new as current
- If an old version already exists when a new version is inserted, the previous old version's Arc is dropped
- lookup(name) always returns the current version
- New method: lookup_old(name) returns the old version if it exists
- New method: module_version_count(name) returns 0 (not loaded), 1 (current only), or 2 (current + old)
- Test: load module A v1, verify lookup returns v1. Load module A v2, verify lookup returns v2, lookup_old returns v1. Load module A v3, verify lookup returns v3, lookup_old returns v2 (v1 dropped).

**Files:**
- modify: crates/beamr/src/module.rs

**Checklist:**
- C147 — Loading a module with the same name as an already-loaded module replaces the current version

### R3: Purge mechanism

WHEN purge_old is called for a module, THE SYSTEM SHALL remove the old version if its Arc::strong_count is 1 (only the registry holds it). IF processes still hold references, THE SYSTEM SHALL return PurgeError::StillReferenced.

**Acceptance:**
- New method: purge_old(name) removes the old version if its strong_count is 1
- PurgeError::StillReferenced { module: Atom, ref_count: usize } returned when processes still hold references
- PurgeError::NoOldVersion { module: Atom } returned when there is no old version to purge
- Test: load v1, load v2 (v1 becomes old). Clone Arc of old version (simulating a process holding it). Attempt purge — fails with StillReferenced. Drop the clone. Purge succeeds.
- After successful purge, lookup_old(name) returns None and module_version_count(name) returns 1
- Purge calls must be serialized (single code-server thread) to avoid TOCTOU race on strong_count

**Files:**
- modify: crates/beamr/src/module.rs

**Checklist:**
- C145 — Module registry stores loaded modules indexed by atom name

### R4: Module generation counter

WHEN a module is loaded, THE SYSTEM SHALL assign it a monotonically increasing generation counter so that different versions of the same module can be distinguished.

**Acceptance:**
- Module gains a generation: u64 field
- ModuleRegistry maintains a per-module generation counter; each insert increments it
- Module::generation() accessor returns the generation
- Two Arc<Module> instances for the same module name can be distinguished by generation
- CodePointer gains a generation: u64 field alongside its existing module and label

**Files:**
- modify: crates/beamr/src/module.rs

**Checklist:**
- C146 — Module lookup by name returns the current version of the module or an error if not loaded

### R5: Label-to-IP index

WHEN a module is loaded, THE SYSTEM SHALL build a HashMap<u32, usize> mapping label numbers to instruction indices. THE SYSTEM SHALL use this index for all label-to-IP resolution instead of linear scan.

**Acceptance:**
- Module gains a label_index: HashMap<u32, usize> field, populated during loading
- label_ip(module, label) becomes an O(1) HashMap lookup instead of O(n) linear scan
- Remove the duplicate label_ip implementation in scheduler/mod.rs — use a single implementation, either in core.rs or as a method on Module
- All existing tests pass
- Test: load a module, verify label_index contains correct mappings for all Label instructions in code

**Files:**
- modify: crates/beamr/src/module.rs
- modify: crates/beamr/src/loader/load.rs
- modify: crates/beamr/src/interpreter/opcodes/core.rs
- modify: crates/beamr/src/scheduler/mod.rs

**Checklist:**
- C148 — Function lookup by MFA (module, function, arity) resolves to a code pointer in the target module

## Boundaries

- SHALL NOT implement process version pinning (that is B-042)
- SHALL NOT implement import re-resolution (that is B-043)
- SHALL NOT implement the on_load opcode (that is B-045)
- SHALL implement a label-to-IP index on Module to replace the linear scan in label_ip as a prerequisite optimisation for hot code loading performance

## Verification

- cargo test --workspace passes with all 620+ tests
- ModuleRegistry stores and retrieves current and old versions correctly
- Purge succeeds when no references exist and fails when references exist
- label_ip uses HashMap lookup, linear scan implementation removed
- Module generation counters increment correctly across reloads
