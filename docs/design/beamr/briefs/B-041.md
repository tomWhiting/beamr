# B-041: Dual-version module registry with purge detection

> **Cluster:** beamr
> **Depends on:** B-032
> **Checklist:**
> - C110 — Module registry maps module names to loaded Module structs

## Purpose

The module registry is currently single-version: loading a new version of an existing module silently drops the old one. For hot code loading, the BEAM keeps two versions of each module — the "current" version (used for new fully-qualified calls) and the "old" version (still executing in processes that started before the reload). The old version is purged only when no process references it.

Without dual-version support, any module reload during workflow execution will corrupt running processes: their instruction pointers index into the old code layout, but the registry returns the new layout.

## Task

Replace the single-version ModuleRegistry with a dual-version registry that retains both current and old module versions, provides version-aware lookup, and supports safe purge of old code when no processes reference it.

## Boundaries

- This brief SHALL NOT implement process version pinning (that's B-042)
- This brief SHALL NOT implement import re-resolution (that's B-043)
- This brief SHALL NOT implement the on_load opcode (that's B-045)
- This brief SHALL implement a label-to-IP index on Module to replace the linear scan in label_ip — this is a prerequisite optimisation for hot code loading performance

## Requirements

### R1: ModuleVersions struct

Add a `ModuleVersions` wrapper that holds both the current and optionally the old module version.

**Acceptance:**
- `ModuleVersions` struct with fields: `current: Arc<Module>`, `old: Option<Arc<Module>>`
- ModuleRegistry internal storage changes from `DashMap<Atom, Arc<Module>>` to `DashMap<Atom, ModuleVersions>`
- All existing registry methods (`lookup`, `lookup_mfa`, `insert`, `insert_arc`) continue to work, returning the `current` version by default
- All 620 existing tests pass without modification

**Files:**
- modify: crates/beamr/src/module.rs

**Checklist:** C110

### R2: Version-aware insert with old-version retention

When inserting a new module version, the existing current becomes old, and the new version becomes current. If there was already an old version, it is dropped (only two versions are kept, matching BEAM semantics).

**Acceptance:**
- `insert(module)` promotes current to old, inserts new as current
- If an old version already exists when a new version is inserted, the previous old version's `Arc` is dropped
- `lookup(name)` always returns the current version
- New method: `lookup_old(name) -> Option<Arc<Module>>` returns the old version if it exists
- New method: `module_version_count(name) -> usize` returns 0 (not loaded), 1 (current only), or 2 (current + old)
- Test: load module A v1, verify lookup returns v1. Load module A v2, verify lookup returns v2, lookup_old returns v1. Load module A v3, verify lookup returns v3, lookup_old returns v2 (v1 dropped).

**Files:**
- modify: crates/beamr/src/module.rs

**Checklist:** C110

### R3: Purge mechanism

Provide a method to explicitly purge the old version of a module. Purge succeeds only when no external references to the old Arc exist (Arc::strong_count == 1, meaning only the registry holds it). If processes still reference the old version, purge returns an error indicating the module is still in use.

**Acceptance:**
- New method: `purge_old(name) -> Result<(), PurgeError>` removes the old version if its strong_count is 1
- `PurgeError::StillReferenced { module: Atom, ref_count: usize }` returned when processes still hold references
- `PurgeError::NoOldVersion { module: Atom }` returned when there is no old version to purge
- Test: load v1, load v2 (v1 becomes old). Clone Arc of old version (simulating a process holding it). Attempt purge — fails with StillReferenced. Drop the clone. Purge succeeds.
- After successful purge, `lookup_old(name)` returns None and `module_version_count(name)` returns 1

**Files:**
- modify: crates/beamr/src/module.rs

**Checklist:** C110

### R4: Module generation counter

Add a monotonically increasing generation counter to Module so that different versions of the same module can be distinguished.

**Acceptance:**
- `Module` gains a `generation: u64` field
- `ModuleRegistry` maintains a per-module generation counter; each `insert` increments it
- `Module::generation()` accessor returns the generation
- Two `Arc<Module>` instances for the same module name can be distinguished by generation
- CodePointer gains a `generation: u64` field alongside its existing `module` and `label`

**Files:**
- modify: crates/beamr/src/module.rs

**Checklist:** C110

### R5: Label-to-IP index

Build a `HashMap<u32, usize>` mapping label numbers to instruction indices at module load time. Replace the linear-scan `label_ip` function with an index lookup.

**Acceptance:**
- `Module` gains a `label_index: HashMap<u32, usize>` field, populated during loading
- `label_ip(module, label)` becomes an O(1) HashMap lookup instead of O(n) linear scan
- Remove the duplicate `label_ip` implementation in `crates/beamr/src/scheduler/mod.rs` — use the single implementation from `core.rs` or move it to `module.rs` as a method on Module
- All existing tests pass
- Test: load a module, verify label_index contains correct mappings for all Label instructions in code

**Files:**
- modify: crates/beamr/src/module.rs
- modify: crates/beamr/src/loader/load.rs
- modify: crates/beamr/src/interpreter/opcodes/core.rs (label_ip)
- modify: crates/beamr/src/scheduler/mod.rs (remove duplicate label_ip)

**Checklist:** C110
