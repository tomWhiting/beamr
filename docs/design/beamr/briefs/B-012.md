## Purpose

The scheduler is what makes fairness real. Without it, processes are data structures with no heartbeat. This brief delivers the scheduler thread pool (one thread per core by default), per-thread run queues, the schedule-execute-yield loop, work stealing (idle threads take half from the busiest queue), and clean shutdown. No async runtime in the hot path — plain OS threads plus lock-free queues. This is the piece that turns 'we have processes' into 'processes actually run fairly.'

## Task

Implement the scheduler in crates/beamr/src/scheduler/. The Scheduler creates N OS threads (default: available_parallelism). Each thread owns a RunQueue (lock-free deque). The main loop: pop a process from the run queue, set its status to Running, call the interpreter for one time slice (reduction budget), handle the result (Yielded → push back to run queue, Waiting → move to wait set, Exited → run exit handlers + remove). Work stealing: when a thread's queue is empty, it tries to steal half the processes from a random other thread's queue. Implement Scheduler::spawn(entry_module, entry_function, args) to create and enqueue a new process. Implement Scheduler::shutdown() for clean termination.

### R2: Per-thread run queue

beamr::scheduler::run_queue SHALL define a RunQueue backed by a lock-free deque (crossbeam-deque). The owning thread pushes and pops from the back (LIFO for cache locality). Stealers pop from the front (FIFO for fairness). RunQueue::push(process) adds a process. RunQueue::pop() removes and returns a process for execution. RunQueue::steal_half(target) takes approximately half the processes from target's queue.

**Acceptance:**
- push then pop returns the same process
- Stealing from a queue with 10 processes takes approximately 5
- Stealing from an empty queue returns nothing (no error, no panic)
- Push and steal from different threads do not race

**Files:**
- modify: crates/beamr/src/scheduler/run_queue.rs
- modify: crates/beamr/Cargo.toml

**Checklist:**

### R3: Schedule-execute-yield loop

EACH scheduler thread SHALL run a loop: (1) pop a process from its run queue (or steal if empty), (2) set process status to Running, (3) call the interpreter execute function for one time slice, (4) handle the result — Yielded: reset reductions, push back to run queue; Waiting: move to the global wait set; Exited: run cleanup. WHEN no process is available (own queue empty, steal fails) THE THREAD SHALL park briefly then retry.

**Acceptance:**
- A spawned process runs to completion
- A process that exhausts its reduction budget yields and re-runs later
- A process that enters receive with no messages transitions to Waiting
- An exited process is removed from the process table
- An idle thread does not spin-loop at 100% CPU (it parks)

### R4: Work stealing

WHEN a scheduler thread's run queue is empty THE SYSTEM SHALL attempt to steal work from another thread's queue. The victim is selected by round-robin (not random — deterministic for testing). Stealing takes approximately half the victim's processes. IF all queues are empty THE SYSTEM SHALL park the thread for a short duration before retrying. Work stealing SHALL NOT starve any single queue — a queue with only one process SHALL NOT be stolen from.

**Acceptance:**
- A thread with an empty queue steals from a thread with processes
- Stolen processes execute on the stealing thread
- A queue with 1 process is not stolen from (the owning thread keeps it)
- Round-robin victim selection visits all threads before repeating

### R5: Message arrival wakes waiting process

WHEN a message is delivered to a process in the Waiting state THE SYSTEM SHALL move it back to a run queue so the scheduler picks it up. The wake-up SHALL be triggered by the mailbox send path (B-011) notifying the scheduler. It SHALL NOT busy-poll — the transition is event-driven.

**Acceptance:**
- A Waiting process moves to a run queue when a message is delivered to its mailbox
- The wake-up is event-driven, not polled
- A Running or Yielded process receiving a message is not affected (already schedulable)

### R6: Clean shutdown

Scheduler::shutdown() SHALL signal all scheduler threads to stop, wait for in-progress processes to complete their current time slice, then join all threads. Processes still in run queues or wait sets at shutdown are abandoned (not executed). Shutdown SHALL complete in bounded time — it does not wait for blocked processes.

**Acceptance:**
- shutdown() returns after all threads have joined
- A process mid-execution finishes its current time slice before the thread exits
- shutdown() does not hang on a process in Waiting status
- Calling shutdown() twice does not panic
