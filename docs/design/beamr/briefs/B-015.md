## Purpose

Gleam uses closures extensively — every fn(x) { ... } compiles to make_fun2 + call_fun. Dynamic dispatch (apply) is used for behaviours and callback invocation. Map operations are used for Gleam records (which compile to Erlang maps). Without these opcodes, the interpreter cannot run any Gleam code that uses anonymous functions, callback patterns, or record access.

## Task

Implement closure, dynamic dispatch, and map opcodes in crates/beamr/src/interpreter/opcodes.rs. make_fun2 creates a closure term on the heap capturing free variables from registers. call_fun invokes a closure. apply/apply_last perform dynamic calls with module and function as runtime terms. has_map_fields tests for key presence, get_map_elements extracts values by key, and put_map_assoc/put_map_exact create or update maps.

## Requirements

### R1: Closure creation — make_fun2

WHEN the interpreter executes make_fun2 THE SYSTEM SHALL allocate a closure term on the process heap capturing: the module atom, function index, arity, and num_free free variables copied from X registers. The result SHALL be stored in x(0). It SHALL NOT capture more registers than num_free specifies.

**Acceptance:**
- make_fun2 with num_free=0 creates a closure with no captured variables
- make_fun2 with num_free=2 captures x(0) and x(1) as free variables
- The resulting closure term's arity matches the lambda table entry
- Closure::free_var(0) on the result returns the first captured value

### R2: Closure invocation — call_fun

WHEN the interpreter executes call_fun THE SYSTEM SHALL extract the closure from the register, verify the arity matches the number of arguments, restore the captured free variables, and jump to the closure's code entry point. IF the term is not a closure THE SYSTEM SHALL raise a badfun error. IF the arity doesn't match THE SYSTEM SHALL raise a badarity error.

**Acceptance:**
- call_fun on a valid closure with matching arity jumps to the closure's code
- Captured free variables are accessible in the closure's execution context
- call_fun on a non-closure term raises {badfun, Term}
- call_fun with wrong number of arguments raises {badarity, {Fun, Args}}

### R3: Dynamic dispatch — apply and apply_last

WHEN the interpreter executes apply THE SYSTEM SHALL read the module and function names from x(N) and x(N+1) (as atoms), look up the function in the module registry by MFA, and call it. apply_last SHALL do the same but as a tail call (deallocating the current frame). IF the module or function is not found THE SYSTEM SHALL raise an undef error. It SHALL NOT allow calling non-exported functions via apply.

**Acceptance:**
- apply with module=math, function=add, arity=2 calls math:add/2
- apply with a non-existent module raises {undef, {Module, Function, Args}}
- apply_last deallocates the current stack frame before jumping
- apply cannot call non-exported functions

### R4: Map field test — has_map_fields

WHEN the interpreter executes has_map_fields THE SYSTEM SHALL test that a map term contains all specified keys. IF any key is missing THE SYSTEM SHALL branch to the fail label. It SHALL NOT modify the map or any register other than the instruction pointer.

**Acceptance:**
- has_map_fields on a map with keys [a, b] checking for [a] falls through
- has_map_fields on a map with keys [a, b] checking for [a, c] jumps to fail label
- has_map_fields on a non-map term jumps to fail label

### R5: Map element extraction — get_map_elements

WHEN the interpreter executes get_map_elements THE SYSTEM SHALL extract values for the specified keys from a map and store them in destination registers. IF any key is missing THE SYSTEM SHALL branch to the fail label. It SHALL NOT modify the source map.

**Acceptance:**
- get_map_elements extracts key 'a' from map #{a => 1, b => 2} and stores 1 in the destination register
- Extracting a missing key jumps to fail label
- Multiple keys can be extracted in one instruction

### R6: Map construction and update — put_map_assoc/put_map_exact

WHEN the interpreter executes put_map_assoc THE SYSTEM SHALL create a new map by adding or updating key-value pairs in an existing map (or empty map). put_map_exact SHALL update existing keys only — if a key does not exist in the source map, it SHALL branch to the fail label. Both SHALL allocate the new map on the process heap. It SHALL NOT modify the source map in place — maps are immutable.

**Acceptance:**
- put_map_assoc on empty map with pairs [a=>1, b=>2] creates #{a => 1, b => 2}
- put_map_assoc on #{a => 1} with pairs [a=>2] creates #{a => 2} (update)
- put_map_exact on #{a => 1} with pairs [a=>2] creates #{a => 2}
- put_map_exact on #{a => 1} with pairs [b=>2] jumps to fail label (key b doesn't exist)
- Source map is not modified
