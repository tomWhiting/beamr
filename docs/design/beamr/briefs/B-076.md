# B-076: Binary append optimization — in-place append when refcount==1

> **Cluster:** beamr
> **Depends on:** B-072, B-075
> **Checklist:**
> - C150 — BIF registry maps (Module, Function, Arity) to a Rust function pointer

## Purpose

Binary construction via repeated append (`<<Acc/binary, NewByte>>`) is the most common binary-building pattern in Erlang/Gleam. Without optimization, each append allocates a new binary and copies all previous bytes — O(n^2) for building an n-byte binary. BEAM optimizes this: when a ProcBin has ref count 1 (only one reference exists) and has spare capacity, append writes directly into the existing allocation without copying. This turns O(n^2) into amortized O(n).

## Task

Implement the append optimization for ProcBin terms. When `bs_put_binary` or `iodata_append` detects a ProcBin target with ref count 1 and sufficient capacity, extend in-place instead of allocating a new binary.

## Requirements

### R1: SharedBinary with capacity tracking

Extend SharedBinary to track allocated capacity separately from used length, enabling in-place growth.

**Acceptance:**
- SharedBinary tracks `capacity: usize` alongside `len: usize`
- `SharedBinary::with_capacity(initial_bytes, extra_capacity) -> SharedBinary` pre-allocates space
- `SharedBinary::try_append(&mut self, bytes: &[u8]) -> bool` appends in-place if capacity allows, returns false if not
- Capacity is only accessible when `Arc::strong_count == 1` (exclusive ownership)
- Test: create SharedBinary with capacity 1024, append 100 bytes — succeeds without reallocation
- Test: append beyond capacity returns false

**Files:**
- modify: crates/beamr/src/term/shared_binary.rs

**Checklist:**
- C150 — BIF registry maps (Module, Function, Arity) to a Rust function pointer

### R2: BinaryBuilder detects appendable ProcBin

When `bs_init` initializes a binary builder and the accumulator is a ProcBin with ref count 1, the builder reuses the existing SharedBinary instead of allocating a new one.

**Acceptance:**
- `bs_init` checks if the source binary is a ProcBin with `Arc::strong_count == 1`
- If appendable: builder wraps the existing SharedBinary, writes continue from the current position
- If not appendable (ref count > 1, inline binary, or no ProcBin): existing behavior (new allocation)
- Test: build binary in a loop — first iteration allocates, subsequent iterations append in-place

**Files:**
- modify: crates/beamr/src/interpreter/opcodes/binary.rs

**Checklist:**
- C150 — BIF registry maps (Module, Function, Arity) to a Rust function pointer

### R3: Capacity growth strategy

When append exceeds capacity, grow the SharedBinary's allocation with a doubling strategy (matching BEAM's approach).

**Acceptance:**
- When `try_append` fails due to capacity, allocate a new SharedBinary with `max(current_capacity * 2, current_len + new_bytes)`
- Copy existing data to the new allocation
- Update the ProcBin's pointer to the new SharedBinary
- The old allocation is freed (Arc ref count was 1)
- Test: build a binary byte-by-byte to 10,000 bytes — fewer than 20 allocations total (doubling strategy)

**Files:**
- modify: crates/beamr/src/term/shared_binary.rs
- modify: crates/beamr/src/interpreter/opcodes/binary.rs

**Checklist:**
- C150 — BIF registry maps (Module, Function, Arity) to a Rust function pointer

### R4: iodata_append optimization

Apply the same optimization to the `iodata_append` BIF — when appending to a ProcBin with ref count 1, extend in-place.

**Acceptance:**
- `iodata_append(ProcBin_ref1, data)` appends in-place when capacity allows
- `iodata_append(ProcBin_ref2+, data)` creates a new binary (copy-on-write)
- `iodata_append(inline_binary, data)` existing behavior (creates new binary)
- Test: repeated iodata_append to a sole-owner ProcBin is O(n), not O(n^2)

**Files:**
- modify: crates/beamr/src/native/stdlib_stubs/mod.rs

**Checklist:**
- C150 — BIF registry maps (Module, Function, Arity) to a Rust function pointer

## Boundaries

- SHALL NOT implement in-place append for inline binaries — only ProcBin with capacity
- SHALL NOT modify binaries with ref count > 1 — copy-on-write semantics must be preserved
- SHALL NOT break the immutability guarantee for shared binaries — only the sole owner can append
