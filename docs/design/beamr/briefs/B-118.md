# B-118: Remote message send — Pid ! Msg across nodes

> **Cluster:** beamr
> **Depends on:** B-117
> **Checklist:**
> - C150 — BIF registry maps (Module, Function, Arity) to a Rust function pointer

## Purpose

The fundamental distribution operation: sending a message to a process on another node. When the target PID is remote, the message is ETF-encoded, wrapped in a SEND control message, and written to the connection. The remote node decodes the message and delivers it to the target process's mailbox.

## Task

Implement remote message send and the SEND/REG_SEND control message handlers.

## Requirements

### R1: Remote send dispatch

When the send instruction (or erlang:send/2) targets a remote PID, route the message through the distribution layer instead of the local mailbox.

**Acceptance:**
- send to remote PID encodes the message via ETF and writes to the connection
- send to local PID is unchanged (existing behavior)
- If no connection exists to the target node, attempt to connect (or return noconnection)
- Test: send message from node A to node B, verify delivery

**Files:**
- modify: crates/beamr/src/interpreter/opcodes/messaging.rs
- modify: crates/beamr/src/distribution/mod.rs

**Checklist:**
- C150 — BIF registry maps (Module, Function, Arity) to a Rust function pointer

### R2: SEND and REG_SEND control messages

Implement the distribution control message types for message delivery.

**Acceptance:**
- SEND control: {2, Cookie, ToPid} + message payload — delivers to PID on receiving node
- REG_SEND control: {6, FromPid, Cookie, ToName} + message payload — delivers to registered name
- Incoming SEND decodes the message and enqueues in target PID's mailbox
- Incoming REG_SEND resolves the name via process registry
- Test: send via SEND control, verify mailbox delivery

**Files:**
- create: crates/beamr/src/distribution/control.rs

**Checklist:**
- C150 — BIF registry maps (Module, Function, Arity) to a Rust function pointer

### R3: Integration test — cross-node message passing

End-to-end test with two beamr instances communicating.

**Acceptance:**
- Test spawns two beamr schedulers with different node names
- Node A sends a message to a PID on node B
- Node B's process receives the message in its mailbox
- Round-trip: B replies to A, A receives the reply
- Test uses loopback TCP (127.0.0.1)

**Files:**
- create: crates/beamr/tests/distribution_e2e.rs

**Checklist:**
- C150 — BIF registry maps (Module, Function, Arity) to a Rust function pointer

## Boundaries

- SHALL NOT implement port-based distribution — only PID-to-PID messaging
- SHALL NOT implement message ordering guarantees beyond TCP's in-order delivery
- SHALL handle connection failure during send (return noconnection error)
