# B-045: Code server API, on_load handler, and hot code loading integration

> **Cluster:** beamr
> **Depends on:** B-041, B-042, B-043, B-044
> **Checklist:**
> - C110 — Module registry maps module names to loaded Module structs
> - C130 — Process struct holds x-registers, y-registers, stack, mailbox, code position
> - C150 — BIF registry maps (Module, Function, Arity) to a Rust function pointer

## Purpose

This brief ties together the dual-version registry (B-041), process pinning (B-042), dynamic imports (B-043), and closure versioning (B-044) into a cohesive hot code loading API. It provides the public-facing interface for loading new module versions at runtime, implements the `on_load` instruction that BEAM modules use to run initialisation before becoming current, and adds the BIFs that Erlang code uses to query and manage code loading.

## Boundaries

- This brief SHALL implement the full hot code loading lifecycle: load → on_load → commit/rollback → purge
- This brief SHALL expose the API through both the Rust scheduler interface (for embedding) and as BIFs (for in-VM use)
- This brief SHALL NOT implement code paths or code archives (.ez files) — modules are loaded from raw bytes
- This brief SHALL add integration tests that exercise the full hot-load → execute → purge lifecycle with real compiled modules

## Requirements

### R1: Scheduler hot_load_module API

Add a method on Scheduler for loading a new version of an existing module at runtime, triggering the dual-version promotion from B-041.

**Acceptance:**
- `Scheduler::hot_load_module(bytes: &[u8]) -> Result<HotLoadResult, LoadError>` loads the module, promotes current to old, installs new as current
- `HotLoadResult` struct: `{ module_name: Atom, generation: u64, had_old_version: bool, on_load_required: bool }`
- If the module has never been loaded before, it is simply inserted (no old version)
- If the module was already loaded, current becomes old, new becomes current
- If an old version already exists (two-deep), and purge has not happened, return `LoadError::OldCodeStillRunning` — caller must purge first (matching BEAM semantics: you can't load a third version while old code is still referenced)
- The module's BIF imports are resolved against the current BIF registry
- The module's code imports use the `Deferred` variant from B-043 for any unresolved cross-module dependencies
- Test: hot_load_module with a fresh module, then with a replacement, then attempt a third load without purging

**Files:**
- modify: crates/beamr/src/scheduler/mod.rs
- modify: crates/beamr/src/loader/mod.rs

**Checklist:** C110

### R2: on_load opcode handler

Implement the `on_load` instruction. When a module with an `on_load` attribute is loaded, the on_load function must execute successfully before the module becomes the current version.

**Acceptance:**
- `on_load` instruction in the opcode dispatch returns `InstructionOutcome::OnLoadComplete` (new variant)
- When `hot_load_module` detects an `on_load` instruction in the new module's code, it spawns a temporary process to execute the on_load function
- If on_load returns normally (`ExitReason::Normal`), the module load is committed — the new version becomes current
- If on_load fails (any other exit reason), the module load is rolled back — the new version is discarded and the previous current version remains
- `HotLoadResult` indicates whether on_load was required and whether it succeeded
- Test: load a module with on_load that returns `ok`. Verify module is installed. Load a module with on_load that crashes. Verify old version is retained.

**Files:**
- modify: crates/beamr/src/interpreter/opcodes/mod.rs (on_load dispatch)
- modify: crates/beamr/src/scheduler/mod.rs (on_load execution)

**Checklist:** C130

### R3: Purge API

Expose the purge mechanism from B-041 R3 through the scheduler, with the ability to force-kill processes running old code (matching BEAM's `code:purge/1` behaviour).

**Acceptance:**
- `Scheduler::purge_module(name: Atom) -> Result<PurgeResult, PurgeError>` attempts a safe purge (fails if processes still reference old code)
- `Scheduler::force_purge_module(name: Atom) -> Result<PurgeResult, PurgeError>` kills all processes running old code, then purges
- `PurgeResult` struct: `{ module_name: Atom, processes_killed: usize }`
- Force purge iterates the process table, checks each process's pinned_module, and terminates any process pinned to the old version
- After purge, the old version's `Arc<Module>` is removed from the registry. If no other references exist, it is deallocated.
- Test: load v1, spawn a process running v1. Load v2. Safe purge fails (process running old code). Force purge succeeds, process is killed, old version is freed.

**Files:**
- modify: crates/beamr/src/scheduler/mod.rs
- modify: crates/beamr/src/module.rs

**Checklist:** C110

### R4: Code management BIFs

Implement BIFs for in-VM code management, matching the subset of `erlang` and `code` module functions relevant to hot code loading.

**Acceptance:**
- `erlang:load_module/2` — load raw bytes as a new module version. Args: module name atom, binary. Returns `{module, Name}` on success.
- `erlang:purge_module/1` — safe purge of old code. Returns true on success.
- `erlang:delete_module/1` — remove a module from the registry entirely. Returns true.
- `erlang:check_old_code/1` — returns true if old code exists for the module, false otherwise.
- `erlang:check_process_code/2` — returns true if the process is running old code for the given module.
- All BIFs registered in the BIF registry via existing registration pattern
- Test: from BEAM code, call load_module to hot-load a new version, verify check_old_code returns true, purge the old code, verify check_old_code returns false.

**Files:**
- create: crates/beamr/src/native/code_management_bifs.rs
- modify: crates/beamr/src/native/mod.rs (registration)

**Checklist:** C150

### R5: Integration test — full hot code loading lifecycle

An end-to-end test that exercises the complete hot code loading flow: load module v1, start processes, hot-load v2, verify running processes continue on v1, new processes use v2, purge old code after processes exit.

**Acceptance:**
- Integration test in crates/beamr/tests/hot_code_loading.rs
- Test scenario:
  1. Load module `counter` v1 with function `version/0` returning 1
  2. Spawn process P1 calling counter:version/0 — returns 1
  3. Hot-load counter v2 with version/0 returning 2
  4. P1 (if still pinned) continues seeing v1 semantics
  5. Spawn process P2 calling counter:version/0 — returns 2
  6. Verify check_old_code returns true
  7. After P1 exits, purge succeeds
  8. check_old_code returns false
- Test uses real compiled .beam files (not synthetic), placed in test fixtures
- Test covers: version transition at external call boundary, closure version binding, stack frame return across versions

**Files:**
- create: crates/beamr/tests/hot_code_loading.rs
- create: crates/beamr/tests/fixtures/hot_code/ (test .beam files, v1 and v2)

**Checklist:** C110, C130, C150
