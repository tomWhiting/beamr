# B-045: Code server API, on_load handler, and hot code loading integration

> **Cluster:** beamr
> **Depends on:** B-041, B-042, B-043, B-044
> **Checklist:**
> - C145 — Module registry stores loaded modules indexed by atom name

> **Stories:**

> - S9 (Meridian Runtime, Embedding beamr for Workflow Execution) — As the Meridian runtime, I want to register Rust functions as callable BIFs and NIFs before loading modules so that Gleam code can call Yggdrasil, git, and Meridian operations.

## Purpose

This brief ties together the dual-version registry (B-041), process pinning (B-042), dynamic imports (B-043), and closure versioning (B-044) into a cohesive hot code loading API. It provides the public-facing interface for loading new module versions at runtime, implements the on_load instruction, and adds the BIFs that code uses to query and manage code loading.

## Task

Implement the full hot code loading lifecycle: load → on_load → commit/rollback → purge. Expose the API through both the Rust scheduler interface (for embedding) and as BIFs (for in-VM use). Add integration tests exercising the full lifecycle with real compiled modules.

## Requirements

### R1: Scheduler hot_load_module API

WHEN Scheduler::hot_load_module(bytes) is called, THE SYSTEM SHALL load the module, promote current to old, and install new as current. IF an old version already exists and has not been purged, THE SYSTEM SHALL return LoadError::OldCodeStillRunning.

**Acceptance:**
- Scheduler::hot_load_module(bytes: &[u8]) returns Result<HotLoadResult, LoadError>
- HotLoadResult struct: { module_name: Atom, generation: u64, had_old_version: bool, on_load_required: bool }
- If the module has never been loaded before, it is simply inserted (no old version)
- If the module was already loaded, current becomes old, new becomes current
- If an old version already exists (two-deep) and purge has not happened, return LoadError::OldCodeStillRunning
- The module's BIF imports are resolved against the current BIF registry
- The module's code imports use the Deferred variant from B-043 for any unresolved dependencies
- Test: hot_load_module with a fresh module, then with a replacement, then attempt a third load without purging

### R2: on_load opcode handler

WHEN a module with an on_load attribute is hot-loaded, THE SYSTEM SHALL spawn a temporary process to execute the on_load function. IF on_load returns normally, THE SYSTEM SHALL commit the module load. IF on_load fails, THE SYSTEM SHALL roll back and retain the previous version.

**Acceptance:**
- on_load instruction in the opcode dispatch returns InstructionOutcome::OnLoadComplete (new variant)
- hot_load_module detects an on_load instruction in the new module's code and spawns a temporary process to execute it
- If on_load returns ExitReason::Normal, the module load is committed — new version becomes current
- If on_load fails (any other exit reason), the module load is rolled back — new version is discarded, previous version remains
- HotLoadResult indicates whether on_load was required and whether it succeeded
- Test: load a module with on_load that returns ok. Verify module is installed. Load a module with on_load that crashes. Verify old version is retained.

### R3: Purge API

WHEN Scheduler::purge_module(name) is called, THE SYSTEM SHALL attempt a safe purge. WHEN Scheduler::force_purge_module(name) is called, THE SYSTEM SHALL kill all processes running old code, then purge.

**Acceptance:**
- Scheduler::purge_module(name) attempts safe purge (fails if processes still reference old code)
- Scheduler::force_purge_module(name) kills all processes running old code, then purges
- PurgeResult struct: { module_name: Atom, processes_killed: usize }
- Force purge iterates the process table, checks each process's current_module and stack frame pinned_modules, terminates any process referencing the old version
- After purge, the old version's Arc<Module> is removed from the registry
- Test: load v1, spawn a process running v1. Load v2. Safe purge fails (process running old code). Force purge succeeds, process is killed, old version is freed.

### R4: Code management BIFs

THE SYSTEM SHALL implement BIFs for in-VM code management matching the subset of erlang module functions relevant to hot code loading.

**Acceptance:**
- erlang:load_module/2 — load raw bytes as a new module version. Returns {module, Name} on success.
- erlang:purge_module/1 — safe purge of old code. Returns true on success.
- erlang:delete_module/1 — remove a module from the registry entirely. Returns true.
- erlang:check_old_code/1 — returns true if old code exists for the module, false otherwise.
- erlang:check_process_code/2 — returns true if the process is running old code for the given module.
- All BIFs registered in the BIF registry via existing registration pattern
- Test: from BEAM code, call load_module to hot-load a new version, verify check_old_code returns true, purge the old code, verify check_old_code returns false.
