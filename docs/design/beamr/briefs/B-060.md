# B-060: erlang:throw/1 ŌĆö throw exception type

> **Cluster:** beamr
> **Depends on:** B-015 (exception handling)
> **Checklist:**
> - C150 ŌĆö BIF registry maps (Module, Function, Arity) to a Rust function pointer
> - C86 ŌĆö try/try_end/try_case/try_case_end exception handling instructions
> - C87 ŌĆö raise instruction re-raises exceptions with class, reason, and stacktrace

## Purpose

Erlang has three exception classes: `error` (from `erlang:error/1`), `throw` (from `erlang:throw/1`), and `exit` (from `erlang:exit/1`). beamr implements `error` and `exit` but not `throw`. The `throw` class is used by Gleam's `rescue` and by OTP's `gen_server` for non-local returns. Without it, any code calling `erlang:throw/1` gets an unresolved import error, and `try/catch` blocks that match on `throw:Reason` never fire.

The key challenge: the current BIF error path always wraps `Err(reason)` as `Exception { class: atom(error), ... }`. For `throw`, the exception class must be `atom(throw)`.

## Task

Register `erlang:throw/1` as a BIF and add a mechanism for BIFs to specify the exception class. The simplest approach: add an `exception_class` field to `ProcessContext` that BIFs can set before returning `Err`. The interpreter reads this field when constructing the `Exception`, then resets it to `error`.

## Boundaries

- SHALL NOT change the `NativeFn` return type ŌĆö BIFs still return `Result<Term, Term>`
- SHALL NOT implement `erlang:error/2` (error with args) ŌĆö that's a separate brief
- SHALL NOT modify the exception propagation path (try/catch/raise) ŌĆö only the exception construction at the BIFŌåÆinterpreter boundary
- SHALL verify that existing `try/catch` dispatch correctly matches `throw` class exceptions

## Requirements

### R1: ProcessContext exception class field

Add an `exception_class` field to `ProcessContext` that defaults to `ExceptionClass::Error`. BIFs can set it to `ExceptionClass::Throw` before returning `Err`.

**Acceptance:**
- `ProcessContext` gains `exception_class: ExceptionClass` field (default: `Error`)
- `ExceptionClass` enum with variants: `Error`, `Throw`, `Exit`
- `ProcessContext::set_exception_class(class)` setter
- `ProcessContext::take_exception_class() -> ExceptionClass` reads and resets to `Error`
- Existing BIFs (error/1, exit/2) do not set this field and continue to work unchanged

**Files:**
- modify: crates/beamr/src/native/context.rs

**Checklist:** C86

### R2: Interpreter reads exception class from ProcessContext

Modify the BIF error handling path in the interpreter to read `ProcessContext::take_exception_class()` instead of hardcoding `atom(error)` as the exception class.

**Acceptance:**
- When a BIF returns `Err(reason)`, the interpreter constructs `Exception { class: atom(take_exception_class()), reason, stacktrace }`
- If no BIF set the exception class, `take_exception_class()` returns `Error` (backward compatible)
- The exception class atom is correctly mapped: `Error` ŌåÆ `atom(error)`, `Throw` ŌåÆ `atom(throw)`, `Exit` ŌåÆ `atom(exit)`
- Test: a BIF returning `Err` without setting exception class produces `Exception { class: atom(error) }` (existing behavior preserved)

**Files:**
- modify: crates/beamr/src/interpreter/opcodes/core.rs (BIF call error handling path)

**Checklist:** C86

### R3: erlang:throw/1 BIF registration

Register `erlang:throw/1` as a BIF. The implementation sets `exception_class` to `Throw` on ProcessContext, then returns `Err(reason)`.

**Acceptance:**
- `throw/1` registered in the BIF registry under the `erlang` module
- Implementation: `context.set_exception_class(ExceptionClass::Throw); Err(*reason)`
- Import resolution resolves `erlang:throw/1` without unresolved-import errors
- Test: calling `throw(hello)` produces `Exception { class: atom(throw), reason: atom(hello) }`

**Files:**
- modify: crates/beamr/src/native/bifs.rs (bifs.rs at 625 lines ŌĆö add ~10 lines for throw/1 registration and handler)

**Checklist:** C150

### R4: try/catch correctly dispatches throw class

Verify that the existing `try_case` handler correctly matches `throw` class exceptions. In BEAM semantics, `try_case` writes `(class, reason, stacktrace)` to X registers. The `class` must be `atom(throw)` for thrown exceptions, not `atom(error)`.

**Acceptance:**
- A `try/catch` block that catches `throw:Reason` matches when `erlang:throw(Reason)` is called inside the try
- A `try/catch` block that only catches `error:Reason` does NOT match a `throw` exception
- Test: `try throw(42) catch throw:V -> V end` returns `42`
- Test: `try throw(42) catch error:V -> V end` does not catch ŌĆö exception propagates

**Files:**
- modify: crates/beamr/src/interpreter/opcodes/messaging.rs (only if changes needed ŌĆö may just need tests)

**Checklist:** C86, C87

### R5: erlang:exit/1 uses ExceptionClass::Exit

While we're adding the exception class mechanism, wire `erlang:exit/1` (self-exit, different from `exit/2` which sends to another process) through the same path for consistency.

**Acceptance:**
- `erlang:exit/1` sets `ExceptionClass::Exit` on ProcessContext before returning `Err`
- `try exit(reason) catch exit:R -> R end` returns `reason`
- Existing exit/2 behavior (sending exit signals to other processes) is unchanged

**Files:**
- modify: crates/beamr/src/native/bifs.rs

**Checklist:** C150
