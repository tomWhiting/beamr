# B-072: Shared binary heap and ProcBin — off-heap reference-counted binaries

> **Cluster:** beamr
> **Depends on:** B-023a (binary term representation)
> **Checklist:**
> - C150 — BIF registry maps (Module, Function, Arity) to a Rust function pointer

## Purpose

Binaries are currently stored inline on the process heap — every byte is packed into heap words and copied during GC. A 100KB binary consumes 12,500 heap words and is fully copied on every minor collection. For workflows that process files, HTTP bodies, or serialized data, this makes GC prohibitively expensive.

Reference-counted binaries (refc binaries) solve this by storing large binary data off-heap in a shared `Arc<Vec<u8>>`. The process heap holds only a small pointer term (`ProcBin`, 3 words) that references the shared data. Multiple processes can share the same binary data without copying. GC moves only the ProcBin pointer, not the data.

This brief implements the SharedBinary storage, the ProcBin on-heap term, and the accessor that transparently handles both inline and refc binaries.

## Task

Add `SharedBinary` (off-heap storage), `ProcBin` (on-heap reference term with BoxedTag 0x19), and a unified `BinaryRef` accessor that abstracts over inline Binary and ProcBin terms.

## Boundaries

- SHALL NOT change the inline Binary format (BoxedTag 0x16) — small binaries continue to use inline storage
- SHALL NOT implement the size threshold / automatic promotion — that's B-073
- SHALL NOT implement sub-binaries — that's B-074
- SHALL NOT implement GC integration for refc binaries — that's B-075
- SHALL NOT implement append optimization — that's B-076

## Requirements

### R1: SharedBinary off-heap storage

Define `SharedBinary` as an `Arc`-wrapped byte buffer stored outside the process heap.

**Acceptance:**
- `SharedBinary` wraps `Arc<Vec<u8>>` (or `Arc<[u8]>` for immutable slice)
- `SharedBinary::new(bytes: Vec<u8>) -> SharedBinary` creates a new shared binary
- `SharedBinary::as_bytes(&self) -> &[u8]` provides read access
- `SharedBinary::len(&self) -> usize` returns byte length
- `SharedBinary::ref_count(&self) -> usize` returns `Arc::strong_count`
- Clone is cheap (Arc clone, not data copy)
- Send + Sync for cross-process sharing
- Test: create SharedBinary, clone it, verify both see same data, ref_count is 2

**Files:**
- create: crates/beamr/src/term/shared_binary.rs
- modify: crates/beamr/src/term/mod.rs

**Checklist:** C150

### R2: ProcBin on-heap term

Define the ProcBin heap layout — a small on-heap term that holds a pointer to a SharedBinary. Uses BoxedTag 0x19.

**Acceptance:**
- ProcBin heap layout: `[header(tag=0x19, size=2)] [flags: u64] [arc_ptr: u64]` — 3 words total. On 64-bit, a raw pointer from `Arc::into_raw` fits in a single u64.
- `write_proc_bin(heap: &mut [u64], shared: &SharedBinary) -> Option<Term>` writes the ProcBin to heap
- The `Arc<Vec<u8>>` is stored as a raw pointer in one heap word (via `Arc::into_raw` / `Arc::from_raw`)
- ProcBin is always 3 words (header + 2 payload) regardless of binary size
- Flags field reserved for future use (sub-binary flag, writable flag)
- Test: write a ProcBin for a 100KB SharedBinary — only 3 heap words used

**Files:**
- modify: crates/beamr/src/term/shared_binary.rs
- modify: crates/beamr/src/term/boxed/mod.rs (add BoxedTag::ProcBin = 0x19)

**Checklist:** C150

### R3: ProcBin accessor

Add a `ProcBin` accessor struct (same pattern as `Binary`, `Tuple`, etc.) that reads the SharedBinary from heap words and provides byte access.

**Acceptance:**
- `ProcBin::new(term: Term) -> Option<Self>` validates BoxedTag::ProcBin
- `ProcBin::as_bytes(self) -> &[u8]` returns the SharedBinary's data
- `ProcBin::len(self) -> usize` returns byte length
- `ProcBin::shared_binary(self) -> SharedBinary` returns a clone of the Arc (incrementing ref count)
- Reading a ProcBin does NOT copy the binary data — only follows the pointer
- Test: write ProcBin, read back via accessor, verify data matches

**Files:**
- modify: crates/beamr/src/term/boxed/accessors.rs

**Checklist:** C150

### R4: Unified BinaryRef accessor

Create a `BinaryRef` enum that abstracts over inline Binary and ProcBin, providing uniform byte access regardless of storage strategy.

**Acceptance:**
- `BinaryRef::new(term: Term) -> Option<Self>` returns `BinaryRef::Inline(Binary)` or `BinaryRef::Refc(ProcBin)` based on boxed tag
- `BinaryRef::as_bytes(&self) -> &[u8]` works for both variants
- `BinaryRef::len(&self) -> usize` works for both variants
- All existing code that calls `Binary::new(term)` can migrate to `BinaryRef::new(term)` for transparent handling
- Test: BinaryRef::new on an inline binary returns Inline variant
- Test: BinaryRef::new on a ProcBin returns Refc variant
- Test: both variants return correct bytes

**Files:**
- create: crates/beamr/src/term/binary_ref.rs (or add to shared_binary.rs)
- modify: crates/beamr/src/term/mod.rs

**Checklist:** C150

### R5: object_size and GC awareness stub

Add `BoxedTag::ProcBin` to `rewrite_copied_object()` in gc/mod.rs so GC can copy ProcBin terms. Note: `object_size()` already works generically via `1 + BoxedHeader::size(header)` — no per-tag case needed. The real requirement is handling ProcBin in the exhaustive `rewrite_copied_object` match (adding ProcBin to BoxedTag without updating this match is a compile error). Between B-072 and B-075 landing, copied ProcBins increment the Arc but unreachable ones are never decremented — SharedBinary data leaks until B-075 lands. This is acceptable for phased rollout.

**Acceptance:**
- `rewrite_copied_object()` handles ProcBin — increments Arc ref count on copy (unsafe: Arc::from_raw + clone + into_raw)
- `rewrite_copied_object()` handles ProcBin as a leaf (no internal Term pointers) — but must increment the Arc ref count when copying
- A process with a ProcBin term survives a minor GC without crashing
- Test: allocate ProcBin, trigger GC, verify ProcBin is still valid after collection

**Files:**
- modify: crates/beamr/src/gc/mod.rs

**Checklist:** C150
