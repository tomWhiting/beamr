# B-120: Remote spawn — SPAWN_REQUEST/REPLY across nodes

> **Cluster:** beamr
> **Depends on:** B-119
> **Checklist:**
> - C150 — BIF registry maps (Module, Function, Arity) to a Rust function pointer

## Purpose

SPAWN_REQUEST allows a process on node A to spawn a process on node B. The request carries the MFA and options; the reply carries the new PID. This enables distributed supervision trees where supervisors spawn children on remote nodes.

## Task

Implement SPAWN_REQUEST and SPAWN_REPLY control messages.

## Requirements

### R1: SPAWN_REQUEST handler

Handle incoming SPAWN_REQUEST: spawn the process locally and reply with the new PID.

**Acceptance:**
- Incoming SPAWN_REQUEST decoded: {ReqId, From, GroupLeader, {M,F,A}, OptList}
- Spawns the process on the local scheduler
- Sends SPAWN_REPLY with the new PID back to the requesting node
- Spawn options (link, monitor) applied atomically
- Test: remote spawn request creates a local process

**Files:**
- modify: crates/beamr/src/distribution/control.rs

**Checklist:**
- C150 — BIF registry maps (Module, Function, Arity) to a Rust function pointer

### R2: Remote spawn BIF

Register erlang:spawn/4 variant that specifies a target node.

**Acceptance:**
- spawn(Node, Module, Function, Args) sends SPAWN_REQUEST to target node
- Blocks (or uses async callback) until SPAWN_REPLY received
- Returns the remote PID
- spawn_link and spawn_monitor variants include link/monitor options in the request
- Test: spawn process on remote node, verify PID is node-qualified

**Files:**
- modify: crates/beamr/src/native/process_bifs/mod.rs

**Checklist:**
- C150 — BIF registry maps (Module, Function, Arity) to a Rust function pointer

## Boundaries

- SHALL NOT implement spawn_opt across nodes with all options — only link and monitor
- SHALL NOT implement node selection strategies — caller specifies the exact target node
