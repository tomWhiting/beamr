# B-023c: Monitor management BIFs — monitor, demonitor, exit

> **Cluster:** beamr
> **Depends on:** B-023a
> **Checklist:**
> - C134 — When a linked process exits, an exit signal with the exit reason is sent to all linked processes
> - C135 — A process receiving an exit signal with a non-normal reason exits with that reason (unless trap_exit)
> - C136 — A process receiving an exit signal with reason 'normal' is not terminated
> - C138 — An exit signal with reason 'kill' terminates unconditionally
> - C139 — A process killed by 'kill' propagates exit reason 'killed' (not 'kill') to its links
> - C140 — monitor(Pid) returns a reference and registers a monitor; death sends {'DOWN', Ref, process, Pid, Reason}
> - C141 — demonitor(Ref) removes the monitor; no DOWN message sent after demonitor
> - C143 — Monitoring a process that has already exited immediately sends a DOWN message
> **Stories:**
> - S2 (Workflow Author) — As a workflow author, I want each workflow step to run in its own isolated process so that a crash in one step does not corrupt the state of another.
> - S6 (Workflow Author) — As a workflow author, I want to define supervision trees over my workflow steps so that failed steps are automatically restarted according to a declared strategy.

## Purpose

The `MonitorSet` data structure provides unidirectional monitoring with unique references, DOWN message construction, and immediate DOWN for already-dead targets. The `LinkSet` provides exit signal delivery. But running BEAM code cannot call `monitor/2`, `demonitor/1`, or `exit/2`. This brief adds those three BIFs, completing the process-lifecycle BIF cluster. Together with B-023b, these enable Gleam code to build full supervision graphs.

## Task

Add three BIFs to `crates/beamr/src/native/process_bifs.rs` (created in B-023a). Extend `ProcessContext` with monitor set access so that `monitor/2` and `demonitor/1` can manage monitors. `exit/2` delivers exit signals to target processes. Register all three in `register_gate2_bifs()`.

## Boundaries

- This brief SHALL NOT implement erlang:exit/1 (self-exit — different semantics from exit/2)
- This brief SHALL NOT implement monitor by name (only by PID)
- This brief SHALL NOT implement demonitor with options ([flush, info])

## Requirements

### R1: erlang:monitor/2

`erlang:monitor/2` SHALL accept `(process, Pid)` and establish a unidirectional monitor from the calling process to the target. It SHALL return a unique reference term. If the target PID does not exist or has already exited, it SHALL immediately enqueue a `{DOWN, Ref, process, Pid, noproc}` message to the caller's mailbox.

**Acceptance:**
- monitor(process, Pid) returns a unique reference term
- When the monitored process exits, a DOWN message is delivered to the watcher
- Monitoring a dead process immediately sends DOWN with reason noproc
- Multiple monitors on the same target get independent references

**Files:**
- modify: crates/beamr/src/native/process_bifs.rs

**Checklist:**
- C140
- C143

### R2: erlang:demonitor/1

`erlang:demonitor/1` SHALL accept a reference and remove the corresponding monitor. After demonitor, no DOWN message SHALL be sent for that monitor even if the target process exits. If the reference does not correspond to an active monitor, the call SHALL be a no-op returning `true`.

**Acceptance:**
- demonitor(Ref) removes the monitor
- After demonitor, target exit does not send DOWN
- demonitor of invalid reference is a no-op returning true

**Files:**
- modify: crates/beamr/src/native/process_bifs.rs

**Checklist:**
- C141

### R3: erlang:exit/2

`erlang:exit/2` SHALL accept `(Pid, Reason)` and send an exit signal to the target process. If the target traps exits, it SHALL receive `{EXIT, CallerPid, Reason}` as a message in its mailbox. If the target does not trap exits and the reason is not `normal`, it SHALL be terminated. `exit(Pid, kill)` SHALL terminate the target unconditionally, regardless of trap_exit. A process terminated by `kill` SHALL propagate exit reason `killed` (not `kill`) to its linked processes.

**Acceptance:**
- exit(Pid, normal) does not kill target (even without trap_exit)
- exit(Pid, reason) kills non-trapping target
- exit(Pid, reason) delivers {EXIT, Caller, reason} to trapping target
- exit(Pid, kill) kills regardless of trap_exit
- Process killed by kill propagates 'killed' to its links
