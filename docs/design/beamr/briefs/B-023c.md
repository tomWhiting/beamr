# B-023c: Monitor management BIFs — monitor, demonitor, exit

> **Cluster:** beamr
> **Depends on:** B-023a
> **Checklist:**
> - C134 — When a linked process exits, an exit signal with the exit reason is sent to all linked processes
> - C135 — A process receiving an exit signal with a non-normal reason exits with that reason (unless it traps exits)
> - C136 — A process receiving an exit signal with reason 'normal' is not terminated
> - C138 — An exit signal with reason 'kill' terminates the target process unconditionally, even if it traps exits
> - C139 — A process killed by 'kill' propagates exit reason 'killed' (not 'kill') to its links
> - C140 — monitor(Pid) returns a reference and registers a monitor; the monitored process's death sends {'DOWN', Ref, process, Pid, Reason} to the monitor owner
> - C141 — demonitor(Ref) removes the monitor; no DOWN message is sent after demonitor
> - C143 — Monitoring a process that has already exited immediately sends a DOWN message
> **Stories:**
> - S2 (Workflow Author, Writing Gleam Workflows) — As a workflow author, I want each workflow step to run in its own isolated process so that a crash in one step does not corrupt the state of another.
> - S6 (Workflow Author, Writing Gleam Workflows) — As a workflow author, I want to define supervision trees over my workflow steps so that failed steps are automatically restarted according to a declared strategy.

## Purpose

MonitorSet and LinkSet data structures exist but are not callable from bytecode. This brief adds monitor/2, demonitor/1, and exit/2 as BIFs.

## Task

Add three BIFs to process_bifs.rs. Extend ProcessContext with monitor set access. Register in register_gate2_bifs().

## Requirements

### R1: erlang:monitor/2

erlang:monitor/2 SHALL accept (process, Pid), establish a unidirectional monitor, return a unique reference. Dead target immediately enqueues DOWN.

**Acceptance:**
- monitor(process, Pid) returns unique reference
- Monitored process exit delivers DOWN
- Monitoring dead process sends immediate DOWN with noproc
- Multiple monitors get independent references

**Files:**
- modify: crates/beamr/src/native/process_bifs.rs

**Checklist:**
- C140 — monitor(Pid) returns a reference and registers a monitor; the monitored process's death sends {'DOWN', Ref, process, Pid, Reason} to the monitor owner
- C143 — Monitoring a process that has already exited immediately sends a DOWN message

**Stories:**
- S6 (Workflow Author, Writing Gleam Workflows) — As a workflow author, I want to define supervision trees over my workflow steps so that failed steps are automatically restarted according to a declared strategy.

### R2: erlang:demonitor/1

erlang:demonitor/1 SHALL remove the monitor by reference. Invalid reference is a no-op.

**Acceptance:**
- demonitor(Ref) removes the monitor
- After demonitor, target exit does not send DOWN
- Invalid reference is no-op returning true

**Files:**
- modify: crates/beamr/src/native/process_bifs.rs

**Checklist:**
- C141 — demonitor(Ref) removes the monitor; no DOWN message is sent after demonitor

**Stories:**
- S6 (Workflow Author, Writing Gleam Workflows) — As a workflow author, I want to define supervision trees over my workflow steps so that failed steps are automatically restarted according to a declared strategy.

### R3: erlang:exit/2

erlang:exit/2 SHALL send an exit signal to target. Trapping target receives {EXIT, Caller, Reason}. Non-trapping with non-normal reason terminates. kill terminates unconditionally and propagates as killed.

**Acceptance:**
- exit(Pid, normal) does not kill target
- exit(Pid, reason) kills non-trapping target
- exit(Pid, reason) delivers {EXIT, Caller, reason} to trapping target
- exit(Pid, kill) kills regardless of trap_exit
- kill propagates as killed to links

**Files:**
- modify: crates/beamr/src/native/process_bifs.rs

- C138 — An exit signal with reason 'kill' terminates the target process unconditionally, even if it traps exits
- C139 — A process killed by 'kill' propagates exit reason 'killed' (not 'kill') to its links

## Boundaries

- This brief SHALL NOT implement erlang:exit/1 (self-exit — different semantics from exit/2)
- This brief SHALL NOT implement monitor by name (only by PID)
- This brief SHALL NOT implement demonitor with options ([flush, info])
