## Purpose

This brief closes the loader pipeline: take a ParsedModule from B-003, resolve its imports against loaded modules and the BIF registry, validate instruction operands, store the result in the module registry, and report any unresolved imports. The unresolved-import report is the mechanism that keeps the project scoped — it tells you exactly which BIFs a workflow needs, and no more. Without this brief, parsed modules are data with dangling references; with it, they are runnable.

## Task

Implement import resolution in the loader: walk the ParsedModule's import entries and resolve each against (1) the module registry (for cross-module calls) and (2) any implementation of a BifRegistry trait (for native calls). B-004 defines the BifRegistry trait; B-008 provides the concrete implementation. Unresolved imports are collected into a report, not treated as fatal errors — a module can load with unresolved imports but will fail at runtime if they are called. Implement instruction validation in validate.rs: register indices in range, label targets exist, arities match. Define the Module struct (name, exports, code, literals, resolved imports) and store it in the module registry. Provide a top-level load_module() function that chains B-003's parser with resolution, validation, and registration.

## Requirements

### R1: Import resolution against module registry and BIF registry

WHEN a ParsedModule's imports are resolved THE SYSTEM SHALL check each ImportEntry against the module registry (for loaded modules' exports) and a BifRegistry trait (for native functions). B-004 SHALL define a minimal BifRegistry trait with a lookup(module, function, arity) method. B-008 later provides the concrete implementation. This brief SHALL be testable with an empty BifRegistry implementation. Resolved imports SHALL point to a callable target (code offset or native function reference). Unresolved imports SHALL be collected but SHALL NOT prevent module loading — they become runtime undef errors only if called.

**Acceptance:**
- An import for a loaded module's exported function resolves to a code offset
- An import for a registered BIF resolves to a native function reference
- An import for a function that exists in neither registry appears in the unresolved list
- A module with unresolved imports loads successfully (it is not rejected)
- A BifRegistry trait is defined with a lookup(module, function, arity) method
- Tests pass with an empty (no-op) BifRegistry implementation

### R2: Unresolved-import report

WHEN import resolution completes THE SYSTEM SHALL produce an UnresolvedImportReport listing every import that could not be resolved, grouped by module name. Each entry SHALL include the module, function, and arity. The report SHALL be returned alongside the loaded module so callers can inspect it. It SHALL NOT silently discard unresolved imports.

**Acceptance:**
- UnresolvedImportReport is a public struct with entries grouped by module atom
- Each entry contains module: Atom, function: Atom, arity: u8
- Loading a module that calls erlang:unknown/0 produces a report with one entry for erlang:unknown/0
- Loading a module with all imports resolved produces an empty report
- The report implements Display for human-readable output

### R3: Instruction operand validation

AFTER decoding and import resolution THE SYSTEM SHALL validate instruction operands: X register indices SHALL be < 256, Y register indices SHALL be < the allocate frame size, label targets SHALL refer to existing labels, and call arities SHALL match import/export arities. IF validation fails THEN THE SYSTEM SHALL return LoadError::ValidationError naming the instruction index and the specific operand that failed. It SHALL NOT silently accept out-of-range operands.

**Acceptance:**
- A module with valid operands passes validation without errors
- An instruction referencing X register 999 produces LoadError::ValidationError mentioning the register index
- An instruction jumping to a non-existent label produces LoadError::ValidationError mentioning the label
- Validation errors include the instruction index for debuggability

### R4: Module struct definition

beamr::module SHALL define a Module struct containing: name (Atom), exports (map from (Atom, u8) to label), code (Vec<Instruction>), literals (Vec<Literal>), resolved imports (Vec<ResolvedImport>), lambdas (Vec<LambdaEntry>), string table (Vec<u8>), and line info (Vec<LineInfo>). Module SHALL be stored behind an Arc for shared ownership between the registry and running processes. It SHALL NOT contain mutable runtime state — Module is immutable after loading.

**Acceptance:**
- Module is a public struct in beamr::module
- Module.exports maps (function_atom, arity) to a code label
- Module.name is an Atom
- Module is wrapped in Arc when stored in the registry
- Module has no Cell, RefCell, Mutex, or other interior mutability

### R5: Store loaded module in the module registry

WHEN a module passes validation THE SYSTEM SHALL store it in the module registry indexed by its atom name. WHEN a module with the same name is already loaded THE SYSTEM SHALL replace the existing entry with the new one. The registry SHALL use a concurrent map for thread-safe access. It SHALL NOT support storing more than one version simultaneously (hot code loading is out of scope per DESIGN.md non-goals).

**Acceptance:**
- After load_module succeeds, registry.lookup(module_name) returns Some(Arc<Module>)
- Loading a second module with the same name replaces the first
- registry.lookup(unloaded_name) returns None
- Registry is safe to access from multiple threads

### R6: Function lookup by MFA

WHEN the interpreter needs to call a function THE SYSTEM SHALL look it up by MFA (module atom, function atom, arity) in the module registry. The lookup SHALL first find the module by name, then find the export by (function, arity). IF the module is not loaded or the function is not exported THEN THE SYSTEM SHALL return an ExecError::Undef with the full MFA. It SHALL NOT return a generic 'not found' — the error must name the module, function, and arity.

**Acceptance:**
- registry.lookup_mfa(module, function, arity) returns a code pointer for a loaded, exported function
- Calling lookup_mfa for an unloaded module returns Err(ExecError::Undef { .. })
- Calling lookup_mfa for a loaded module with a non-exported function returns Err(ExecError::Undef { .. })
- Calling lookup_mfa with wrong arity returns Err(ExecError::Undef { .. })

### R7: Top-level load_module API

beamr::loader SHALL expose a public load_module(bytes: &[u8], atom_table: &AtomTable, module_registry: &ModuleRegistry, bif_registry: &impl BifRegistry) -> Result<(Arc<Module>, UnresolvedImportReport), LoadError> function. This SHALL chain: parse (B-003) → resolve imports → validate → register. The bif_registry parameter SHALL accept any implementation of the BifRegistry trait. WHEN loading a .beam file produced by gleam build + erlc with only standard erlang: BIFs THE SYSTEM SHALL succeed, returning the module and a report of unresolved erlang: imports.

**Acceptance:**
- load_module is a public function in beamr::loader
- load_module(valid_beam_bytes, ..) returns Ok((Arc<Module>, report))
- The returned Module is also stored in the module_registry
- load_module(garbage_bytes, ..) returns Err(LoadError::InvalidFormat)
- Loading a Gleam module succeeds; unresolved erlang: BIFs appear in the report
- load_module is defined in loader/load.rs, not in loader/mod.rs

## Boundaries

- SHALL NOT implement hot code loading or two-version module storage (explicitly out of scope per DESIGN.md non-goals)
- SHALL NOT implement any BIF function bodies (later briefs scope — this brief only resolves against the BIF registry)
- SHALL NOT implement the BIF registry itself (B-008 scope — this brief receives it as a parameter)
- SHALL NOT modify the parser or chunk decoders from B-003 (if changes are needed, coordinate with B-003)
- SHALL NOT implement Term or heap types — Module stores Literal values from the loader, not runtime Terms
