# B-114: Distribution connection manager — TCP connections and connection table

> **Cluster:** beamr
> **Depends on:** B-113
> **Checklist:**
> - C150 — BIF registry maps (Module, Function, Arity) to a Rust function pointer

## Purpose

Distribution connections are long-lived TCP connections between nodes. The connection manager owns the connection table (mapping node names to active connections), handles outbound connection setup (resolve → connect → handshake), accepts inbound connections, and detects connection failures. Each connection is a bidirectional byte stream used for control messages and term data.

## Task

Implement the ConnectionManager with a connection table, outbound connect path, inbound accept loop, and connection lifecycle management.

## Requirements

### R1: Connection table and ConnectionManager

ConnectionManager owns a map of active distribution connections indexed by node name.

**Acceptance:**
- ConnectionManager struct with connections: DashMap<Atom, Arc<DistConnection>>
- DistConnection wraps a TCP stream (tokio::net::TcpStream or equivalent) with read/write halves
- get_connection(node: Atom) -> Option<Arc<DistConnection>> looks up active connection
- ConnectionManager::start(listen_addr, resolver) spawns the accept loop
- Test: create manager, verify no connections initially

**Files:**
- create: crates/beamr/src/distribution/connection.rs

**Checklist:**
- C150 — BIF registry maps (Module, Function, Arity) to a Rust function pointer

### R2: Outbound connection establishment

Connect to a remote node by resolving its name and establishing a TCP connection.

**Acceptance:**
- connect(node_name: &str) -> Result<Arc<DistConnection>, ConnectError> resolves name via NodeResolver, opens TCP connection
- Connection is added to the connection table on success
- ConnectError variants: ResolveFailure, ConnectionRefused, Timeout
- Configurable connection timeout (default 5 seconds)
- Test: connect to a local listener, verify connection established

**Files:**
- modify: crates/beamr/src/distribution/connection.rs

**Checklist:**
- C150 — BIF registry maps (Module, Function, Arity) to a Rust function pointer

### R3: Inbound connection accept loop

Listen for incoming distribution connections and add them to the connection table after identification.

**Acceptance:**
- Accept loop runs on a dedicated async task
- Incoming connections are held pending until the handshake identifies the remote node (B-115)
- Accepted connections are added to the connection table with the remote node name as key
- Test: spawn two ConnectionManagers, connect one to the other, verify bidirectional connection

**Files:**
- modify: crates/beamr/src/distribution/connection.rs

**Checklist:**
- C150 — BIF registry maps (Module, Function, Arity) to a Rust function pointer

### R4: Connection failure detection and cleanup

Detect TCP connection failures and remove dead connections from the table.

**Acceptance:**
- Connection read/write errors trigger connection removal from the table
- A connection_down callback notifies the distribution layer (for link/monitor cleanup in B-122/B-121)
- Reconnection is NOT automatic (net_kernel handles that in B-123)
- Test: establish connection, drop remote end, verify connection removed from table

**Files:**
- modify: crates/beamr/src/distribution/connection.rs

**Checklist:**
- C150 — BIF registry maps (Module, Function, Arity) to a Rust function pointer

## Boundaries

- SHALL NOT implement the handshake protocol — that's B-115
- SHALL NOT implement message encoding/decoding — that's B-117
- SHALL NOT implement automatic reconnection — that's B-123's net_kernel responsibility
- SHALL use async I/O (tokio) for connection management
