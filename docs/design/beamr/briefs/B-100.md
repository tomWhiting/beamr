# B-100: Completion IO ring abstraction and io_uring backend

> **Cluster:** beamr

## Purpose

BEAM's legacy port driver model routes all I/O through a synchronous interface that blocks scheduler threads. Modern Linux kernels provide io_uring — a completion-based async I/O interface that submits operations via a submission queue (SQE) and receives results via a completion queue (CQE) without blocking. This brief defines beamr's platform-abstracted completion I/O trait (CompletionRing) and provides the primary io_uring implementation for Linux. The trait is the foundation for all file, TCP, UDP, and stdio operations in B-101 through B-111. A kqueue backend for macOS development is a separate follow-on brief (B-100b) — this brief targets Linux production with the abstraction layer designed to accommodate other backends. No external async runtime (tokio) is pulled in; beamr manages its own threads and scheduling. mio may be used as a thin platform abstraction for socket readiness if it simplifies the kqueue backend later, but is not required for the io_uring path.

## Task

Define the CompletionRing trait with methods for submitting I/O operations (read, write, accept, connect, close, etc.) and polling for completions. Implement the trait using Linux io_uring via the io-uring crate. Run the ring on a dedicated OS thread that polls CQEs and dispatches completions to registered callbacks. Define operation types, completion results, and the submission/completion lifecycle. Add the io-uring crate as a platform-conditional dependency (cfg(target_os = linux)).

## Requirements

### R1: CompletionRing trait definition

THE SYSTEM SHALL define a CompletionRing trait that abstracts platform-specific completion I/O. THE SYSTEM SHALL define IoOp enum for operation types and IoCompletion struct for results. THE SYSTEM SHALL support operations: read, write, accept, connect, close, fsync, openat, statx, and nop.

**Acceptance:**
- CompletionRing trait defined in crates/beamr/src/io/ring.rs
- IoOp enum with variants: Read { fd, buf_len, offset }, Write { fd, data: Vec<u8>, offset }, Accept { listener_fd }, Connect { fd, addr: SocketAddr }, Close { fd }, Fsync { fd }, Openat { dir_fd, path, flags, mode }, Statx { dir_fd, path, flags, mask }, Nop
- IoCompletion struct: { op_id: u64, result: io::Result<IoResult> }
- IoResult enum: BytesRead(usize, Vec<u8>), BytesWritten(usize), Accepted(RawFd, SocketAddr), Connected, Closed, Synced, Opened(RawFd), StatResult(StatxData), Completed
- CompletionRing methods: submit(op: IoOp) -> u64 (returns op_id), poll_completions(timeout: Duration) -> Vec<IoCompletion>, pending_count() -> usize, shutdown()
- Op IDs are monotonically increasing u64 via AtomicU64
- Trait is Send + Sync for cross-thread use

**Files:**
- create: crates/beamr/src/io/ring.rs
- create: crates/beamr/src/io/mod.rs

### R2: io_uring backend implementation

WHEN running on Linux, THE SYSTEM SHALL provide an IoUringRing struct implementing CompletionRing backed by the io-uring crate. THE SYSTEM SHALL run the ring on a dedicated OS thread that submits SQEs and polls CQEs.

**Acceptance:**
- IoUringRing struct in crates/beamr/src/io/uring.rs, gated with #[cfg(target_os = "linux")]
- Constructor: IoUringRing::new(ring_depth: u32) -> io::Result<Self> (default depth 256)
- Dedicated ring thread spawned on construction, polls CQEs in a loop
- submit() encodes the IoOp as an io_uring SQE with the op_id as user_data
- CQE results decoded into IoCompletion and sent to a crossbeam channel
- poll_completions() drains the channel with timeout
- Ring thread exits cleanly on shutdown()
- io-uring crate added as conditional dependency: [target.'cfg(target_os = "linux")'.dependencies] io-uring = "0.7"
- Test (Linux-only): submit a Nop, poll completion, verify success
- Test (Linux-only): submit a read on a pipe fd, write to pipe from another thread, verify bytes read

**Files:**
- create: crates/beamr/src/io/uring.rs
- modify: crates/beamr/Cargo.toml

### R3: Thread-pool fallback backend for non-Linux platforms

WHEN running on non-Linux platforms, THE SYSTEM SHALL provide a ThreadPoolRing struct implementing CompletionRing that executes I/O operations on a thread pool using blocking std::fs and std::net calls, simulating completion semantics.

**Acceptance:**
- ThreadPoolRing struct in crates/beamr/src/io/thread_pool.rs, gated with #[cfg(not(target_os = "linux"))]
- Constructor: ThreadPoolRing::new(pool_size: usize) -> Self (default pool_size = 4)
- submit() dispatches the IoOp to a thread pool worker via crossbeam channel
- Worker executes the blocking operation, sends IoCompletion to a result channel
- poll_completions() drains the result channel with timeout
- No external dependencies beyond std and crossbeam
- Test: submit read/write ops on a temp file, verify completions
- Performance note: this is a development fallback, not production — real async comes from io_uring on Linux

**Files:**
- create: crates/beamr/src/io/thread_pool.rs

### R4: Platform-conditional ring construction

THE SYSTEM SHALL provide a factory function that returns the platform-appropriate CompletionRing implementation: IoUringRing on Linux, ThreadPoolRing elsewhere.

**Acceptance:**
- pub fn create_ring(config: RingConfig) -> Box<dyn CompletionRing> factory function in io/mod.rs
- RingConfig struct with ring_depth: u32 (default 256) and fallback_pool_size: usize (default 4)
- On Linux: returns IoUringRing
- On other platforms: returns ThreadPoolRing
- Scheduler can call create_ring() during construction without platform-specific code

**Files:**
- modify: crates/beamr/src/io/mod.rs

## Boundaries

- SHALL NOT pull in tokio or any external async runtime — beamr manages its own threads
- SHALL NOT implement process wakeup on completion — that is B-101's concern
- SHALL NOT implement file descriptor lifecycle management — that is B-102's concern
- SHALL NOT implement a kqueue-specific backend — that is a follow-on brief (B-100b). The ThreadPoolRing provides macOS development support.
- SHALL NOT expose io_uring internals through the CompletionRing trait — the trait must be backend-agnostic

## Verification

- cargo test passes on macOS (ThreadPoolRing used) and Linux (IoUringRing used)
- CompletionRing trait compiles on all platforms without feature gates on the trait itself
- IoUringRing submits and completes a file read on Linux
- ThreadPoolRing submits and completes a file read on macOS
- Ring shutdown is clean with no thread leaks
- No tokio, mio, or async-std in dependency tree
