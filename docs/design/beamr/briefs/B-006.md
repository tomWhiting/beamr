## Purpose

Boxed terms are heap-allocated values too large to fit in a single machine word: tuples, lists, floats, binaries, big integers, closures, maps, and references. Every data structure a Gleam program builds is a boxed term. The interpreter cannot execute beyond trivial arithmetic without them. This brief defines the heap layout (header word + payload) for each boxed type and provides typed constructors and accessors.

## Task

Implement boxed term types in crates/beamr/src/term/boxed.rs and crates/beamr/src/term/binary.rs. Each boxed value lives on a process-local heap as a header word (encoding type and size) followed by payload words. Define header encoding/decoding, and for each boxed type: constructor that writes the header + payload to a &mut [u64] slice (the heap), and accessor that reads back from a *const u64 pointer. Tuples, cons cells, floats, binaries, big integers, closures, maps (flatmap), and references. The heap itself is not this brief's responsibility — boxed terms write to a provided slice and read from a raw pointer.

## Requirements

### R1: Boxed header encoding

Each boxed term on the heap SHALL start with a header word encoding the boxed type tag and the size (in words) of the payload. beamr::term::boxed SHALL define header construction and extraction functions. The header tag SHALL distinguish: tuple, float, big integer, closure, map, reference, and binary. It SHALL NOT collide with the primary tags used by immediates.

**Acceptance:**
- BoxedHeader::new(tag, size) creates a header word
- BoxedHeader::tag(header_word) extracts the boxed type tag
- BoxedHeader::size(header_word) extracts the payload size in words
- Header tags for tuple, float, bigint, closure, map, reference, binary are all distinct

### R2: Tuple boxed type

WHEN a tuple is allocated on the heap THE SYSTEM SHALL write a header word with the tuple tag and arity, followed by arity element words. The Term pointing to the tuple SHALL use the boxed primary tag with a pointer to the header. Tuple::arity() SHALL return the element count. Tuple::get(n) SHALL return the nth element as a Term. It SHALL NOT allow access beyond arity — out-of-bounds returns None.

**Acceptance:**
- A 3-element tuple occupies 4 heap words (1 header + 3 elements)
- Tuple::arity() returns 3 for a 3-element tuple
- Tuple::get(0) returns the first element
- Tuple::get(3) returns None for a 3-element tuple
- A 0-element tuple is valid with arity 0 and 1 heap word

- modify: crates/beamr/src/term/boxed.rs

**Checklist:**

### R3: List cons cell boxed type

WHEN a cons cell is allocated on the heap THE SYSTEM SHALL write two words: head and tail. The Term pointing to the cons cell SHALL use the list primary tag (distinct from boxed tag) with a pointer to the head word. A proper list terminates when the tail is Term::NIL. It SHALL NOT use a header word — cons cells are identified by the list tag on the pointing Term, not by a header. NOTE: cons cells are tag-identified by the pointer (list tag), not self-describing via a header word. The GC brief author must handle cons cells differently from header-tagged boxed terms — they are always exactly 2 words, identified by the list tag on the referencing Term.

**Acceptance:**
- A cons cell occupies exactly 2 heap words (head, tail)
- Cons::head() returns the head term
- Cons::tail() returns the tail term
- A list [1, 2, 3] is 3 cons cells with the last tail being Term::NIL
- The list tag is distinct from the boxed tag in the pointing Term

**Files:**
- modify: crates/beamr/src/term/boxed.rs

**Checklist:**

### R4: Float boxed type

WHEN a float is allocated on the heap THE SYSTEM SHALL write a header word with the float tag followed by an f64 stored as a u64 (transmuted). Float::value() SHALL return the f64. It SHALL NOT use a separate allocation for the f64 — it is inline in the heap words.

**Acceptance:**
- A float occupies 2 heap words (1 header + 1 f64-as-u64)
- Float::value() for a stored 3.14 returns 3.14
- Float::value() for a stored 0.0 returns 0.0
- Float::value() for a stored -1.5 returns -1.5

**Files:**
- modify: crates/beamr/src/term/boxed.rs

**Checklist:**

### R5: Binary boxed type

WHEN a binary is allocated on the heap THE SYSTEM SHALL write a header word with the binary tag, followed by the byte length, followed by the byte data packed into words. Binary::len() SHALL return the byte count. Binary::as_bytes() SHALL return a &[u8] slice of the data. It SHALL NOT require a separate allocation — small binaries are inline on the process heap.

**Acceptance:**
- A 5-byte binary occupies header + 1 (length) + 1 (ceil(5/8)) = 3 heap words
- Binary::len() returns 5 for a 5-byte binary
- Binary::as_bytes() returns the original 5 bytes
- An empty binary (0 bytes) is valid with length 0

- modify: crates/beamr/src/term/binary.rs

**Checklist:**

### R6: Big integer, closure, map, and reference boxed types

beamr::term::boxed SHALL define header layouts and accessors for: big integer (sign + digit limbs), fun/closure (module atom, function index, arity, num_free captured terms), map (flatmap: sorted key array + value array), and reference (unique u64 id). Each SHALL use its own distinct header tag. It SHALL NOT implement full arithmetic on big integers — only the storage format. Map operations beyond construction and element access are deferred.

**Acceptance:**
- BigInt header stores sign and number of limbs; limbs are accessible as &[u64]
- Closure header stores module, function index, arity, num_free; Closure::free_var(0) returns the first captured term; Closure::free_var(n) for n >= num_free returns None
- Map stores keys and values; Map::get(key) performs linear scan on sorted keys
- Reference stores a unique u64 id; Reference::id() returns it
- Each type has a distinct header tag

## Boundaries

- SHALL NOT implement heap allocation or a Heap struct — boxed terms write to provided slices, the heap is a later brief
- SHALL NOT implement term comparison or ordering — B-007 scope
- SHALL NOT implement GC-related forwarding pointers or mark bits — GC brief scope
- SHALL NOT implement big integer arithmetic — only the storage format
- SHALL NOT implement hashmap (large map) representation — flatmap only for now
