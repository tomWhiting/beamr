## Purpose

Processes are the unit of execution and isolation. Nothing runs without them — the scheduler needs something to schedule, the interpreter needs somewhere to store registers, and message passing needs independent address spaces. This brief delivers the Process struct (heap, stack, registers, status, reduction counter), the per-process heap allocator (bump allocator for young generation), the call stack (frames with return address, saved registers, Y-register slots), and the process lifecycle (new, running, waiting, exited). It does NOT deliver the scheduler or mailbox — those are separate briefs that build on top of this one.

## Task

Implement the Process struct and related types in crates/beamr/src/process/. The Process owns a Heap (bump allocator over a Vec<u64>), a Stack (Vec<StackFrame> with Y-register storage), X-registers (fixed array of Term), a reduction counter, a status enum, a PID, and link/monitor sets (empty shells for now — supervision brief fills them). The Heap provides alloc(words) returning a pointer for boxed term construction. The Stack provides push_frame/pop_frame for call/return. Process::new(pid, heap_size) creates a fresh process. Process state transitions: New → Running → Yielded/Waiting/Exited.

## Requirements

### R1: Process struct definition

beamr::process SHALL define a Process struct containing: pid (u64), status (ProcessStatus enum), heap (Heap), stack (Stack), x_regs ([Term; 256] — X registers), y_regs accessible via current stack frame, reduction_counter (u32), current module + instruction pointer (code position), links (HashSet<u64> — PIDs), monitors (Vec<Monitor> — empty for now), trap_exit (bool). Process SHALL NOT be Send or Sync — it is owned by exactly one scheduler thread at a time.

**Acceptance:**
- Process is a public struct in beamr::process
- Process::new(pid, heap_size) creates a process with the given PID and heap capacity
- Process has 256 X registers initialized to Term::NIL
- Process is NOT Send or Sync (compile error if moved across threads)
- Process.pid returns the process ID
- Process.status() returns ProcessStatus::New for a fresh process

### R2: ProcessStatus enum and transitions

beamr::process SHALL define a ProcessStatus enum with variants: New, Running, Yielded, Waiting, Exited(ExitReason). Transitions SHALL be enforced: New→Running, Running→Yielded, Running→Waiting, Running→Exited, Yielded→Running, Waiting→Running. Invalid transitions SHALL return an error. ExitReason SHALL be an enum: Normal, Kill, Error (placeholder — will carry Term when terms land).

**Acceptance:**
- ProcessStatus has variants: New, Running, Yielded, Waiting, Exited(ExitReason)
- A fresh process starts as New
- Transition New→Running succeeds
- Transition New→Exited fails (must go through Running)
- Transition Running→Yielded succeeds
- Transition Exited→anything fails (terminal state)

### R3: Per-process heap — bump allocator

beamr::process::heap SHALL define a Heap struct backed by a Vec<u64>. Heap::alloc(words: usize) SHALL return a *mut u64 pointing to the allocated region, advancing the bump pointer. WHEN the heap is full THE SYSTEM SHALL return Err(HeapFull) — the caller (interpreter) triggers GC. The heap SHALL track: capacity, used words, and high-water mark. Heap::new(capacity) creates a heap with the given word capacity. The heap SHALL NOT grow automatically — GC is responsible for managing space.

**Acceptance:**
- Heap::new(1024) creates a heap with capacity for 1024 words
- Heap::alloc(3) returns a valid pointer and advances used by 3
- Heap::alloc when full returns Err(HeapFull)
- Heap::used() returns the number of words allocated
- Heap::capacity() returns the total word capacity
- Multiple alloc calls return non-overlapping regions

**Checklist:**

### R4: Call stack with frames

beamr::process::stack SHALL define a Stack struct containing a Vec<StackFrame>. Each StackFrame stores: return module (Atom), return instruction pointer (usize), number of Y-register slots (u16), and the Y-register values (Vec<Term>). Stack::push_frame(module, ip, y_slots) creates a new frame. Stack::pop_frame() removes the top frame and returns the saved return point. Stack::current_frame() returns a reference to the top frame. Stack::y_reg(n) accesses the nth Y register in the current frame. Stack overflow (configurable limit, default 10_000 frames) SHALL return Err(StackOverflow).

**Acceptance:**
- Stack::new() creates an empty stack
- push_frame + pop_frame round-trips the return address correctly
- Y registers in a frame are initialized to Term::NIL
- Writing y_reg(0) in frame N does not affect y_reg(0) in frame N-1
- Pushing beyond the frame limit returns Err(StackOverflow)
- pop_frame on an empty stack returns Err (not panic)

**Files:**
- modify: crates/beamr/src/process/stack.rs

**Checklist:**
- C54 — Y registers are stack-allocated: allocate/deallocate instructions create/destroy stack frames with Y slots

### R5: X-register array

The Process SHALL provide X-register access via x_reg(n: u8) -> Term and set_x_reg(n: u8, value: Term). X registers are a flat [Term; 256] array on the Process struct. All 256 registers are initialized to Term::NIL on process creation. Out-of-bounds access is impossible because the index is u8 (max 255).

**Acceptance:**
- Process x_reg(0) returns Term::NIL on a fresh process
- set_x_reg(0, term) followed by x_reg(0) returns that term
- All 256 registers are independently addressable
- Register operations do not allocate or panic

**Files:**

**Checklist:**

### R6: Reduction counter

The Process SHALL have a reduction counter (u32) initialized to a configurable budget (default 4000). Process::decrement_reductions(n: u32) subtracts n from the counter. Process::reductions_exhausted() returns true when the counter reaches 0. Process::reset_reductions(budget: u32) resets the counter for the next time slice. The interpreter calls decrement on every call/return opcode.

**Acceptance:**
- A fresh process has reduction_counter == 4000 (default)
- decrement_reductions(1) reduces counter by 1
- reductions_exhausted() returns true when counter == 0
- reset_reductions(4000) restores the counter
- decrement_reductions saturates at 0 (does not wrap)

### R7: Process table

beamr::process::registry SHALL define a ProcessTable that maps PID (u64) to process ownership handles. ProcessTable::spawn() assigns the next PID and returns it. ProcessTable::get(pid) returns a reference. The table SHALL support concurrent access from scheduler threads (it lives outside any single process). PID assignment SHALL be monotonically increasing and never reused within a single VM run. NOTE: C55 is split — this brief delivers heap deallocation and removal from process table on exit; exit signal propagation to linked processes is deferred to the supervision brief (B-020).

**Acceptance:**
- ProcessTable::new() creates an empty table
- spawn() returns sequential PIDs starting from 0
- get(pid) returns Some for a spawned process
- get(nonexistent_pid) returns None
- PIDs are never reused (two spawns produce different PIDs)

## Boundaries

- SHALL NOT implement the scheduler or scheduling logic — the process yields, the scheduler decides what to do with it (scheduler brief scope)
- SHALL NOT implement the mailbox — message storage and selective receive are a separate brief
- SHALL NOT implement GC — the heap reports HeapFull, GC is triggered by the interpreter/scheduler
- SHALL NOT implement process spawning from bytecode (spawn opcode) — that's interpreter + scheduler coordination
- SHALL NOT implement link/monitor propagation logic — only the empty container sets on the Process struct
- SHALL NOT implement process dictionary — out of scope per DESIGN.md non-goals
