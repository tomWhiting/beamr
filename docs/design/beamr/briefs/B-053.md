# B-053: Namespace infrastructure — multi-registry support on the Scheduler

> **Cluster:** beamr
> **Stories:**
> - S8 (Meridian Runtime, Embedding beamr for Workflow Execution) — As the Meridian runtime, I want to create a beamr VM instance, load .beam modules, and spawn processes from Rust so that I can embed workflow execution in the server process.

## Purpose

Aion runs multiple versions of the same workflow simultaneously — workflow A on v1 code, workflow B on v2 code. Real OTP limits modules to two versions. Aion currently works around this with content-hash bytecode rewriting (renaming every module to name$hash), which is fragile and breaks closure module references. This brief introduces per-namespace module registries so that each namespace has its own module lookup space, eliminating the need for bytecode rewriting. The atom table, BIF registry, scheduler threads, and process table remain shared.

## Task

Add a NamespaceId type and a namespace store (HashMap<NamespaceId, Arc<ModuleRegistry>>) to the Scheduler's SharedState. Provide API methods to create namespaces, load modules into a specific namespace, and spawn processes scoped to a namespace. A default namespace preserves backwards compatibility — existing API calls operate on it. Scope: scheduler infrastructure and public API only. Process-scoped module lookup wiring is B-054.

## Requirements

### R1: NamespaceId type and NamespaceStore

THE SYSTEM SHALL define a NamespaceId type (newtype over u64) and a NamespaceStore (DashMap<NamespaceId, Arc<ModuleRegistry>>) on SharedState. THE SYSTEM SHALL pre-create a DEFAULT_NAMESPACE (NamespaceId(0)) at Scheduler construction, backed by the existing module_registry.

**Acceptance:**
- NamespaceId is a #[derive(Copy, Clone, Debug, Eq, PartialEq, Hash)] newtype over u64
- NamespaceId::DEFAULT is a const for the default namespace (value 0)
- SharedState gains namespace_store: DashMap<NamespaceId, Arc<ModuleRegistry>>
- At Scheduler construction, namespace_store contains DEFAULT_NAMESPACE → Arc::clone of the shared module_registry
- The existing module_registry field on SharedState is unchanged — it IS the default namespace's registry

### R2: Scheduler::create_namespace API

WHEN Scheduler::create_namespace() is called, THE SYSTEM SHALL create a new empty ModuleRegistry, assign it a unique NamespaceId, store it in the namespace_store, and return the NamespaceId.

**Acceptance:**
- create_namespace() -> NamespaceId allocates a new ID via an AtomicU64 counter
- The new namespace starts with an empty ModuleRegistry
- Multiple calls return distinct NamespaceIds
- The namespace is immediately usable for load_module_in and spawn_in

### R3: Scheduler::load_module_in API

WHEN Scheduler::load_module_in(namespace, bytes) is called, THE SYSTEM SHALL load the module into the specified namespace's ModuleRegistry. Import resolution SHALL check the namespace's registry for cross-module references, falling back to the shared BIF registry. THE SYSTEM SHALL NOT resolve imports against the default namespace or other namespaces.

**Acceptance:**
- load_module_in(namespace: NamespaceId, bytes: &[u8]) -> Result<HotLoadResult, LoadError>
- The module is stored in the namespace's registry, not the default registry
- Cross-module imports resolve against the namespace's registry (deferred resolution)
- BIF imports resolve against the shared BIF registry (unchanged)
- Loading the same module name in two different namespaces creates independent entries
- Returns LoadError if the namespace doesn't exist

### R4: Scheduler::spawn_in API

WHEN Scheduler::spawn_in(namespace, module, function, args) is called, THE SYSTEM SHALL resolve the entry point from the namespace's registry and tag the spawned process with the namespace. THE SYSTEM SHALL NOT resolve the entry point from the default namespace.

**Acceptance:**
- spawn_in(namespace: NamespaceId, module: Atom, function: Atom, args: Vec<Term>) -> Result<u64, ExecError>
- The entry module is looked up in the namespace's registry
- The spawned process carries the NamespaceId (stored on the Process or in a per-PID map)
- spawn_in_trap_exit variant also available for Aion's use case
- Returns ExecError::Undef if the module is not in the namespace's registry

### R5: Process carries NamespaceId

THE SYSTEM SHALL add a namespace_id: NamespaceId field to Process. WHEN a process is spawned via spawn_in, THE SYSTEM SHALL set its namespace_id to the specified namespace. WHEN a process is spawned via the existing spawn API, THE SYSTEM SHALL set its namespace_id to DEFAULT_NAMESPACE.

**Acceptance:**
- Process gains namespace_id: NamespaceId field, defaulting to NamespaceId::DEFAULT
- Process::namespace_id() -> NamespaceId accessor
- spawn_in sets the process's namespace_id before making it runnable
- Existing spawn/spawn_trap_exit/spawn_link use DEFAULT_NAMESPACE (backwards compatible)

### R7: Namespace-scoped hot code loading and purge

WHEN hot_load_module_in(namespace, bytes) is called, THE SYSTEM SHALL perform the dual-version load within the namespace's registry. WHEN purge_module_in(namespace, name) is called, THE SYSTEM SHALL purge the old version from the namespace's registry only.

**Acceptance:**
- hot_load_module_in(namespace: NamespaceId, bytes: &[u8]) -> Result<HotLoadResult, LoadError>
- purge_module_in(namespace: NamespaceId, name: Atom) -> Result<(), PurgeError>
- force_purge_module_in(namespace: NamespaceId, name: Atom) -> Result<PurgeResult, PurgeError>
- force_purge only kills processes in the specified namespace (checks process namespace_id)
- Hot load within namespace A does not affect namespace B's view of the same module name

## Boundaries

- SHALL NOT implement per-namespace atom tables — atom interning remains global and shared
- SHALL NOT implement per-namespace BIF registries — BIFs are shared across all namespaces
- SHALL NOT implement per-namespace process tables or PID spaces — processes are globally addressable
- SHALL NOT implement cross-namespace module resolution — each namespace is isolated
- SHALL NOT change the Scheduler constructor signature — namespace support is opt-in via create_namespace

## Verification

- cargo test --workspace passes with all existing tests plus new namespace tests
- Existing spawn/hot_load_module API calls work unchanged (backwards compatible)
- Two namespaces with same-named module see different code
- Cross-namespace messaging works (PIDs are global)
- force_purge_module_in only affects the target namespace's processes
