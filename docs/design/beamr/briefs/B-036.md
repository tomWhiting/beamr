# B-036: beamr-meridian crate scaffold — MeridianRuntime, NIF wiring, run_workflow

> **Cluster:** beamr-meridian (yggdrasil repo)
> **Depends on:** B-033 (stdlib stubs), B-035 (Term↔Value bridge)
> **Checklist:**
> - C150 — BIF registry maps (Module, Function, Arity) to a Rust function pointer

## Purpose

The beamr-meridian crate is the integration layer between beamr (standalone BEAM VM) and Meridian (workflow executor). It creates a beamr runtime, registers workflow NIFs under `meridian_ffi`, loads compiled Gleam workflows, executes them, and converts the results for the Meridian orchestrator. This crate lives in the yggdrasil repo at `crates/beamr-meridian/` with beamr as a git dependency.

## Task

Scaffold the beamr-meridian crate with the MeridianRuntime struct, NIF registrations for all workflow operations, the run_workflow entry point, and a step runner bridge for dispatching AI agent steps.

## Boundaries

- This brief SHALL NOT modify the beamr core crate
- This brief SHALL NOT implement the full StepRunner async bridge (B-037, blocked on Norn library parity fix)
- This brief SHALL implement synchronous NIFs (read_file, write_file, run_cmd, commit) and stub async NIFs (run_step_norn)
- This brief SHALL live in the yggdrasil repo, not the beamr repo

### R2: NIF registrations — synchronous workflow operations

Register under `meridian_ffi` module atom. Each NIF wraps the corresponding Rust function from ygg-orchestrator or standard library:

| NIF | Arity | Rust implementation |
|-----|-------|-------------------|
| read_file | 1 | std::fs::read |
| write_file | 2 | std::fs::write |
| read_json | 1 | std::fs::read + serde_json::from_str |
| write_json | 2 | serde_json::to_string_pretty + std::fs::write |
| parse_json | 1 | serde_json::from_str |
| to_json | 1 | term_to_value + serde_json::to_string |
| run_cmd | 1 | std::process::Command |
| commit | 1 | git2 or shell git |
| render_template | 2 | minijinja::render |
| set_context | 2 | shared context map |
| edit_transcript | 2 | transcript pruning call |

All take binary Term arguments and return `{ok, Result}` or `{error, Reason}` tuples.

**Acceptance:**
- read_file reads a real file and returns binary content
- write_file creates/overwrites a file
- run_cmd executes a shell command and returns stdout
- parse_json converts JSON string to Term via Value→Term bridge
- Tests cover each NIF independently

**Files:**
- create: crates/beamr-meridian/src/nifs.rs

### R3: run_workflow entry point

`MeridianRuntime::run_workflow(module, function, inputs) -> Result<OrchestrationResult, Error>`:
1. Convert `HashMap<String, Value>` inputs to Term arguments via B-035's value_to_term
2. Spawn the Gleam function as a BEAM process via `scheduler.spawn(module, function, args)`
3. Block on `scheduler.run_until_exit(pid)`
4. Convert the return Term to `OrchestrationResult` via term_to_value
5. Map ExitReason to OrchestrationStatus (Normal → Completed, Error → Failed)

**Acceptance:**
- run_workflow with a trivial Gleam function returns Completed
- run_workflow with a failing Gleam function returns Failed
- Input HashMap values are accessible in the Gleam function
- Return value is correctly converted to serde_json::Value
- Tests cover success and failure paths

**Files:**
- modify: crates/beamr-meridian/src/runtime.rs
