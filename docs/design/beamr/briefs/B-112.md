# B-112: Node naming and node term type

> **Cluster:** beamr
> **Depends on:** B-016
> **Checklist:**
> - C150 — BIF registry maps (Module, Function, Arity) to a Rust function pointer

## Purpose

BEAM distribution requires every node to have a unique name (e.g., 'worker@host.local') and every PID/reference to carry a node origin. Currently, PIDs are plain u64 process IDs with no node context. This brief adds the Node concept — a named identity for the VM instance — and extends PID and Reference terms to carry their originating node. Without node-qualified identifiers, remote operations cannot distinguish local from remote processes.

## Task

Define the Node type, add a node registry, extend PID and Reference terms with node information, and implement the erlang:node/0 and erlang:node/1 BIFs.

## Requirements

### R1: Node type and local node identity

Define a Node type representing a named BEAM node. The local node is set at VM startup and is immutable for the lifetime of the instance.

**Acceptance:**
- Node struct with name: Atom and creation: u32 (distinguishes restarts of the same node name)
- Scheduler/SharedState gains local_node: Node field, set at startup
- Node::is_local(other: &Node) -> bool compares name and creation
- Default local node is 'nonode@nohost' (matching BEAM's non-distributed mode)
- Test: create VM with node name, verify node identity

**Files:**
- create: crates/beamr/src/distribution/mod.rs
- create: crates/beamr/src/distribution/node.rs
- modify: crates/beamr/src/lib.rs
- modify: crates/beamr/src/scheduler/mod.rs

**Checklist:**
- C150 — BIF registry maps (Module, Function, Arity) to a Rust function pointer

### R2: Node-qualified PID terms

Extend PID terms to carry their originating node. Local PIDs continue to use the compact immediate representation. Remote PIDs are boxed terms containing the node, process ID, and serial.

**Acceptance:**
- Local PIDs remain immediate terms (no heap allocation, backward compatible)
- Remote PID is a boxed term: [header] [node_atom] [pid_number] [serial]
- BoxedTag::ExternalPid added (or similar)
- Term::is_local_pid() and Term::is_remote_pid() discriminators
- BinaryRef-style PidRef accessor handles both local and remote transparently
- Test: local PID has no node overhead, remote PID carries node info

**Files:**
- modify: crates/beamr/src/term/mod.rs
- modify: crates/beamr/src/term/boxed/mod.rs

**Checklist:**
- C150 — BIF registry maps (Module, Function, Arity) to a Rust function pointer

### R3: erlang:node/0 and erlang:node/1 BIFs

Register node/0 (returns the local node name atom) and node/1 (returns the node of a PID, reference, or port term).

**Acceptance:**
- node() returns the local node name atom
- node(LocalPid) returns the local node name
- node(RemotePid) returns the remote node name
- node(non_pid) returns badarg
- Test: node() returns 'nonode@nohost' in non-distributed mode

**Files:**
- modify: crates/beamr/src/native/gate3_bifs/mod.rs

**Checklist:**
- C150 — BIF registry maps (Module, Function, Arity) to a Rust function pointer

### R4: Node-qualified references

Extend Reference terms to carry their originating node, same pattern as PIDs.

**Acceptance:**
- Local references remain compact (backward compatible)
- Remote references carry node information
- erlang:node/1 works on references
- Test: make_ref() produces a local reference, node(Ref) returns local node

**Files:**
- modify: crates/beamr/src/term/mod.rs
- modify: crates/beamr/src/term/boxed/mod.rs

**Checklist:**
- C150 — BIF registry maps (Module, Function, Arity) to a Rust function pointer

## Boundaries

- SHALL NOT implement network connections or handshake — that's B-114/B-115
- SHALL NOT implement EPMD or name resolution — that's B-113
- SHALL NOT change the local PID immediate representation — only add remote PID support
- SHALL support non-distributed mode ('nonode@nohost') as the default
