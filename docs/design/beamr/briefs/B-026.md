# B-026: Gate 3 erlang BIFs — element, send, make_ref, spawn/1, type queries

> **Cluster:** beamr
> **Depends on:** B-023a, B-024
> **Checklist:**
> - C82 — send instruction: send the term in x(1) to the process identified by x(0)
> - C150 — BIF registry maps (Module, Function, Arity) to a Rust function pointer
> **Stories:**
> - S4 (Workflow Author, Writing Gleam Workflows) — As a workflow author, I want to call Rust-native operations (git, AST merge, dependency graph) from Gleam via registered functions so that I get zero-overhead access to the host's capabilities.
> - S8 (Meridian Runtime, Embedding beamr for Workflow Execution) — As the Meridian runtime, I want to create a beamr VM instance, load .beam modules, and spawn processes from Rust so that I can embed workflow execution in the server process.

## Purpose

gleam_erlang and gleam_otp import erlang BIFs beyond Gate 1+2. This brief registers element/2, send/2, make_ref/0, spawn/1, spawn_link/1, is_process_alive/1, and tuple_size/1 as Gate 3 BIFs.

## Task

Implement 7 BIFs in gate3_bifs.rs and register via register_gate3_bifs().

## Requirements

### R1: erlang:element/2

erlang:element/2 SHALL accept (Index, Tuple) with 1-based indexing. Return element or badarg.

**Acceptance:**
- element(1, {a,b,c}) returns a
- element(3, {a,b,c}) returns c
- element(0, {a}) returns badarg
- Non-tuple returns badarg

**Files:**
- create: crates/beamr/src/native/gate3_bifs.rs
- modify: crates/beamr/src/native/mod.rs

**Checklist:**
- C150 — BIF registry maps (Module, Function, Arity) to a Rust function pointer

**Stories:**
- S4 (Workflow Author, Writing Gleam Workflows) — As a workflow author, I want to call Rust-native operations (git, AST merge, dependency graph) from Gleam via registered functions so that I get zero-overhead access to the host's capabilities.

### R2: erlang:send/2 and erlang:tuple_size/1

send/2 delivers message to target mailbox returning Message. tuple_size/1 returns tuple arity.

**Acceptance:**
- send(Pid, msg) delivers and returns msg
- send to nonexistent pid returns msg (no crash)
- tuple_size({a,b,c}) returns 3
- tuple_size(non_tuple) returns badarg

**Files:**
- create: crates/beamr/src/native/gate3_bifs.rs

**Checklist:**
- C82 — send instruction: send the term in x(1) to the process identified by x(0)
- C150 — BIF registry maps (Module, Function, Arity) to a Rust function pointer

**Stories:**
- S8 (Meridian Runtime, Embedding beamr for Workflow Execution) — As the Meridian runtime, I want to create a beamr VM instance, load .beam modules, and spawn processes from Rust so that I can embed workflow execution in the server process.

### R3: erlang:make_ref/0, is_process_alive/1, spawn/1, spawn_link/1

make_ref returns unique reference. is_process_alive checks process table. spawn/1 and spawn_link/1 accept zero-arity fun (MFA export only for now).

**Acceptance:**
- make_ref() returns unique values
- is_process_alive(living) returns true
- is_process_alive(dead) returns false
- spawn(fun m:f/0) spawns and returns pid
- spawn_link(fun m:f/0) spawns with link
- spawn(closure) returns badarg (documented)
- All 7 BIFs registered via register_gate3_bifs()

**Files:**
- create: crates/beamr/src/native/gate3_bifs.rs
- modify: crates/beamr/src/native/mod.rs
- modify: crates/beamr-cli/src/main.rs

**Checklist:**

**Stories:**
- S8 (Meridian Runtime, Embedding beamr for Workflow Execution) — As the Meridian runtime, I want to create a beamr VM instance, load .beam modules, and spawn processes from Rust so that I can embed workflow execution in the server process.
