# B-098: ETS concurrent access — read/write concurrency

> **Cluster:** beamr
> **Depends on:** B-094
> **Checklist:**
> - C150 — BIF registry maps (Module, Function, Arity) to a Rust function pointer

## Purpose

By default, ETS tables are protected by a single RwLock. The `{read_concurrency, true}` and `{write_concurrency, true}` options enable fine-grained concurrency for tables with skewed read/write patterns. Read concurrency optimizes for many readers by using read-biased locks. Write concurrency uses striped locks (sharding) to allow concurrent writes to different keys.

## Task

Implement the `read_concurrency` and `write_concurrency` table options with appropriate locking strategies.

## Requirements

### R1: read_concurrency option parsing

read_concurrency option parsing

**Acceptance:**
- `{read_concurrency, true}` parsed from ets:new options
- Stored in EtsTableMetadata
- For DashMap-based tables (set/bag): no-op (DashMap already supports concurrent reads)
- For ordered_set: switches from Mutex<BTreeMap> to RwLock<BTreeMap> allowing concurrent reads
- Test: create table with read_concurrency, multiple concurrent readers don't block

**Files:**
- modify: crates/beamr/src/ets/ordered_set.rs
- modify: crates/beamr/src/ets/mod.rs (option parsing)

**Checklist:**
- C150 — BIF registry maps (Module, Function, Arity) to a Rust function pointer

### R2: write_concurrency for set/bag tables only

write_concurrency is supported for set and bag table types (DashMap is already sharded). For ordered_set, write_concurrency is NOT supported in this brief — hash-based sharding would break the global key ordering required by first/next/last/prev traversal. A concurrent ordered data structure (skip list or concurrent B-tree) is a future optimization.

**Acceptance:**
- `{write_concurrency, true}` parsed and applied for set/bag tables (no-op — DashMap already sharded)
- `{write_concurrency, true}` on ordered_set is accepted but logged as ignored (single-lock behavior retained)
- ordered_set first/next/last/prev iteration continues to work correctly
- Test: set table with write_concurrency — concurrent writes succeed
- Test: ordered_set with write_concurrency — iteration order preserved

**Files:**
- modify: crates/beamr/src/ets/ordered_set.rs

**Checklist:**
- C150 — BIF registry maps (Module, Function, Arity) to a Rust function pointer

### R3: Concurrency option interaction

Concurrency option interaction

**Acceptance:**
- Both options can be set simultaneously: `[ordered_set, {read_concurrency, true}, {write_concurrency, true}]`
- Default (no options): single lock per table
- read_concurrency without write_concurrency: readers don't block each other, writers are exclusive
- write_concurrency without read_concurrency: writers to different shards don't block, readers may block
- Test: verify option combinations produce expected behavior under concurrent load

**Files:**
- modify: crates/beamr/src/ets/mod.rs

**Checklist:**
- C150 — BIF registry maps (Module, Function, Arity) to a Rust function pointer

## Boundaries

- SHALL NOT implement `{write_concurrency, auto}` (OTP 25+ auto-tuning) — use explicit boolean only
- SHALL NOT change the DashMap-based set implementation (DashMap already provides sharded write concurrency)
- SHALL focus on ordered_set which currently uses a single Mutex<BTreeMap>
