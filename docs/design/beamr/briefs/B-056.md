# B-056: get_list opcode — list destructuring

> **Cluster:** beamr
> **Depends on:** B-010
> **Checklist:**
> - C73 — get_hd and get_tl instructions: extract head or tail from a cons cell

## Purpose

The `get_list` opcode (opcode 65) destructures a cons cell into its head and tail in a single instruction. It is the inverse of `put_list` and is emitted by the compiler for virtually every list pattern match (`[H|T]`). Without it, any Gleam or Erlang code that pattern-matches on lists fails with `UnsupportedOpcode(65)`. This is the single most blocking missing opcode — it appears in nearly every compiled module.

## Task

Add a dedicated `Instruction::GetList` variant to the loader decoder (currently decoded as `Generic`), wire it into the opcode dispatch table, and implement the handler in `guards.rs` alongside the existing `get_hd` and `get_tl` opcodes.

## Requirements

### R1: Loader decode — dedicated Instruction variant

Replace the `Generic { opcode: 65, name: "get_list", operands }` decode path with a dedicated `Instruction::GetList { source, head, tail }` variant.

**Acceptance:**
- `Instruction::GetList` variant exists with `source: Operand`, `head: Operand`, `tail: Operand`
- Opcode 65 decodes to `Instruction::GetList` instead of `Instruction::Generic`
- Existing .beam files that use `get_list` load without errors
- Tests verify the decode produces the correct variant

**Files:**
- modify: crates/beamr/src/loader/decode/code.rs

**Checklist:**
- C73 — get_hd and get_tl instructions: extract head or tail from a cons cell

### R2: Opcode dispatch wiring

Add `Instruction::GetList` to the dispatch table in `opcodes/mod.rs`, routing to a handler in `guards.rs`.

**Acceptance:**
- `Instruction::GetList` is matched in the dispatch table
- Dispatch routes to `guards::get_list()`
- No `UnsupportedOpcode(65)` error when executing modules containing `get_list`

**Files:**
- modify: crates/beamr/src/interpreter/opcodes/mod.rs

**Checklist:**
- C73 — get_hd and get_tl instructions: extract head or tail from a cons cell

### R3: get_list handler implementation

Implement `get_list` in `guards.rs`. The instruction reads a cons cell from the source operand, writes the head to the head destination and the tail to the tail destination.

**Acceptance:**
- `get_list(source, head_dest, tail_dest)` extracts head and tail from a cons cell
- Source is read via `core::read_term()`, validated as a cons via `Cons::new()`
- Head is written to `head_dest` via `core::write_term()`
- Tail is written to `tail_dest` via `core::write_term()`
- Non-list source (including NIL) raises badarg
- Destinations can be X or Y registers
- Test: `get_list` on `[1, 2, 3]` writes `1` to head and `[2, 3]` to tail
- Test: `get_list` on a single-element list `[42]` writes `42` to head and `NIL` to tail
- Test: `get_list` on NIL raises badarg

**Files:**
- modify: crates/beamr/src/interpreter/opcodes/guards.rs (335 lines — room for ~20 more)

**Checklist:**
- C73 — get_hd and get_tl instructions: extract head or tail from a cons cell

## Boundaries

- SHALL NOT modify the cons cell heap representation — use the existing `Cons` accessor from `term/boxed/accessors.rs`
- SHALL NOT implement `get_list` as two separate `get_hd` + `get_tl` calls — it is a single atomic instruction
- SHALL NOT add branching/failure semantics — `get_list` on a non-list is a badarg (not a branch to fail label)
