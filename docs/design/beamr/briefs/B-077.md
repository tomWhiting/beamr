# B-077: Dirty scheduler thread pools — CPU and IO pools

> **Cluster:** beamr
> **Depends on:** B-032
> **Checklist:**
> - C150 — BIF registry maps (Module, Function, Arity) to a Rust function pointer

## Purpose

Normal scheduler threads run BEAM code with preemptive reduction counting — each process gets a fixed reduction budget, then yields. Long-running native calls (file I/O, crypto, JSON parsing) block the scheduler thread for their entire duration, starving other processes. BEAM solves this with dirty schedulers: separate thread pools dedicated to running long native calls. When a native call is marked dirty, the calling process migrates to a dirty thread, freeing the normal scheduler thread immediately.

BEAM has two dirty pools: CPU (compute-bound, N threads = CPU cores) and IO (I/O-bound, 10 threads by default, expandable). This brief creates both pools and the infrastructure to submit work to them.

## Task

Implement `DirtyPool` in `scheduler/dirty.rs` (currently a scaffold stub). Each pool owns a set of OS threads and a work queue. The pool accepts `DirtyJob` submissions and executes them on its threads.

## Requirements

### R1: DirtyPool struct and thread management

Replace the scaffold stub in `scheduler/dirty.rs` with a `DirtyPool` that owns a configurable number of OS threads.

**Acceptance:**
- `DirtyPool::new(name: &str, thread_count: usize) -> DirtyPool` creates the pool
- Pool spawns `thread_count` OS threads that wait for work
- `DirtyPool::shutdown(&self)` signals all threads to stop and joins them
- Threads are named (e.g., `dirty-cpu-0`, `dirty-io-1`) for debugging
- Default CPU pool: `num_cpus::get()` threads. Default IO pool: 10 threads.
- Test: create pool with 4 threads, verify 4 threads running, shutdown cleanly

**Files:**
- modify: crates/beamr/src/scheduler/dirty.rs

**Checklist:**

### R2: DirtyJob submission and execution

Define a `DirtyJob` that encapsulates a native function call to be executed on a dirty thread. The pool accepts jobs and executes them.

**Acceptance:**
- `DirtyJob` struct: `{ pid: u64, function: NativeFn, args: Vec<Term>, context: ProcessContext, result_sender: oneshot::Sender<Result<Term, Term>> }`
- `DirtyPool::submit(&self, job: DirtyJob)` enqueues the job for a worker thread
- Worker threads dequeue jobs, execute `(job.function)(&job.args, &mut job.context)`, and send the result back via `result_sender`
- Job queue is bounded (configurable max depth, default 1024) — submit blocks if queue is full
- Test: submit a job that returns 42, receive the result on the channel

**Files:**
- modify: crates/beamr/src/scheduler/dirty.rs

**Checklist:**
- C150 — BIF registry maps (Module, Function, Arity) to a Rust function pointer

### R3: Scheduler owns CPU and IO pools

Integrate the dirty pools into the Scheduler's lifecycle. The Scheduler creates both pools at startup and shuts them down on exit.

**Acceptance:**
- `SharedState` gains `dirty_cpu: DirtyPool` and `dirty_io: DirtyPool` fields
- `Scheduler::new()` creates both pools with configurable thread counts
- `Scheduler::shutdown()` shuts down both dirty pools before stopping normal workers
- Dirty pools are accessible from scheduler workers via SharedState
- Test: create Scheduler, verify dirty pools exist, shutdown cleanly

**Files:**

**Checklist:**

### R4: DirtySchedulerKind enum

Define the enum that distinguishes CPU-bound from IO-bound dirty work, matching BEAM's two-pool model.

**Acceptance:**
- `DirtySchedulerKind` enum: `Cpu`, `Io`
- `NativeEntry::dirty_kind` field: `Option<DirtySchedulerKind>` (None = normal scheduler, Some(Cpu) = dirty CPU, Some(Io) = dirty IO)
- Replaces the existing `is_dirty: bool` field with the more expressive enum
- Existing code that checks `is_dirty` is updated to check `dirty_kind.is_some()`
- Test: NativeEntry with dirty_kind=Some(Cpu) reports as dirty
