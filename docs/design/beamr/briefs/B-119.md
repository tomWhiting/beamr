# B-119: Distribution control messages — LINK, EXIT, MONITOR, SEND

> **Cluster:** beamr
> **Depends on:** B-118
> **Checklist:**
> - C150 — BIF registry maps (Module, Function, Arity) to a Rust function pointer

## Purpose

Beyond message send, distribution uses control messages for process lifecycle: LINK establishes cross-node links, EXIT propagates exit signals, MONITOR/DEMONITOR manage cross-node monitors, and EXIT2 sends explicit exit signals. These complete the supervision protocol across node boundaries.

## Task

Implement all distribution control message types for process lifecycle.

## Requirements

### R1: Control message dispatch table

Define the control message type enum and dispatch incoming control messages to appropriate handlers.

**Acceptance:**
- ControlOp enum: Send(2), RegSend(6), Link(1), Exit(3), Unlink(4), MonitorP(19), DemonitorP(20), MonitorPExit(21), Exit2(8), SpawnRequest(29), SpawnReply(31)
- Incoming control messages dispatched by op code
- Unknown op codes logged and ignored (forward compatibility)
- Test: dispatch each known op code to correct handler

**Files:**
- modify: crates/beamr/src/distribution/control.rs

**Checklist:**
- C150 — BIF registry maps (Module, Function, Arity) to a Rust function pointer

### R2: LINK and UNLINK control messages

Implement cross-node link establishment and removal.

**Acceptance:**
- Outbound: when link(LocalPid, RemotePid) is called, send LINK control to remote node
- Inbound: receiving LINK control establishes the link on the local side
- UNLINK removes the cross-node link
- Test: link two processes on different nodes, verify bidirectional

**Files:**
- modify: crates/beamr/src/distribution/control.rs

**Checklist:**
- C150 — BIF registry maps (Module, Function, Arity) to a Rust function pointer

### R3: EXIT and EXIT2 control messages

Propagate exit signals across node boundaries.

**Acceptance:**
- When a linked process exits, EXIT control sent to remote linked processes
- EXIT carries the exit reason as an ETF-encoded term
- EXIT2 (explicit exit/2 to remote PID) sends the signal
- Remote node delivers the exit signal to the target process (same as local exit signal)
- Test: process on A linked to process on B, A exits, B receives exit signal

**Files:**
- modify: crates/beamr/src/distribution/control.rs

**Checklist:**
- C150 — BIF registry maps (Module, Function, Arity) to a Rust function pointer

## Boundaries

- SHALL NOT implement SEND_SENDER (OTP 21+ optimization) — use basic SEND
- SHALL NOT implement PAYLOAD_EXIT (exit with large reason) — use basic EXIT
- SHALL handle all control messages even if some handlers are stubs initially
