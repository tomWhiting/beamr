## Purpose

This is the engine that makes everything else work. The execution loop fetches instructions, dispatches them, counts reductions, and yields when the budget runs out. The foundational opcodes — move, call, return, allocate, deallocate, test_heap, put_list, put_tuple2, get_tuple_element — are the minimum set needed to execute any Gleam function. Without this brief, we have data structures and registries but nothing that runs. Every subsequent interpreter brief (B-013 through B-016) builds on these primitives.

## Task

Implement the core interpreter in crates/beamr/src/interpreter/. The execution loop (mod.rs) fetches the instruction at the current code pointer, dispatches it via the opcode table, and advances the pointer. Reduction counting: decrement on every call/apply, yield when zero. Implement the foundational opcodes in opcodes/core.rs: label/func_info (metadata), move (register/literal/stack transfer), call/call_only/call_ext/call_ext_only/call_last/call_ext_last (function calls + tail calls), return, allocate/allocate_zero/deallocate (stack frame management), test_heap (heap space check), put_list (cons cell), put_tuple2 (tuple), get_tuple_element (element access).

## Requirements

### R1: Execution loop

beamr::interpreter SHALL implement a run(process, module) -> ExecutionResult function that executes instructions in a loop. Each iteration SHALL: read the instruction at the current code pointer, dispatch it to the appropriate handler, and advance the pointer (or jump). ExecutionResult SHALL be one of: Yielded (reduction budget exhausted), Waiting (blocked on receive), Exited(reason). It SHALL NOT return control to the caller mid-instruction — each instruction completes atomically.

**Acceptance:**
- run() with a single 'return' instruction returns Exited(Normal)
- run() with a call chain executes all instructions in sequence
- run() with a tight loop yields after the reduction budget is exhausted
- ExecutionResult has variants: Yielded, Waiting, Exited(ExitReason)

### R3: Reduction counting and preemptive yield

WHEN the interpreter executes a call or apply instruction THE SYSTEM SHALL decrement the process's reduction counter. WHEN the counter reaches zero THE SYSTEM SHALL save the process state (code pointer, registers) and return Yielded. The scheduler resets the budget and re-schedules. It SHALL NOT skip the decrement on any call variant — every function entry point costs a reduction.

**Acceptance:**
- A call instruction decrements the reduction counter by 1
- A process with budget 4000 yields after exactly 4000 call instructions
- After yielding, the process state (registers, code pointer) is preserved
- Resuming a yielded process continues from where it left off

### R4: Label, func_info, and move instructions

label SHALL mark a jump target (no-op during sequential execution). func_info SHALL store module/function/arity for error reporting. move SHALL copy a term from source (X reg, Y reg, literal, atom) to destination (X reg, Y reg). It SHALL NOT allocate heap space — move copies the 64-bit Term value only.

**Acceptance:**
- label is a no-op that does not modify registers or stack
- func_info stores module, function, arity for error reporting
- move from X register to X register copies the term
- move from literal to X register loads the literal
- move does not allocate heap space

### R5: Call instructions — local, external, and tail calls

call SHALL save the return address and jump to a label. call_only SHALL jump without saving. call_ext/call_ext_only SHALL resolve via module or BIF registry. call_last/call_ext_last SHALL deallocate the current frame before jumping (tail call). All call variants SHALL decrement the reduction counter. It SHALL NOT leave stale return addresses after a tail call.

**Acceptance:**
- call saves return address and jumps to label
- call_ext resolves module:function/arity via the module registry
- call_ext to a registered BIF invokes the native function
- call_last deallocates N stack slots then jumps (tail call)
- All call variants decrement the reduction counter

### R6: Return instruction

WHEN the interpreter executes return THE SYSTEM SHALL pop the return address from the stack and jump to it. IF the stack is empty (initial call frame) THE SYSTEM SHALL exit the process with reason Normal.

**Acceptance:**
- return pops the return address and resumes at the caller
- return on the initial frame exits the process with Normal
- After return, the stack frame count is decremented by one

### R7: Stack frame management and heap check

allocate SHALL push a frame with N Y slots. allocate_zero SHALL initialize Y slots to NIL. deallocate SHALL pop the frame. test_heap SHALL verify N words available, signaling GC-needed if not. It SHALL NOT silently ignore heap exhaustion.

**Acceptance:**
- allocate(3) creates a frame with 3 Y-register slots
- allocate_zero(3) creates a frame with 3 slots all set to NIL
- deallocate removes the top frame
- test_heap(10) succeeds when 10+ words available
- test_heap(10) signals GC-needed when insufficient

### R8: Data construction opcodes

put_list SHALL allocate a cons cell on the heap from head and tail. put_tuple2 SHALL allocate a tuple from elements. get_tuple_element SHALL extract the Nth element into a register. It SHALL NOT allow get_tuple_element on non-tuple terms — raises badarg.

**Acceptance:**
- put_list with head=1, tail=NIL creates [1]
- put_tuple2 with [1, 2, 3] creates a 3-tuple
- get_tuple_element(tuple, 0) stores the first element in the destination register
- get_tuple_element on a non-tuple raises badarg
