# B-116: Distribution atom cache — per-connection short index encoding

> **Cluster:** beamr
> **Depends on:** B-115
> **Checklist:**
> - C150 — BIF registry maps (Module, Function, Arity) to a Rust function pointer

## Purpose

Atoms are the most frequent term type in distribution messages. Sending the full atom string on every message wastes bandwidth. The distribution atom cache assigns short indices (0-255) to frequently used atoms per connection, so subsequent messages send the 1-byte index instead of the full string.

## Task

Implement the per-connection atom cache with cache entries, encoding, and decoding.

## Requirements

### R1: AtomCache per connection

Each DistConnection maintains a bidirectional atom cache — one for encoding (local → remote) and one for decoding (remote → local).

**Acceptance:**
- AtomCache struct with 256 slots (matching BEAM's cache size)
- insert(atom: Atom) -> u8 assigns an index, returns it
- lookup_by_index(index: u8) -> Option<Atom> retrieves cached atom
- lookup_by_atom(atom: Atom) -> Option<u8> checks if atom is cached
- Full cache evicts LRU entry
- Test: insert 256 atoms, verify all cached, insert 257th evicts first

**Files:**
- create: crates/beamr/src/distribution/atom_cache.rs

**Checklist:**
- C150 — BIF registry maps (Module, Function, Arity) to a Rust function pointer

### R2: Atom cache header encoding/decoding

Implement the atom cache header format used in distribution messages to communicate new cache entries.

**Acceptance:**
- Encode: when sending a message with an uncached atom, include a NEW_ATOM_CACHE_REF entry in the header
- Decode: when receiving a message with cache header entries, update the local cache
- Cached atoms are sent as 1-byte index references (ATOM_CACHE_REF)
- Test: encode message with new atom, decode on other side, verify cache populated

**Files:**
- modify: crates/beamr/src/distribution/atom_cache.rs

**Checklist:**
- C150 — BIF registry maps (Module, Function, Arity) to a Rust function pointer

## Boundaries

- SHALL NOT implement atom cache for non-distribution contexts — this is connection-specific
- SHALL NOT implement cache synchronization beyond the message header protocol
