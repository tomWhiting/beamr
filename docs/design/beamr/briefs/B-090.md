# B-090: ETS table registry and ownership model

> **Cluster:** beamr
> **Depends on:** B-032
> **Checklist:**
> - C150 — BIF registry maps (Module, Function, Arity) to a Rust function pointer

## Purpose

ETS (Erlang Term Storage) provides shared mutable tables accessible by multiple processes. Every other ETS brief builds on this one — the table registry, table IDs, ownership model, and access protection levels are the foundation. A table is owned by a process; when the owner dies, the table is deleted (unless an heir is configured — B-099). Access protection determines which processes can read/write.

## Task

Define the ETS table registry in SharedState, the `EtsTable` trait, table metadata (name, ID, type, protection, owner), and the table ID allocation scheme.

## Requirements

### R1: EtsTableId and table registry in SharedState

Add a table registry to SharedState that maps table IDs to table instances.

**Acceptance:**
- `EtsTableId` is a unique u64, monotonically increasing (never reused)
- `SharedState` gains `ets_tables: DashMap<EtsTableId, Arc<dyn EtsTable>>`
- `SharedState::create_table(metadata) -> EtsTableId` allocates an ID and inserts the table
- `SharedState::lookup_table(id) -> Option<Arc<dyn EtsTable>>` retrieves a table
- `SharedState::delete_table(id) -> bool` removes a table from the registry
- Named tables can also be looked up by atom: `SharedState::lookup_table_by_name(name: Atom) -> Option<EtsTableId>`
- Test: create table, look up by ID, delete, verify gone

**Files:**
- create: crates/beamr/src/ets/mod.rs
- modify: crates/beamr/src/scheduler/mod.rs (SharedState)
- modify: crates/beamr/src/lib.rs (add pub mod ets)

**Checklist:**
- C150 — BIF registry maps (Module, Function, Arity) to a Rust function pointer

### R2: EtsTable trait and table metadata

Define the trait that all table types implement, plus the metadata struct.

**Acceptance:**
- `EtsTableMetadata` struct: `name: Option<Atom>`, `id: EtsTableId`, `table_type: EtsTableType`, `protection: Protection`, `owner: u64` (PID), `keypos: usize` (1-based element position for key, default 1)
- `EtsTableType` enum: `Set`, `OrderedSet`, `Bag`, `DuplicateBag`
- `Protection` enum: `Public` (any process R/W), `Protected` (owner W, any R), `Private` (owner only)
- `EtsTable` trait with methods: `metadata(&self) -> &EtsTableMetadata`, `insert(&self, tuple: Term) -> Result<(), EtsError>`, `lookup(&self, key: Term) -> Vec<Term>`, `delete_key(&self, key: Term) -> bool`, `tab2list(&self) -> Vec<Term>`
- Test: create metadata, verify all fields accessible

**Files:**

- create: crates/beamr/src/ets/table.rs

**Checklist:**
- C150 — BIF registry maps (Module, Function, Arity) to a Rust function pointer

### R3: Access control enforcement

Implement protection level checks that gate read/write access based on the calling process's PID and the table's protection level.

**Acceptance:**
- `EtsTable::check_access(caller_pid: u64, operation: AccessOp) -> Result<(), EtsError>` where AccessOp is Read or Write
- Public: any PID can read or write
- Protected: any PID can read, only owner can write
- Private: only owner can read or write
- `EtsError::AccessDenied { table_id, caller_pid, operation }` for violations
- Test: private table rejects non-owner read and write
- Test: protected table allows non-owner read but rejects non-owner write
- Test: public table allows everything

**Files:**
- modify: crates/beamr/src/ets/table.rs

**Checklist:**

### R4: Owner death cleanup

When a process dies, all ETS tables it owns are deleted from the registry (unless heir is configured — B-099 adds that; for now, always delete).

**Acceptance:**
- Process cleanup path (supervision exit signal propagation) checks for owned tables
- All tables owned by the dying process are removed from the registry
- Table data is dropped (Arc ref count goes to 0 if no other references)
- Test: create table, kill owner process, verify table is deleted

**Files:**
- modify: crates/beamr/src/scheduler/mod.rs (cleanup_exited_process)

### R5: Term deep-copy infrastructure for ETS storage

ETS stores terms independently of process heaps. Boxed terms (tuples, lists, maps, binaries) contain pointers into the inserting process's heap — after GC or process exit, those pointers become dangling. THE SYSTEM SHALL deep-copy terms into ETS-owned memory on insert and deep-copy back into the requesting process's heap on lookup. Reuse the existing copy_term pattern from mailbox/mod.rs:208.

**Acceptance:**
- ETS tables store deep-copied terms in table-owned memory (Box/Vec allocations, not process heap pointers)
- ets:insert deep-copies the tuple from the process heap into table storage
- ets:lookup deep-copies the result from table storage into the requesting process's heap
- After inserting process GCs, the ETS entry is still valid
- After inserting process exits, the ETS entry is still valid
- copy_term function (or equivalent) handles all term types: atoms (no-copy immediate), integers (no-copy immediate), PIDs (no-copy immediate), tuples (recursive copy), lists (recursive copy), maps (recursive copy), binaries (copy or Arc share for refc binaries), closures (recursive copy)
- Test: insert a tuple from process A, GC process A, lookup from process B — correct data returned
- Test: insert from process A, exit process A, lookup from process B — correct data returned

**Files:**
- create: crates/beamr/src/ets/copy.rs
- modify: crates/beamr/src/ets/mod.rs

**Checklist:**
