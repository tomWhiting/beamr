# B-038: Collection BIFs — maps and lists operations

> **Cluster:** beamr
> **Depends on:** B-033 (stdlib stubs foundation)
> **Checklist:**
> - C150 — BIF registry maps (Module, Function, Arity) to a Rust function pointer

## Purpose

The sample workflow's dependency chain uses Gleam's dict and list modules, which delegate to Erlang's maps and lists modules. 17 functions across these two modules are unresolved. Without them, any Gleam workflow that creates, queries, or transforms collections fails at runtime.

## Task

Implement 17 BIFs across the maps and lists modules as native Rust stubs.

## Boundaries

- SHALL NOT implement maps:intersect_with/3 or maps:groups_from_list/2 (not in the sample workflow import chain)
- SHALL NOT implement lists:sort/2 with custom comparators (stub with natural term ordering)
- SHALL reuse existing Term comparison infrastructure for map key lookups

## Requirements

### R1: maps module BIFs (11 functions)

Register under `maps` module:

| Function | Arity | Behaviour |
|----------|-------|-----------|
| put | 3 | Insert or update key in map |
| find | 2 | Return {ok, Value} or error |
| keys | 1 | Return list of map keys |
| values | 1 | Return list of map values |
| to_list | 1 | Return list of {Key, Value} tuples |
| fold | 3 | Fold over map entries with accumulator |
| filter | 2 | Return map with entries matching predicate |
| merge_with | 3 | Merge two maps, calling resolver on conflicts |
| update_with | 4 | Update value for key using function, with default |
| with | 2 | Return map with only listed keys |
| without | 2 | Return map without listed keys |

**Acceptance:**
- `maps:put(k, v, #{})` returns `#{k => v}`
- `maps:find(k, #{k => v})` returns `{ok, v}`
- `maps:find(missing, #{k => v})` returns `error`
- `maps:keys(#{a => 1, b => 2})` returns `[a, b]`
- `maps:fold(Fun, 0, #{a => 1, b => 2})` accumulates correctly
- `maps:filter(Fun, Map)` retains matching entries
- Tests cover each function with at least 2 cases

**Files:**
- create: crates/beamr/src/native/stdlib_stubs/maps_bifs.rs
- modify: crates/beamr/src/native/stdlib_stubs/mod.rs

**Checklist:** C150

### R2: lists module BIFs (6 functions)

Register under `lists` module:

| Function | Arity | Behaviour |
|----------|-------|-----------|
| append | 1 | Flatten list of lists |
| append | 2 | Concatenate two lists |
| join | 2 | Intersperse separator between list elements |
| map | 2 | Apply function to each element |
| reverse | 2 | Reverse list and prepend to tail |
| seq | 2 | Generate integer sequence [From..To] |

**Acceptance:**
- `lists:append([[1,2],[3,4]])` returns `[1,2,3,4]`
- `lists:append([1,2], [3,4])` returns `[1,2,3,4]`
- `lists:seq(1, 5)` returns `[1,2,3,4,5]`
- `lists:reverse([1,2,3], [4,5])` returns `[3,2,1,4,5]`
- `lists:map(Fun, [1,2,3])` applies function correctly
- Tests cover each function

**Files:**

- modify: crates/beamr/src/native/stdlib_stubs/mod.rs

**Checklist:** C150

### R3: Higher-order function calling convention

maps:fold/3, maps:filter/2, lists:map/2, and maps:merge_with/3 accept closures (fun terms). The BIF must invoke the closure through the interpreter's trampoline mechanism (same as B-033's approach for select/1). Verify the calling convention handles both local and imported closures.

**Acceptance:**
- maps:fold with a closure accumulates correctly
- lists:map with a closure transforms each element
- maps:filter with a closure retains matching entries
- Closures that capture variables work correctly
- Tests verify closure invocation

**Files:**
- modify: crates/beamr/src/native/stdlib_stubs/maps_bifs.rs
- modify: crates/beamr/src/native/stdlib_stubs/lists_bifs.rs

**Checklist:** C150
