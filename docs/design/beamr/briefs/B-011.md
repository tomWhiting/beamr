## Purpose

Message passing is how processes communicate — there is no shared state. The mailbox is a per-process queue where messages arrive from any sender and are consumed by selective receive (pattern matching against the queue, taking the first match, deferring the rest). Without this brief, processes are deaf. This brief delivers the two-part mailbox (MPSC arrival queue + scan list for selective receive), message copying into the receiver's heap, and the save-pointer mechanism that avoids rescanning already-tested messages.

## Task

Implement the Mailbox struct in crates/beamr/src/mailbox/. The mailbox has two parts: (1) an arrival queue backed by a lock-free MPSC channel (crossbeam-channel SegQueue or similar) where any process can deposit messages, and (2) a scan list (VecDeque<Term>) owned by the receiving process for selective receive. When the process begins a receive, pending messages are drained from the arrival queue into the scan list. The process scans the list with a save pointer — messages before the pointer have already been tested and failed to match. When a message matches, it is removed from the list and the save pointer resets. When no message matches and the list is exhausted, the process suspends (Waiting status). Implement message send as a deep copy of the term into the receiver's heap.

### R2: Message send with deep copy

WHEN process A sends a message to process B THE SYSTEM SHALL deep-copy the term from A's heap into B's arrival queue. The copy SHALL be independent — modifying A's original SHALL NOT affect B's copy. For immediates (small int, atom, pid, nil), copying is trivial (they are values, not pointers). For boxed terms (tuples, lists, binaries), the copy SHALL recursively copy all heap-allocated data. The send operation SHALL NOT require the sender to hold a lock on the receiver's mailbox.

**Acceptance:**
- Sending a small integer arrives as the same value in the receiver's mailbox
- Sending a tuple deep-copies the tuple and its elements
- Sending a list deep-copies all cons cells
- The sender does not block or lock the receiver's scan list
- Multiple senders can send concurrently to the same mailbox

### R3: Selective receive with save pointer

beamr::mailbox::selective SHALL implement the BEAM selective receive algorithm. WHEN a process enters a receive expression THE SYSTEM SHALL: (1) drain pending messages from the arrival queue into the scan list, (2) scan the list starting from the save pointer, (3) for each message, test it against the provided match function, (4) if a message matches: remove it from the list, reset the save pointer to 0, return the message, (5) if no message matches: advance the save pointer to the end, return None (the process should suspend). The save pointer avoids rescanning messages that already failed to match in a previous scan.

**Acceptance:**
- A receive with one matching message returns it and removes it from the list
- A receive with no matching messages returns None
- After a failed scan, the save pointer is at the end of the list
- New messages arriving after a failed scan are tested from where the save pointer left off
- After a successful match, the save pointer resets to 0
- Messages before the save pointer are not re-tested

Mailbox SHALL provide drain_arrival() that moves all pending messages from the arrival queue into the scan list. drain_arrival is called at the start of each receive attempt. NOTE: C131 is split — this brief delivers the re-scan mechanism (drain arrival queue into scan buffer on wake); the actual scheduler wake-up (moving process from wait set to run queue) is scheduler scope (B-012 R5).
