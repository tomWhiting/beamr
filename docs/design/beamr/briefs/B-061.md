# B-061: build_stacktrace — stack trace construction

> **Cluster:** beamr
> **Depends on:** B-059 (catch/catch_end — exceptions.rs extraction, stack unwinding)
> **Checklist:**
> - C86 — Exception handling instructions (extends with stacktrace construction)
> - C49 — Process struct contains: pid, heap, stack, mailbox (extends with raw stacktrace field)

## Purpose

When an exception is caught via `try`/`catch`, the BEAM compiler emits `build_stacktrace` (opcode 160) to convert the raw exception context into the standard Erlang stacktrace format: a list of `{Module, Function, Arity, Info}` tuples. Without this opcode, any Gleam `rescue` or Erlang `catch Class:Reason:Stacktrace` pattern fails with `UnsupportedOpcode`.

The stacktrace construction has two phases:

1. **Capture** — at exception time, `raise_exception` snapshots the current code position and call stack return addresses into a raw trace on the Process.
2. **Build** — `build_stacktrace` reads the raw trace, resolves each entry to `{Module, Function, Arity, [{line, N}]}` using the module's function table and line info, allocates the list on the process heap, and stores it in `x(0)`.

The resolution requires two lookup tables on Module that don't currently exist: a function table mapping IP ranges to (function, arity) — derived from `func_info` instructions in the code — and a line table mapping IPs to line numbers — derived from `Line` instructions.

## Boundaries

- This brief SHALL capture raw stacktraces in `raise_exception` (modifying the function added/moved by B-059 R6)
- This brief SHALL build function and line lookup tables on Module at load time
- This brief SHALL implement the `build_stacktrace` opcode handler (opcode 160)
- This brief SHALL NOT implement `raw_raise` (B-062)
- This brief SHALL NOT include file name information in the `Info` proplist — only `[{line, N}]` (file name requires binary construction from the Line chunk's file table, deferred to a later brief)

## Requirements

### R1: Raw stacktrace capture in raise_exception

When an exception is raised, snapshot the current execution context before dispatching to the handler.

**Acceptance:**
- New struct `RawStackEntry { module: Arc<Module>, ip: usize, mfa: Option<(Atom, Atom, u8)> }` — carries the pinned module version, instruction pointer, and optional MFA from `func_info`
- Process gains field `raw_stacktrace: Vec<RawStackEntry>`
- `raise_exception` (in `exceptions.rs` after B-059 R6) captures the raw trace before popping the handler:
  1. First entry: current module from `process.current_module()`, IP from `process.code_position()`, MFA from `process.current_mfa()`
  2. Remaining entries: one per stack frame, using `frame.pinned_module()` and `frame.return_ip()` (MFA resolved at build time via function table)
- Raw stacktrace is captured for ALL handler kinds (Try and Catch) — both need traces
- `process.clear_raw_stacktrace()` called when exception is cleared (in `try_end`, `catch_end`)
- Accessor: `process.raw_stacktrace() -> &[RawStackEntry]`
- Test: raise exception with 3 stack frames, verify raw_stacktrace has 4 entries (current + 3 returns), each with correct module and IP

**Files:**
- modify: crates/beamr/src/process/mod.rs (RawStackEntry, field, accessors)
- modify: crates/beamr/src/interpreter/opcodes/exceptions.rs (raise_exception capture)

**Checklist:** C49

### R2: Function lookup table on Module

Build a sorted table mapping IP ranges to function identity, derived from `func_info` instructions in the code.

**Acceptance:**
- Module gains field `function_table: Vec<(usize, Atom, u8)>` — sorted by IP, each entry is `(func_info_ip, function_name, arity)`
- Built at load time in `load_module()` by scanning the code Vec for `Instruction::FuncInfo` entries and extracting their operands
- New method: `Module::function_at_ip(ip: usize) -> Option<(Atom, u8)>` — binary search for the entry with the largest IP ≤ target IP, returns `(function_name, arity)`
- Private functions are included (func_info appears before every function, not just exports)
- Test: load a module with 3 functions (2 exported, 1 private). Verify function_at_ip resolves all three correctly. Verify IP between functions resolves to the earlier function.

**Files:**
- modify: crates/beamr/src/module.rs (field, function_at_ip method)
- modify: crates/beamr/src/loader/load.rs (build function_table at load time)

**Checklist:** C86

### R3: Line lookup table on Module

Build a sorted table mapping IPs to line numbers, derived from `Line` instructions in the code.

**Acceptance:**
- Module gains field `line_table: Vec<(usize, usize)>` — sorted by IP, each entry is `(line_instruction_ip, line_info_index)`
- Built at load time in `load_module()` by scanning the code Vec for `Instruction::Line { index }` entries
- New method: `Module::line_at_ip(ip: usize) -> Option<u32>` — binary search for the entry with the largest IP ≤ target IP, returns the line number from `self.line_info[index].line`
- Returns `None` if no `Line` instruction precedes the target IP or if the line info index is out of bounds
- Test: load a module with known line instructions. Verify line_at_ip returns correct line numbers. Verify IP before any Line instruction returns None.

**Files:**
- modify: crates/beamr/src/module.rs (field, line_at_ip method)
- modify: crates/beamr/src/loader/load.rs (build line_table at load time)

**Checklist:** C86

### R4: build_stacktrace opcode handler (opcode 160)

Implement the `build_stacktrace` instruction. Reads the raw stacktrace from Process, builds the standard `[{M, F, A, Info}]` list on the process heap, and stores it in `x(0)`.

**Acceptance:**
- `build_stacktrace(process)` reads `process.raw_stacktrace()`
- For each `RawStackEntry`:
  1. Module name from `entry.module.name` (Atom, becomes an atom term)
  2. Function and arity: use `entry.mfa` if available (first frame), otherwise resolve via `entry.module.function_at_ip(entry.ip)`. If unresolvable, use `(Atom::UNDEFINED, 0)`.
  3. Line number: `entry.module.line_at_ip(entry.ip)`. If available, `Info = [{line, N}]`. If not, `Info = []` (NIL).
- Each stacktrace entry is a 4-tuple `{Module, Function, Arity, Info}` allocated on the process heap
- The `Info` proplist `[{line, N}]` is a cons cell containing a 2-tuple
- Entries are linked into a proper list (most recent frame first)
- Result stored in `x(0)`
- Raw stacktrace is NOT cleared here (it persists until `try_end`/`catch_end` clears it)
- New common atom: `Atom::LINE` = `"line"` for the info proplist key (index 32, after B-059's THROW=30, EXIT_CLASS=31)
- `Instruction::BuildStacktrace` dispatched in `opcodes/mod.rs` to the new handler
- If process heap is insufficient, return `ExecError::HeapFull` (stacktrace construction is best-effort under memory pressure)
- Test: raise exception in a function with known line info. Call build_stacktrace. Verify x(0) is a proper list of 4-tuples with correct M/F/A/line values.
- Test: empty raw stacktrace (shouldn't happen, but defensive) produces NIL in x(0)

**Files:**
- modify: crates/beamr/src/interpreter/opcodes/exceptions.rs (handler)
- modify: crates/beamr/src/interpreter/opcodes/mod.rs (dispatch arm)
- modify: crates/beamr/src/atom/table.rs (Atom::LINE)

**Checklist:** C86
