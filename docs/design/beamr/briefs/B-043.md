# B-043: Dynamic import resolution for cross-module calls

> **Cluster:** beamr
> **Depends on:** B-041
> **Checklist:**
> - C150 — BIF registry maps (Module, Function, Arity) to a Rust function pointer
> - C110 — Module registry maps module names to loaded Module structs

## Purpose

Import resolution is currently frozen at module load time: `resolve_imports` in the loader captures the target module's export label as a u32 and stores it in the calling module's `resolved_imports` table. At call time, the interpreter re-looks up the target module from the registry but uses the cached label to find the entry point.

Under hot code loading, if a target module is reloaded with different export label positions, the cached labels in all calling modules become stale. A call to `module_b:foo/0` would jump to whatever code is now at the old label position — a different function, a mid-function instruction, or out of bounds.

This brief makes cross-module call resolution fully dynamic: resolve the function by (module, function, arity) at call time, not by cached label.

## Boundaries

- This brief SHALL NOT change intra-module call resolution (calls within the same module use labels, which are always consistent within a single version)
- This brief SHALL NOT implement on_load (that's B-045)
- This brief SHALL maintain backward compatibility — modules loaded without hot code loading still work identically

## Requirements

### R1: Export lookup method on Module

Add a direct method on Module that resolves a (function, arity) pair to an instruction pointer via the label index (from B-041 R5).

**Acceptance:**
- `Module::export_ip(function: Atom, arity: u8) -> Result<usize, ExecError>` looks up the export label in `self.exports`, then resolves it to an IP via `self.label_index`
- Returns `ExecError::Undef` if the function is not exported
- Returns `ExecError::InvalidLabel` if the label exists in exports but not in the label index (shouldn't happen, but defensive)
- Test: load a module with known exports, verify export_ip returns correct IPs

**Files:**
- modify: crates/beamr/src/module.rs

**Checklist:** C110

### R2: Dynamic cross-module call dispatch

Modify `call_external_target` to resolve cross-module calls by (module, function, arity) at call time rather than using the cached label from `resolved_imports`.

**Acceptance:**
- For `ResolvedImportTarget::Code` targets: instead of using the stored `label`, use the target module's `export_ip(function, arity)` where function and arity come from the `ResolvedImport` entry
- The `ResolvedImport` already stores `module: Atom`, `function: Atom`, `arity: u8` — these are used for the dynamic lookup
- The cached `label` field in `ResolvedImportTarget::Code` is retained for backward compatibility but no longer used for dispatch
- BIF targets (`ResolvedImportTarget::Native`) are unchanged — they are version-independent
- Test: load module B v1 with `foo/0` at label 5. Load module A that imports B:foo/0. Call B:foo/0 from A — works. Reload module B v2 with `foo/0` at label 12 (different position). Call B:foo/0 from A again — resolves correctly to the new position via dynamic lookup.

**Files:**
- modify: crates/beamr/src/interpreter/opcodes/core.rs (call_external_target)

**Checklist:** C150

### R3: Unresolved import fallback

Currently, imports that can't be resolved at load time (target module not yet loaded) cause the import to be marked unresolved and reported. Under hot code loading, a module might be loaded before its dependency — the import should be resolvable at call time even if it wasn't at load time.

**Acceptance:**
- Add a new variant: `ResolvedImportTarget::Deferred { module: Atom, function: Atom, arity: u8 }` for imports where the target module was not in the registry at load time but the MFA is known
- At call time, `call_external_target` handles `Deferred` by doing a full registry lookup + export resolution. If found, the call proceeds normally. If not found, raise `undef` error.
- `UnresolvedImportReport` distinguishes between truly unresolvable imports (unknown BIFs) and deferred imports (module not loaded yet)
- Test: load module A that imports B:foo/0, where B is not loaded yet. Import is deferred. Then load module B. Call B:foo/0 from A — succeeds via deferred resolution.

**Files:**
- modify: crates/beamr/src/module.rs (ResolvedImportTarget)
- modify: crates/beamr/src/loader/load.rs (resolve_imports)
- modify: crates/beamr/src/interpreter/opcodes/core.rs (call_external_target)
- modify: crates/beamr/src/loader/mod.rs (UnresolvedImportReport)

**Checklist:** C150

### R4: Apply instruction dynamic resolution

The `apply` and `apply_last` opcodes resolve a call dynamically from terms on the stack (module, function atoms + argument list). Verify these already work correctly under hot code loading since they do full MFA resolution at call time, and add explicit tests.

**Acceptance:**
- Audit `apply` and `apply_last` in the opcode dispatch — confirm they use `registry.lookup_mfa()` or equivalent dynamic resolution
- If they use any cached data, migrate to dynamic resolution
- Test: module A calls `apply(module_b, foo, [])`. Reload module B. Next apply call picks up the new version.

**Files:**
- modify: crates/beamr/src/interpreter/opcodes/core.rs (apply, apply_last — if changes needed)

**Checklist:** C150
