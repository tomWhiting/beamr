# B-067: Process dictionary — put/get/erase BIFs

> **Cluster:** beamr
> **Depends on:** B-029 (GC — landed)
> **Checklist:**
> - C49 — Process struct contains: pid, heap, stack, mailbox (extends with dictionary)
> - C150 — BIF registry maps (Module, Function, Arity) to a Rust function pointer

## Purpose

Every BEAM process has a mutable key-value dictionary, local and private to that process. It's the one piece of mutable state in the process model that breaks referential transparency — which is precisely why it exists: pragmatic escape hatch for per-process caches, counters, and metadata that would otherwise require threading state through every function call.

The process dictionary is accessed via BIFs: `put/2`, `get/1`, `get/0`, `erase/1`, `erase/0`, `get_keys/1`. OTP libraries use it extensively — `gen_server` stores `$gen_call` tags, `logger` stores process metadata, `random` stores seed state. Without it, these modules fail.

The critical design concern is GC integration: the dictionary contains Terms that may point to heap-allocated objects (tuples, lists, closures, etc.). The GC must trace these Terms as roots and rewrite them when objects are relocated during collection. The existing root tracing pattern (x_regs → y_regs → mailbox → exception) must extend to include dictionary entries.

## Boundaries

- This brief SHALL add a `Vec<(Term, Term)>` field to Process (NOT HashMap — see R1 rationale)
- This brief SHALL integrate dictionary terms into the GC root set
- This brief SHALL implement 6 process dictionary BIFs
- This brief SHALL NOT add any locking — the dictionary is process-local, no concurrency
- This brief SHALL NOT implement `process_info(Pid, dictionary)` — that's a separate introspection concern

## Requirements

### R1: Dictionary field on Process with GC root integration

Add the dictionary storage and wire it into the GC root tracing.

**Acceptance:**
- Process gains field `dictionary: Vec<(Term, Term)>` — NOT HashMap. HashMap<Term, Term> is broken for boxed keys: Term's Hash impl uses the raw u64 (pointer value). After GC relocates a boxed term, its pointer changes. The entry stays in the old hash bucket, but lookups with the new pointer hash to a different bucket — silent miss. GC root rewriting updates the Term value but cannot re-bucket the HashMap entry. Vec with linear scan avoids this entirely. Process dictionaries are typically small (<20 entries); O(n) lookup is fine.
- Initialized empty in `Process::new()`
- `roots_with_live_x()` chains dictionary entries into the root collection after exception roots. Order: pairs in Vec order, key then value for each pair. This order is stable across GC cycles since Vec preserves insertion order.
- `replace_roots_with_live_x()` writes back dictionary roots in the same order. Both keys and values are rewritten in place when GC relocates heap objects. Since we use Vec (not HashMap), the rewritten key values don't break any bucketing.
- Test: allocate a tuple on the heap, store it in the dictionary via put. Trigger minor GC. Verify the dictionary entry still points to a valid tuple (the reference was rewritten by GC).
- Test: dictionary entries with non-heap terms (small integers, atoms) survive GC unchanged.

**Files:**
- modify: crates/beamr/src/process/mod.rs (field, roots, replace_roots)

**Checklist:** C49

### R2: Dictionary accessor methods on Process

Expose the dictionary operations as Process methods (BIFs in R3 delegate to these).

**Acceptance:**
- `Process::dict_put(&mut self, key: Term, value: Term) -> Term` — linear scan for matching key (using `compare::exact_eq`), insert or update, returns old value or `Term::atom(Atom::UNDEFINED)`
- `Process::dict_get(&self, key: Term) -> Term` — linear scan for matching key (using `compare::exact_eq`), returns value or `Term::atom(Atom::UNDEFINED)`
- `Process::dict_get_all(&self) -> &[(Term, Term)]` — borrow all entries
- `Process::dict_erase(&mut self, key: Term) -> Term` — linear scan, remove via `swap_remove`, returns old value or `Term::atom(Atom::UNDEFINED)`
- `Process::dict_erase_all(&mut self) -> Vec<(Term, Term)>` — drain and return all entries
- `Process::dict_get_keys(&self, value: Term) -> Vec<Term>` — linear scan for matching value (uses `compare::exact_eq`, per BEAM semantics)
- Test: put/get round-trip
- Test: put replaces existing entry, returns old value
- Test: get on missing key returns undefined
- Test: erase on missing key returns undefined

**Files:**
- modify: crates/beamr/src/process/mod.rs

**Checklist:** C49

### R3: Process dictionary BIFs

Register the 6 dictionary BIFs. All are `Capability::ProcessLocal` (they access process state, not pure functions).

**Acceptance:**
- `erlang:put/2` — calls `process.dict_put(key, value)`, returns old value
- `erlang:get/1` — calls `process.dict_get(key)`, returns value or `undefined`
- `erlang:get/0` — calls `process.dict_get_all()`, builds list of `{Key, Value}` 2-tuples on heap, returns list
- `erlang:erase/1` — calls `process.dict_erase(key)`, returns old value
- `erlang:erase/0` — calls `process.dict_erase_all()`, builds list of `{Key, Value}` 2-tuples on heap, returns list
- `erlang:get_keys/1` — calls `process.dict_get_keys(value)`, builds list of keys on heap, returns list
- For `get/0`, `erase/0`, and `get_keys/1`: heap allocation for the result list. If heap is full, return `ExecError::HeapFull`.
- Test: from BIF mechanism, put a value, get it back, erase it, verify get returns undefined
- Test: get/0 returns complete dictionary as list of tuples
